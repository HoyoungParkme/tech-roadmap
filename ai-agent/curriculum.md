# Part I. AI Agent란 무엇인가

> "에이전트"라는 말을 들으면 스파이 영화가 떠오를 수 있다. 하지만 AI Agent는 사용자 대신 **관찰하고, 생각하고, 행동하는** 소프트웨어 도우미다. Part I은 에이전트가 왜 필요한지, 기존 챗봇과 뭐가 다른지, 핵심 구성요소가 무엇인지를 비유 중심으로 설명한다.

## Chapter I-1. 에이전트의 탄생

### I-1-1. 챗봇 vs AI Agent

- **난이도**: ★☆☆
- **선수 지식**: 없음
- **학습 목표**: 단순 챗봇과 AI Agent의 차이를 설명할 수 있다
- **설명**:
  챗봇은 **무전기**와 같다. 상대방의 말을 받아서 정해진 대로 대답만 한다. "오늘 날씨 알려줘"라고 하면 날씨를 알려주지만, 우산을 챙겨주진 않는다.
  AI Agent는 **비서**와 같다. "출장 준비해줘"라고 하면 날씨를 확인하고, 일정을 잡고, 교통편을 예약하고, 우산이 필요하면 챙겨둔다. 스스로 판단하고 행동한다.
  핵심 차이는 **자율적 판단과 행동 능력**이다.
- **핵심 키워드**: Chatbot, AI Agent, 자율성, 판단, 행동
- **시각화 연결**: [I-1-1 시각화 보기](part1/I-1-1.html) — 챗봇(고정 응답) vs Agent(관찰-사고-행동 루프) 비교 (5단계)
- **다음 섹션**: I-1-2

### I-1-2. Agent의 3요소: 관찰-사고-행동

- **난이도**: ★☆☆
- **선수 지식**: I-1-1
- **학습 목표**: Perception-Reasoning-Action 루프를 그릴 수 있다
- **설명**:
  음식점 매니저를 상상해보자. 매니저는 (1) 주방 상황과 손님 수를 **관찰**하고, (2) "지금 4번 테이블에 디저트를 먼저 내보내야겠다"고 **판단**하고, (3) 직원에게 지시를 내려 **행동**한다.
  AI Agent도 똑같다. **Observe**(환경 관찰) → **Think**(추론/판단) → **Act**(행동 실행) → 결과를 다시 관찰 → 반복. 이 순환이 에이전트의 심장이다.
- **핵심 키워드**: Perception, Reasoning, Action, 루프, 순환
- **시각화 연결**: [I-1-2 시각화 보기](part1/I-1-2.html) — Observe → Think → Act 순환 다이어그램 (5단계)
- **다음 섹션**: I-1-3

### I-1-3. 왜 Agent가 필요한가

- **난이도**: ★☆☆
- **선수 지식**: I-1-2
- **학습 목표**: LLM 단독 사용의 한계와 에이전트가 해결하는 문제를 설명할 수 있다
- **설명**:
  LLM만 있는 사람은 **백과사전만 들고 있는 사람**이다. 뭘 물어봐도 답은 할 수 있지만, 직접 전화를 걸거나, 인터넷을 검색하거나, 메모를 남기는 건 못 한다.
  Agent는 백과사전(LLM) + **전화기**(외부 API 호출) + **인터넷**(검색) + **수첩**(메모리)을 가진 사람이다.
  LLM의 3가지 한계: (1) 학습 이후 정보를 모른다 (지식 단절), (2) 가끔 지어낸다 (환각), (3) 대답만 하고 실행은 못 한다 (행동 불가). Agent가 이 세 가지를 해결한다.
- **핵심 키워드**: 지식 단절, 환각(Hallucination), 행동 불가, 도구 사용
- **시각화 연결**: [I-1-3 시각화 보기](part1/I-1-3.html) — LLM 한계 3가지 → Agent 해결책 매핑 (4단계)
- **다음 섹션**: I-2-1

## Chapter I-2. Agent의 구성요소

### I-2-1. Agent 아키텍처 전체 지도

- **난이도**: ★☆☆
- **선수 지식**: I-1-3
- **학습 목표**: LLM + Memory + Tools + Planning이 어떻게 조합되는지 전체 그림을 그릴 수 있다
- **설명**:
  회사를 운영한다고 생각해보자.
  - **사장 (LLM)**: 모든 판단의 중심. 보고를 받고, 지시를 내린다.
  - **수첩 (Memory)**: 이전 대화, 중요 정보를 기록해둔다. 단기 기억(대화 맥락)과 장기 기억(저장된 지식)이 있다.
  - **도구함 (Tools)**: 계산기, 검색엔진, 이메일, 코드 실행기 등. 사장이 직접 못 하는 일을 대신 해준다.
  - **일정표 (Planning)**: 복잡한 작업을 단계별로 분해하고 순서를 정한다.
  이 네 가지가 모여 하나의 Agent가 된다.
- **핵심 키워드**: LLM(뇌), Memory(기억), Tools(도구), Planning(계획)
- **시각화 연결**: [I-2-1 시각화 보기](part1/I-2-1.html) — 중앙 LLM + 4개 모듈 아키텍처 다이어그램 (5단계)
- **다음 섹션**: I-2-2

### I-2-2. 자율성 스펙트럼

- **난이도**: ★☆☆
- **선수 지식**: I-2-1
- **학습 목표**: "완전 수동"부터 "완전 자율"까지 에이전트의 자율성 수준을 분류할 수 있다
- **설명**:
  자동차를 생각해보자.
  - **Level 0 (수동)**: 리모컨 자동차. 사람이 모든 걸 조종한다. → 단순 API 호출 (사람이 매번 지시)
  - **Level 1 (보조)**: 크루즈 컨트롤. 속도만 유지해준다. → Copilot (코드 자동완성)
  - **Level 2 (반자율)**: 차선 유지 + 속도 조절. → ReAct Agent (판단+행동, 사람이 감독)
  - **Level 3 (조건부 자율)**: 고속도로에서 자율주행, 시내에서는 사람. → 특정 도메인 자율 Agent
  - **Level 4 (완전 자율)**: 목적지만 말하면 알아서 간다. → 범용 자율 Agent (아직 연구 중)
  현재 실용적인 에이전트는 대부분 Level 2~3에 위치한다.
- **핵심 키워드**: 자율성, Human-in-the-Loop, 감독, Copilot, 자율 Agent
- **시각화 연결**: [I-2-2 시각화 보기](part1/I-2-2.html) — 자율성 스펙트럼 슬라이더 (5단계)
- **다음 섹션**: II-1-1

# Part II. LLM 기초

> AI Agent의 "뇌" 역할을 하는 LLM을 이해해야 에이전트를 제대로 쓸 수 있다. Part II는 프롬프트, 토큰, API 호출의 기본을 다룬다. 코드를 직접 치지 않아도 되는 수준으로 설명하되, 간단한 API 호출 예시는 코드와 함께 보여준다.

## Chapter II-1. LLM이란

### II-1-1. LLM의 작동 원리 (다음 토큰 예측)

- **난이도**: ★☆☆
- **선수 지식**: I-2-1
- **학습 목표**: LLM이 "다음 단어 예측" 방식으로 동작한다는 원리를 설명할 수 있다
- **설명**:
  스마트폰의 **자동완성 키보드**를 떠올려보자. "오늘 점심은"이라고 치면 "뭐 먹을까?", "맛있었다" 같은 후보가 뜬다. LLM도 정확히 이 방식이다.
  다만, 스마트폰은 2~3개 후보만 보여주지만, LLM은 수십만 개의 단어 중에서 확률적으로 가장 적절한 다음 단어(토큰)를 고른다. 이걸 반복하면 문장이 완성된다.
  핵심: LLM은 "이해"하는 게 아니라 **"예측"**하는 것이다.
- **핵심 키워드**: 토큰, 다음 토큰 예측, 확률, 자동완성
- **시각화 연결**: [II-1-1 시각화 보기](part2/II-1-1.html) — 토큰 시퀀스 생성 애니메이션 (5단계)
- **다음 섹션**: II-1-2

### II-1-2. 토큰과 컨텍스트 윈도우

- **난이도**: ★☆☆
- **선수 지식**: II-1-1
- **학습 목표**: 토큰이 무엇인지, 컨텍스트 윈도우의 크기 제한이 왜 중요한지 설명할 수 있다
- **설명**:
  **책상 크기**를 생각해보자. 책상이 크면 많은 서류를 펼쳐놓고 동시에 볼 수 있다. 작으면 서류를 자주 치우고 꺼내야 한다.
  컨텍스트 윈도우 = 책상 크기. GPT-4는 128K 토큰, Claude는 200K 토큰. 이 안에 시스템 메시지, 대화 이력, 도구 결과 등이 모두 들어가야 한다.
  토큰은 단어보다 작은 단위다. "Hello" = 1토큰, "안녕하세요" = 3~4토큰. 한국어가 영어보다 토큰을 더 많이 쓴다.
- **핵심 키워드**: 토큰, 컨텍스트 윈도우, 토큰 제한, 토큰화(Tokenization)
- **시각화 연결**: [II-1-2 시각화 보기](part2/II-1-2.html) — 문장 → 토큰 분해 + 윈도우 크기 제한 (5단계)
- **다음 섹션**: II-2-1

## Chapter II-2. 프롬프트 엔지니어링

### II-2-1. 프롬프트의 구조 (System/User/Assistant)

- **난이도**: ★★☆
- **선수 지식**: II-1-2
- **학습 목표**: System, User, Assistant 메시지의 역할을 구분하고 구성할 수 있다
- **설명**:
  **연극 대본**을 생각해보자.
  - **System 메시지** = 무대 설정. "너는 친절한 요리사야. 항상 한국어로 대답해." 이건 관객(유저)은 안 보지만, 배우(LLM)의 연기 방향을 정한다.
  - **User 메시지** = 관객의 질문. "파스타 만드는 법 알려줘."
  - **Assistant 메시지** = 배우의 대사. LLM이 생성한 응답.
  이 세 가지가 배열(messages)로 API에 전달된다. 대화가 이어지면 User/Assistant 메시지가 번갈아 쌓인다.
- **핵심 키워드**: System message, User message, Assistant message, messages 배열
- **시각화 연결**: [II-2-1 시각화 보기](part2/II-2-1.html) — 메시지 구조 + JSON 배열 코드 (5단계)
- **다음 섹션**: II-2-2

### II-2-2. API 호출 기본

- **난이도**: ★★☆
- **선수 지식**: II-2-1
- **학습 목표**: OpenAI/Anthropic API를 호출하는 기본 코드를 읽을 수 있다
- **설명**:
  **식당 주문서**를 생각해보자.
  - **메뉴 (model)**: "gpt-4o" 또는 "claude-sonnet-4-20250514" — 어떤 모델을 쓸지
  - **요청사항 (messages)**: System + User 메시지 배열
  - **추가옵션 (temperature)**: 0이면 정확하게, 1이면 창의적으로
  - **최대 분량 (max_tokens)**: 응답의 최대 길이
  주문서(요청)를 보내면 요리(응답)가 나온다. API 키는 신용카드 같은 것 — 인증과 과금에 사용된다.
- **핵심 키워드**: API Key, model, messages, temperature, max_tokens
- **시각화 연결**: [II-2-2 시각화 보기](part2/II-2-2.html) — 요청/응답 흐름 + Python API 호출 코드 (5단계)
- **다음 섹션**: II-3-1

## Chapter II-3. LLM의 한계

### II-3-1. 환각, 지식 단절, 행동 불가

- **난이도**: ★☆☆
- **선수 지식**: II-2-2
- **학습 목표**: LLM의 3가지 주요 한계와 Agent가 이를 보완하는 방식을 설명할 수 있다
- **설명**:
  **시험을 보는 학생**을 상상해보자.
  - **환각 (Hallucination)**: 모르는 문제에 자신감 넘치게 틀린 답을 쓴다. LLM도 모르면 지어낸다.
  - **지식 단절 (Knowledge Cutoff)**: 시험 범위 밖의 문제가 나온다. LLM은 학습 시점 이후의 정보를 모른다.
  - **행동 불가 (No Action)**: 답안지에 "계산기를 쓰면 됩니다"라고만 쓰고, 실제로 계산은 안 한다. LLM은 말만 하고 실행은 못 한다.
  Agent의 해결책: Tools(도구)로 검색하고 실행, Memory로 최신 정보 보완, Planning으로 검증 단계 추가.
- **핵심 키워드**: Hallucination, Knowledge Cutoff, Grounding, Tool Use
- **시각화 연결**: [II-3-1 시각화 보기](part2/II-3-1.html) — 3개 한계 → Agent 해결책 매핑 (5단계)
- **다음 섹션**: III-1-1

# Part III. 핵심 패턴

> Agent의 "사고방식"을 결정하는 핵심 패턴들이다. Chain of Thought는 에이전트가 생각을 드러내는 방법, ReAct는 생각과 행동을 번갈아 하는 패턴, Tool Use는 외부 도구를 쓰는 방법이다. 이 Part가 AI Agent의 심장이다.

## Chapter III-1. Chain of Thought (CoT)

### III-1-1. 생각의 사슬: 단계적 추론

- **난이도**: ★★☆
- **선수 지식**: II-3-1
- **학습 목표**: CoT가 LLM의 추론 정확도를 높이는 원리를 설명할 수 있다
- **설명**:
  **수학 시험**을 생각해보자. "정답: 42"만 쓰면 실수할 확률이 높다. 하지만 "풀이: A=5, B=A*3=15, C=B+27=42. 정답: 42"처럼 과정을 쓰면 중간에 실수를 잡을 수 있다.
  LLM도 마찬가지다. "단계별로 생각해봐 (Let's think step by step)"라고 말해주면, 추론 과정을 하나씩 펼치면서 정확도가 올라간다. 이것이 Chain of Thought다.
- **핵심 키워드**: Chain of Thought, 단계적 추론, Zero-shot CoT, 정확도 향상
- **시각화 연결**: [III-1-1 시각화 보기](part3/III-1-1.html) — 직접 답변 vs CoT 답변 비교 (5단계)
- **다음 섹션**: III-1-2

### III-1-2. Few-shot CoT

- **난이도**: ★★☆
- **선수 지식**: III-1-1
- **학습 목표**: Few-shot 예시를 포함한 CoT 프롬프트를 작성할 수 있다
- **설명**:
  **선생님이 예제를 풀어주는 것**과 같다. "이 문제는 이렇게 풀어. 1단계: ..., 2단계: ..., 답: ..." 이런 예시를 2~3개 보여주고 나서 "자, 이제 이 문제를 같은 방식으로 풀어봐"라고 한다.
  LLM에게도 정답까지의 사고 과정을 포함한 예시(few-shot)를 주면, 새로운 문제에서도 같은 패턴으로 추론한다. Zero-shot CoT(예시 없이 "단계별로 생각해")보다 정확도가 높다.
- **핵심 키워드**: Few-shot, 예시 기반 학습, 프롬프트 템플릿, In-Context Learning
- **시각화 연결**: [III-1-2 시각화 보기](part3/III-1-2.html) — Few-shot 프롬프트 구조 + 코드 (5단계)
- **다음 섹션**: III-2-1

---

## Chapter III-2. ReAct 패턴

### III-2-1. ReAct란: Reasoning + Acting

- **난이도**: ★★☆
- **선수 지식**: III-1-1
- **학습 목표**: ReAct 패턴의 Thought → Action → Observation 루프를 설명할 수 있다
- **설명**:
  **형사의 수사**를 생각해보자.
  1. **Thought (추론)**: "피해자의 마지막 통화 기록을 확인해야겠다."
  2. **Action (행동)**: 통신사에 통화 기록을 요청한다.
  3. **Observation (관찰)**: 기록을 받아서 읽는다. "마지막 통화 상대가 김씨로군."
  4. **Thought**: "김씨의 알리바이를 확인해야겠다."
  5. **Action**: 김씨를 찾아가 질문한다.
  6. **Observation**: ...
  이렇게 **생각하고 행동하고 관찰하기를 반복**하며 문제를 해결한다. ReAct = Reasoning(생각) + Acting(행동)의 합성어다.
- **핵심 키워드**: ReAct, Thought, Action, Observation, 추론-행동 루프
- **시각화 연결**: [III-2-1 시각화 보기](part3/III-2-1.html) — Thought → Action → Observation 순환 루프 (6단계)
- **다음 섹션**: III-2-2

### III-2-2. ReAct 실행 흐름

- **난이도**: ★★★
- **선수 지식**: III-2-1
- **학습 목표**: ReAct 루프를 코드로 구현하는 흐름을 추적할 수 있다
- **설명**:
  ReAct를 코드로 옮기면 `while True` 루프가 된다.
  ```
  while True:
      thought = llm.think(관찰 결과)      # 1. 추론
      action = llm.decide_action(thought)  # 2. 행동 결정
      observation = execute(action)        # 3. 행동 실행 + 결과 관찰
      if action == "finish":               # 4. 종료 조건
          break
  ```
  매 반복마다 LLM이 "지금까지의 관찰을 바탕으로 다음에 뭘 할지" 결정한다. 종료 조건을 만나면 최종 답변을 반환한다.
- **핵심 키워드**: while 루프, LLM 호출, Action 파싱, 종료 조건, 최대 반복
- **시각화 연결**: [III-2-2 시각화 보기](part3/III-2-2.html) — ReAct 루프 + pseudo-Python 코드 (6단계)
- **다음 섹션**: III-3-1

## Chapter III-3. Tool Use

### III-3-1. Tool Use란: LLM이 도구를 쓴다

- **난이도**: ★★☆
- **선수 지식**: III-2-1
- **학습 목표**: Function Calling / Tool Use의 개념과 동작 원리를 설명할 수 있다
- **설명**:
  **사무실 직원**을 생각해보자. 직원이 있으면:
  - "매출 합계 구해야지" → **계산기**를 쓴다
  - "경쟁사 가격 알아봐야지" → **검색엔진**을 쓴다
  - "고객에게 알려줘야지" → **이메일**을 보낸다
  직원(LLM)은 도구 목록을 보고, 상황에 맞는 도구를 **골라서**, 올바른 입력값으로 **호출**한다. 결과를 받아서 다음 판단에 활용한다.
  이것이 Tool Use (= Function Calling)다.
- **핵심 키워드**: Tool Use, Function Calling, 도구 선택, 파라미터 추출
- **시각화 연결**: [III-3-1 시각화 보기](part3/III-3-1.html) — LLM의 도구 선택 → 호출 → 결과 수신 흐름 (5단계)
- **다음 섹션**: III-3-2

### III-3-2. Function Calling 상세

- **난이도**: ★★★
- **선수 지식**: III-3-1
- **학습 목표**: JSON Schema로 도구를 정의하고 LLM에 전달하는 코드를 읽을 수 있다
- **설명**:
  LLM에게 도구를 쓰게 하려면, 먼저 **도구 사용 설명서**를 줘야 한다. 이 설명서가 JSON Schema다.
  ```json
  {
    "name": "get_weather",
    "description": "특정 도시의 현재 날씨를 조회한다",
    "parameters": {
      "city": { "type": "string", "description": "도시명" }
    }
  }
  ```
  LLM은 이 설명서를 읽고, "지금 날씨를 알아야 하니까 get_weather를 city='서울'로 호출하자"고 판단한다. 개발자는 실제 API를 호출하고, 결과를 LLM에게 다시 전달한다.
- **핵심 키워드**: JSON Schema, tools 배열, function_call, tool_choice, 파라미터 정의
- **시각화 연결**: [III-3-2 시각화 보기](part3/III-3-2.html) — 도구 정의 + 호출 흐름 + 코드 (6단계)
- **다음 섹션**: III-3-3

### III-3-3. Tool Use + ReAct 통합

- **난이도**: ★★★
- **선수 지식**: III-2-2, III-3-2
- **학습 목표**: ReAct 루프 안에서 Tool Use가 어떻게 결합되는지 전체 흐름을 추적할 수 있다
- **설명**:
  **형사가 감식 도구를 사용하는 수사**를 생각해보자. 형사(ReAct)는 추리하면서, 필요할 때 지문 분석기, DNA 검사기 등의 도구(Tool Use)를 쓴다.
  코드에서는:
  1. ReAct 루프가 돌면서 LLM이 "Thought"를 생성한다
  2. LLM이 "이 상황에서는 검색 도구를 써야겠다"고 판단하면 → Tool Call 생성
  3. 개발자 코드가 실제 도구를 실행하고 결과를 반환한다
  4. LLM이 결과를 Observation으로 받아서 다음 Thought를 생성한다
  이것이 현대 AI Agent의 표준 아키텍처다.
- **핵심 키워드**: ReAct + Tool Use, Agent 루프, 통합 아키텍처, 표준 패턴
- **시각화 연결**: [III-3-3 시각화 보기](part3/III-3-3.html) — ReAct + Tool Use 통합 아키텍처 + 코드 (6단계)
- **다음 섹션**: IV-1-1

# Part IV. MCP (Model Context Protocol)

> MCP는 AI Agent가 외부 시스템(파일, DB, API 등)과 소통하는 **표준화된 프로토콜**이다. USB-C가 다양한 기기를 하나의 포트로 연결하듯, MCP는 다양한 도구/데이터 소스를 하나의 규격으로 연결한다.

## Chapter IV-1. MCP 개요

### IV-1-1. MCP란 무엇인가

- **난이도**: ★★☆
- **선수 지식**: III-3-1
- **학습 목표**: MCP의 목적과 기존 Tool Use와의 차이를 설명할 수 있다
- **설명**:
  **USB-C**를 생각해보자. 예전에는 충전기가 기기마다 달랐다 — 아이폰은 Lightning, 안드로이드는 Micro-USB, 노트북은 전용 어댑터. 기기를 바꿀 때마다 충전기를 새로 사야 했다.
  USB-C는 이 문제를 해결했다. 하나의 포트, 하나의 케이블로 모든 기기를 연결한다.
  MCP도 같다. 기존에는 Tool마다 커스텀 연동 코드를 작성해야 했다 (스파게티). MCP는 **표준 프로토콜**로 어떤 도구든 같은 방식으로 연결한다.
- **핵심 키워드**: MCP, 표준화, 프로토콜, 상호운용성, Anthropic
- **시각화 연결**: [IV-1-1 시각화 보기](part4/IV-1-1.html) — 커스텀 통합(스파게티) vs MCP 표준화 비교 (5단계)
- **다음 섹션**: IV-1-2

### IV-1-2. MCP 아키텍처: Host, Client, Server

- **난이도**: ★★☆
- **선수 지식**: IV-1-1
- **학습 목표**: MCP의 Host, Client, Server 3계층을 구분하고 역할을 설명할 수 있다
- **설명**:
  **호텔**을 생각해보자.
  - **Host (호텔 건물)**: Claude Desktop, VS Code 같은 AI 애플리케이션. 전체를 감싸는 컨테이너.
  - **Client (프런트 데스크)**: Host 안에서 서버와 1:1로 통신하는 연결 관리자.
  - **Server (서비스 부서)**: 세탁실, 룸서비스, 택시 — 각각 독립적인 기능을 제공하는 서버.
  투숙객(LLM)이 프런트(Client)에 "세탁 좀 해주세요" 하면, 프런트가 세탁실(Server)에 연락한다. 결과가 오면 프런트가 투숙객에게 전달한다.
- **핵심 키워드**: Host, Client, Server, 3계층, Transport
- **시각화 연결**: [IV-1-2 시각화 보기](part4/IV-1-2.html) — 3계층 아키텍처 + 요청 흐름 (5단계)
- **다음 섹션**: IV-2-1

---

## Chapter IV-2. MCP 구성요소

### IV-2-1. Resources, Tools, Prompts

- **난이도**: ★★☆
- **선수 지식**: IV-1-2
- **학습 목표**: MCP의 3가지 핵심 primitive의 역할을 구분할 수 있다
- **설명**:
  **도서관**을 생각해보자.
  - **Resources (책)**: 읽기 전용 데이터. 파일 내용, DB 레코드, API 응답 등. LLM이 참고 자료로 읽는다.
  - **Tools (복사기/프린터)**: 행동을 수행하는 기능. 파일 쓰기, 이메일 발송, 코드 실행 등. 부작용(side effect)이 있다.
  - **Prompts (사서의 추천 목록)**: 자주 쓰는 프롬프트 템플릿. "이 코드를 리뷰해줘" 같은 사전 정의된 작업 흐름.
  Resources는 읽기 전용(안전), Tools는 실행(주의 필요), Prompts는 편의 기능이다.
- **핵심 키워드**: Resources, Tools, Prompts, Primitive, 읽기 전용 vs 실행
- **시각화 연결**: [IV-2-1 시각화 보기](part4/IV-2-1.html) — 3가지 primitive 카드 + 역할 비교 (5단계)
- **다음 섹션**: IV-2-2

### IV-2-2. MCP 통신 흐름

- **난이도**: ★★★
- **선수 지식**: IV-2-1
- **학습 목표**: JSON-RPC 기반 MCP 메시지가 주고받아지는 과정을 추적할 수 있다
- **설명**:
  **편지 교환**을 생각해보자. 봉투(JSON-RPC)에 요청을 담아 보내면, 상대가 같은 형식으로 응답을 보내준다.
  MCP는 JSON-RPC 2.0 프로토콜을 사용한다:
  - **요청**: `{ "jsonrpc": "2.0", "method": "tools/call", "params": {...}, "id": 1 }`
  - **응답**: `{ "jsonrpc": "2.0", "result": {...}, "id": 1 }`
  Transport는 두 가지: stdio(같은 컴퓨터) 또는 HTTP+SSE(원격 서버). 초기화 → 기능 협상 → 요청/응답 반복.
- **핵심 키워드**: JSON-RPC, stdio, HTTP+SSE, 초기화, 기능 협상
- **시각화 연결**: [IV-2-2 시각화 보기](part4/IV-2-2.html) — Client-Server 시퀀스 + JSON-RPC 코드 (6단계)
- **다음 섹션**: IV-3-1

## Chapter V-1. 에이전트 설계

### V-1-1. Single Agent 설계 패턴

- **난이도**: ★★★
- **선수 지식**: III-3-3, IV-2-1
- **학습 목표**: Router, Orchestrator, Evaluator 패턴을 비교하고 상황에 맞게 선택할 수 있다
- **설명**:
  에이전트를 어떤 구조로 만들지 결정해야 한다. 3가지 대표 패턴이 있다.
  - **Router (교통 정리)**: 요청을 분류해서 적절한 처리기로 보낸다. 1인 식당에서 "이건 구이, 이건 튀김" 판단하는 것.
  - **Orchestrator (지휘자)**: 복잡한 작업을 여러 단계로 분해하고, 각 단계를 순서대로 실행한다. 오케스트라 지휘자가 악기별 타이밍을 조율하는 것.
  - **Evaluator-Optimizer (품질검사원)**: 결과를 평가하고, 기준에 못 미치면 다시 시도한다. 공장 품질검사에서 불량품을 걸러내는 것.
- **핵심 키워드**: Router, Orchestrator, Evaluator-Optimizer, 설계 패턴
- **시각화 연결**: [V-1-1 시각화 보기](part5/V-1-1.html) — 3가지 패턴 비교 다이어그램 (6단계)
- **다음 섹션**: V-1-2

### V-1-2. 메모리와 상태 관리

- **난이도**: ★★★
- **선수 지식**: V-1-1
- **학습 목표**: Short-term / Long-term Memory의 차이와 구현 전략을 설명할 수 있다
- **설명**:
  - **화이트보드 (Short-term Memory)**: 현재 대화에서만 유지되는 기억. 대화가 끝나면 지워진다. 컨텍스트 윈도우 안에 있는 대화 이력이 이것이다.
  - **서랍장 (Long-term Memory)**: 대화가 끝나도 보존되는 기억. 벡터 DB에 저장하거나, 파일에 기록한다.
  - **실전 전략**: 짧은 대화는 화이트보드로 충분, 긴 대화나 사용자 맞춤은 서랍장 필요. 검색(RAG) 방식으로 관련 기억만 꺼내 쓴다.
- **핵심 키워드**: Short-term Memory, Long-term Memory, 대화 이력, 벡터 DB, RAG

### V-2-1. 에이전트 루프 구현

- **난이도**: ★★★

- **학습 목표**: ReAct + Tool Use + Memory를 결합한 에이전트 메인 루프 코드를 읽을 수 있다
- **설명**:
  **자동차 엔진의 4행정 사이클**과 같다.
  1. **흡입 (입력)**: 사용자 메시지 + 메모리에서 관련 정보 검색
  2. **압축 (추론)**: LLM이 현재 상황을 분석하고 다음 행동을 결정
  3. **폭발 (행동)**: 도구 호출, 코드 실행 등 실제 행동 수행
  4. **배기 (결과)**: 결과를 메모리에 저장하고 사용자에게 반환
  이 사이클이 종료 조건을 만날 때까지 반복된다.
- **핵심 키워드**: 에이전트 루프, 입력→추론→행동→결과, 종료 조건, 최대 반복

### V-2-2. Multi-Agent 개요

- **난이도**: ★★★
- **선수 지식**: V-2-1
- **학습 목표**: Multi-Agent 패턴(Supervisor, Peer-to-Peer)의 개념을 설명할 수 있다
- **설명**:
  **회사 조직도**를 생각해보자.
  - **Supervisor 패턴 (팀장-팀원)**: 팀장(Supervisor Agent)이 업무를 분배하고, 팀원(Sub-agent)이 실행한다. 팀장이 결과를 종합해서 최종 보고한다. 예: 리서치 에이전트 + 코딩 에이전트 + 리뷰 에이전트를 관리하는 PM 에이전트.
  - **Peer-to-Peer 패턴 (동료 협업)**: 동등한 에이전트들이 서로 메시지를 주고받으며 협업한다. 토론처럼 서로의 결과를 검증한다.
  복잡한 문제는 하나의 에이전트로 해결하기 어렵다. 역할을 나누면 각 에이전트가 전문화된 작업을 더 잘 수행한다.
- **핵심 키워드**: Multi-Agent, Supervisor, Peer-to-Peer, 역할 분담, 협업
