## 이 자료가 만들어진 이유

"클라우드를 써야 한다"는 말을 들었을 때, 처음에는 무엇부터 시작해야 할지 막막하다. 콘솔 화면은 복잡하고, 서비스 이름은 낯설고, 가격 정책은 어렵다. 이 자료는 그 막막함을 없애기 위해 만들었다.

Part I에서는 "왜 클라우드를 써야 하는가"부터 시작한다. 코딩 지식이 없어도 읽을 수 있다. Part IV까지 오면 실제 Cloud Run에 서비스를 배포할 수 있고, Part VII까지 마치면 실무 수준의 GCP 운영 능력을 갖추게 된다.

각 Section은 **설명 + 비유 + 핵심 키워드**로 구성되어, 글을 읽으며 자연스럽게 개념이 머릿속에 쌓이도록 설계되어 있다.

## Part I. 왜 클라우드인가

> 내 집 서버 vs 전기세 걱정 없는 호텔. 클라우드가 무엇인지, 왜 필요한지를 비유로 이해한다.

---

### I-1-1. 내 서버를 직접 운영한다는 것

- **난이도**: ★☆☆
- **선수 지식**: 없음
- **학습 목표**: 이 Section을 마치면 "온프레미스(On-Premise)"가 무엇인지, 왜 비용과 관리 부담이 큰지 설명할 수 있다.
- **설명**: 집을 직접 짓는다고 상상해 보자. 땅을 사고, 설계도를 그리고, 건물을 올리고, 수도와 전기를 연결해야 한다. 완공 뒤에도 끝이 아니다. 보일러가 고장 나면 직접 수리하고, 지붕이 새면 직접 올라가야 한다. 회사가 서버를 직접 사서 운영하는 방식, 즉 **온프레미스(On-Premise)**가 정확히 이 구조다.

  온프레미스는 서버 장비를 구입해 회사 내부 또는 임대한 **데이터센터(Data Center)** 공간에 설치하고, 네트워크·전력·냉각까지 직접 구성하는 방식이다. 장비 구입처럼 한 번에 큰돈이 나가는 비용을 **초기 투자 비용(CapEx, Capital Expenditure)**이라 부르고, 전기세·유지보수·운영 인력처럼 매달 꾸준히 나가는 비용을 **운영 비용(OpEx, Operating Expenditure)**이라 부른다. 온프레미스는 이 두 가지 비용을 모두 회사가 직접 감당한다.

  문제는 유연성이다. 연말 이벤트나 신제품 출시로 트래픽이 갑자기 10배가 되어도, 새 서버를 주문하고 설치하고 설정하는 데는 몇 주가 걸린다. 반대로 한산한 시기에는 비싸게 사들인 서버가 전력만 소모하며 놀고 있다. 실무에서 온프레미스로 서비스를 운영해 본 개발자라면 "서버가 부족해서 배포를 미룬 경험"이 한 번쯤 있을 것이다.

  클라우드는 이 구조를 호텔로 바꾼다. 방이 더 필요하면 그날 바로 예약하면 되고, 수도·전기·청소는 호텔 측이 알아서 처리한다. 온프레미스의 한계를 이해하면, 다음 Section에서 클라우드가 왜 등장했는지 자연스럽게 이해된다.
- **핵심 키워드**: 온프레미스(On-Premise), 데이터센터(Data Center), 서버(Server), 초기 투자 비용(CapEx), 운영 비용(OpEx)
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-1-2

---

### I-1-2. 클라우드가 해결하는 세 가지 문제

- **난이도**: ★☆☆
- **선수 지식**: I-1-1
- **학습 목표**: 이 Section을 마치면 클라우드가 해결하는 핵심 문제 세 가지(초기 비용, 확장성, 가용성)를 설명할 수 있다.
- **설명**: 호텔에 묵는다고 생각해 보자. 체크인 당일 방이 부족하면 그 자리에서 룸을 추가할 수 있다. 다음 달 출장이 없으면 예약하지 않으면 된다. 전구가 나가도 프런트에 전화 한 통이면 해결된다. 클라우드는 서버 세계에서 이 경험을 가능하게 만드는 기술이다. 구체적으로 세 가지 문제를 해결한다.

  첫째, **초기 비용** 문제다. 온프레미스에서는 서비스 시작 전부터 서버 장비를 구입해야 한다. 클라우드는 서버를 사는 대신 빌린다. 종량제(Pay-as-you-go) 방식으로 실제로 사용한 만큼만 내기 때문에, 아이디어를 검증하는 초기 단계에서도 큰 초기 투자 없이 시작할 수 있다. 둘째, **확장성(Scalability)** 문제다. 서비스가 갑자기 인기를 얻어 트래픽이 급증해도, **Auto Scaling** 기능이 자동으로 서버 수를 늘려준다. 개발자가 새벽에 일어나 서버를 손수 추가할 필요가 없다. 셋째, **가용성(Availability)** 문제다. AWS, Google, Microsoft 같은 클라우드 제공자는 전 세계 수십 개 **리전(Region)**에 데이터센터를 분산 운영한다. 서울 리전에 장애가 생겨도 일본 리전이 요청을 이어받을 수 있어, 한 곳의 고장이 전체 서비스 중단으로 이어지지 않는다.

  주니어 개발자에게 가장 체감되는 부분은 확장성이다. 로컬에서 잘 돌아가던 앱을 배포한 뒤, 갑작스러운 트래픽 급증으로 서버가 멈추는 상황을 겪으면 Auto Scaling의 가치를 바로 느낄 수 있다. 클라우드 환경에서는 서버 증설이 코드 한 줄 또는 콘솔 클릭 한 번으로 해결된다.

  이 세 가지 — 초기 비용 절감, 확장성, 가용성 — 가 클라우드를 선택하는 핵심 이유다. 다음 Section에서는 클라우드 서비스를 어떤 방식으로 빌려 쓰느냐에 따른 세 가지 형태(IaaS/PaaS/SaaS)를 살펴본다.
- **핵심 키워드**: 확장성(Scalability), 가용성(Availability), 리전(Region), Auto Scaling, 종량제(Pay-as-you-go)
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-1-3

### I-1-3. IaaS / PaaS / SaaS — 클라우드의 세 가지 형태

- **난이도**: ★☆☆
- **선수 지식**: I-1-2
- **학습 목표**: 이 Section을 마치면 IaaS·PaaS·SaaS의 차이를 피자 비유로 설명할 수 있다.
- **설명**: 피자를 먹고 싶을 때 선택지는 세 가지다. 밀가루와 토마토소스부터 직접 사서 오븐을 켜거나, 반죽이 미리 준비된 피자 키트를 받아서 원하는 토핑만 올리거나, 완성된 피자를 앱으로 주문해 받거나. 클라우드 서비스도 이 세 가지 방식으로 나뉜다.

  클라우드에서 "얼마나 직접 관리하느냐"에 따라 서비스 형태가 달라진다. **IaaS(Infrastructure as a Service)**는 가상 머신(VM)을 빌려서 운영체제(OS)와 미들웨어를 직접 설치하고 관리하는 방식이다. 피자 재료를 사다가 직접 굽는 것처럼, 인프라에 대한 통제권이 높지만 그만큼 할 일이 많다. **PaaS(Platform as a Service)**는 반죽과 소스까지 준비된 피자 키트처럼, 인프라 걱정 없이 코드만 올리면 플랫폼이 배포와 실행을 대신 처리해 준다. **SaaS(Software as a Service)**는 완성된 피자를 배달받는 것이다. Gmail, Google Docs처럼 설치도 설정도 필요 없이 브라우저에서 바로 쓴다.

  관리 범위의 차이가 실무에서 어떻게 드러나는지 보면 더 명확하다. GCP Compute Engine은 대표적인 IaaS로, VM을 직접 만들고 OS 패치부터 네트워크 설정까지 개발자가 책임진다. 반면 Cloud Run은 PaaS에 해당한다. 컨테이너 이미지를 올리면 플랫폼이 서버 관리, 스케일링, 트래픽 처리를 알아서 담당한다. 개발자는 코드와 비즈니스 로직에만 집중할 수 있다. 이 교재에서 다루는 GCP의 핵심 서비스가 Cloud Run인 이유도 여기에 있다.

  IaaS, PaaS, SaaS는 통제권과 편의성의 교환이다. 통제가 많을수록 자유롭지만 관리 부담도 커진다. 어느 형태를 선택하느냐는 팀의 규모와 역량, 서비스의 특성에 따라 달라진다. 다음 파트에서는 클라우드를 실제로 쓸 때 바로 만나는 문제인 비용 구조를 살펴본다.
- **핵심 키워드**: IaaS, PaaS, SaaS, 가상 머신(VM), 관리 범위
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-2-1

### I-2-1. 클라우드 비용 구조 이해하기

- **난이도**: ★☆☆
- **선수 지식**: I-1-2
- **학습 목표**: 이 Section을 마치면 클라우드 종량제 비용 구조와 무료 체험(Free Tier)을 이해하고, 예상치 못한 청구를 피하는 방법을 알 수 있다.
- **설명**: 전기를 쓴 만큼 요금이 나오고 월말에 청구서가 날아오는 것처럼, 클라우드도 사용한 만큼 비용이 발생한다. 이것을 종량제(Pay-as-you-go)라고 부른다. 서버를 미리 사서 데이터센터에 쌓아 두는 방식과 달리, 클라우드는 필요한 순간에만 자원을 빌리고 쓰지 않으면 비용이 멈춘다.

  GCP는 새로 가입한 계정에 $300 크레딧을 90일간 제공한다. 이 기간 동안 대부분의 서비스를 무료로 실험해 볼 수 있다. 크레딧을 다 쓰거나 90일이 지나도 Cloud Run을 비롯한 일부 서비스는 Always Free 한도 안에서 영구 무료로 사용할 수 있다. Free Tier는 학습과 소규모 프로젝트를 시작하는 데 충분한 여유를 준다.

  그러나 종량제에는 함정이 있다. 테스트용으로 만든 VM을 끄지 않고 방치하거나, 대용량 데이터를 실수로 저장하면 예상하지 못한 청구서가 나온다. 이를 막는 가장 확실한 방법은 **예산 알림(Budget Alert)**을 설정하는 것이다. 설정한 금액에 근접하면 이메일로 알림이 오므로, 초과 청구를 사전에 차단할 수 있다. Cloud Run은 트래픽이 없을 때 인스턴스가 0으로 줄어드는 **Scale to Zero** 기능을 기본 제공하므로, 사용하지 않는 시간에도 서버를 켜 두는 낭비가 없다.

  클라우드 비용의 핵심은 "끄면 안 내도 된다"는 단순한 원리에 있다. Budget Alert 설정과 Scale to Zero 구조를 이해하고 나면, 비용 걱정 없이 GCP를 실습할 수 있다. 다음 파트에서는 AWS·Azure·GCP 세 클라우드의 포지션을 비교하며 GCP를 선택하는 이유를 살펴본다.
- **핵심 키워드**: 종량제, Free Tier, Always Free, Budget Alert, Scale to Zero
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-1-1

## Part II. 클라우드 비교 (AWS / Azure / GCP)

> 세 클라우드의 포지션과 강점을 비교해 GCP를 선택하는 이유를 이해한다.

---

### II-1-1. 세 공룡의 포지션

- **난이도**: ★☆☆
- **선수 지식**: I-1-3
- **학습 목표**: 이 Section을 마치면 AWS·Azure·GCP의 시장 위치와 대표 강점을 한 문장씩 설명할 수 있다.
- **설명**: 클라우드 시장은 마치 세 마리 공룡이 영역을 나눠 차지한 생태계처럼 보인다. 덩치와 성격이 제각각인 세 공룡, 즉 AWS·Azure·GCP는 저마다 다른 무기를 앞세워 시장을 분할하고 있다. 어떤 공룡이 옳고 그른 것이 아니라, 누가 무엇을 잘하는지를 파악하는 것이 출발점이다.

  첫 번째 공룡인 **AWS(Amazon Web Services)**는 2006년에 가장 먼저 시장을 개척한 맏형이다. 서비스 수가 세 클라우드 중 가장 많고, 커뮤니티와 레퍼런스가 풍부해 "일단 뭘 만들어야 하는데 어디서 시작할지 모르겠다"는 팀이 가장 많이 선택한다. 두 번째 공룡 **Azure(Microsoft Azure)**는 기업 사무실의 오랜 파트너다. Windows Server와 Active Directory(사내 계정·권한 관리 시스템)를 이미 쓰는 기업이라면 Azure로의 이동이 자연스럽다. Office 365, Teams, 사내 시스템과의 연동이 매끄럽기 때문이다. 세 번째 공룡 **GCP(Google Cloud Platform)**는 데이터·AI·Kubernetes(쿠버네티스, 컨테이너 실행을 자동 관리하는 도구)의 본고장이다. 구글이 내부에서 실제로 쓰던 인프라 기술을 클라우드 서비스로 공개한 것이 많아, 데이터 분석이나 머신러닝 파이프라인을 주력으로 다루는 팀에게 강하다.

  실무에서 클라우드를 선택할 때는 단순히 시장 점유율만 보고 결정하지 않는다. 팀이 이미 익숙한 기술 스택이 무엇인지, 당장 필요한 서비스가 어느 공룡에게서 더 잘 제공되는지, 그리고 장기적으로 비용을 어떻게 맞출 것인지를 함께 고려해야 한다. 예를 들어 사내에 .NET 기반 시스템이 많다면 Azure가 이미 반 이상 답이고, 머신러닝 모델을 대규모로 굴려야 한다면 GCP가 유력 후보에 오른다.

  세 공룡의 이름과 성격을 머릿속에 고정해두는 것만으로도 클라우드 기술 뉴스나 채용 공고를 읽을 때 맥락이 보이기 시작한다. 다음 Section에서는 이 중 GCP가 특히 강점을 가지는 세 영역, 데이터·AI·컨테이너를 집중적으로 살펴본다.
- **핵심 키워드**: AWS, Azure, GCP, 시장 점유율, Kubernetes(쿠버네티스)
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-1-2

### II-1-2. GCP의 특화 영역 — 데이터 · AI · 컨테이너

- **난이도**: ★★☆
- **선수 지식**: II-1-1
- **학습 목표**: 이 Section을 마치면 BigQuery·Vertex AI·GKE가 GCP에서 갖는 위치를 설명할 수 있다.
- **설명**: GCP의 서비스 상당수는 구글이 자사 서비스를 운영하면서 직접 만들어 쓰던 내부 기술을 외부에 공개한 것이다. 구글 검색, 유튜브, 지메일을 돌리려면 수십억 건의 요청을 매일 처리해야 한다. 그 과정에서 탄생한 기술들이 지금 GCP의 핵심 서비스로 들어와 있다.

  가장 먼저 꼽히는 것이 **BigQuery**다. BigQuery는 페타바이트(PB, 1PB는 약 100만 GB) 규모의 데이터를 SQL(데이터베이스 조회 언어)로 분석할 수 있는 **관리형 서비스(Managed Service)**로, 원래 구글 광고 분석팀이 내부에서 쓰던 시스템에서 출발했다. 서버를 직접 띄우거나 관리할 필요 없이 SQL 쿼리만 날리면 수억 행의 데이터도 수초 안에 결과가 나온다. 다음은 **Vertex AI**다. 머신러닝 모델을 만들면 학습, 배포, 모니터링 단계가 각각 따로 관리되기 쉬운데, Vertex AI는 이 세 단계를 하나의 플랫폼에서 통합해 관리한다. 마지막으로 **GKE(Google Kubernetes Engine)**는 Kubernetes를 만든 구글이 직접 운영하는 관리형 쿠버네티스 클러스터 서비스다. 오픈소스 Kubernetes를 직접 설치하고 유지보수하는 수고 없이, GKE 위에서 컨테이너 애플리케이션을 바로 배포하고 운영할 수 있다.

  주니어 개발자 입장에서 실감할 수 있는 장면을 하나 떠올려보자. 로그 데이터 수백만 건을 분석해야 하는데, 로컬 PC에서 pandas로 돌리면 몇 시간이 걸릴 수 있다. 이때 BigQuery에 데이터를 올리고 SQL을 실행하면 같은 작업이 1분 이내로 끝나기도 한다. 이 경험이 GCP를 선택하는 이유의 핵심이다.

  BigQuery·Vertex AI·GKE, 이 세 서비스가 GCP를 선택하는 이유의 대부분을 차지한다. 이 세 가지가 GCP의 얼굴이라고 기억해두면 된다. 다음 Section에서는 이미 AWS를 써본 독자를 위해 AWS 서비스 이름과 GCP 서비스 이름을 나란히 비교하는 대응표를 살펴본다.
- **핵심 키워드**: BigQuery, Vertex AI, GKE(Google Kubernetes Engine), 관리형 서비스(Managed Service)
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-1-3

### II-1-3. 주요 서비스 대응표 (AWS vs GCP)

- **난이도**: ★★☆
- **선수 지식**: II-1-1
- **학습 목표**: 이 Section을 마치면 AWS 주요 서비스에 대응하는 GCP 서비스를 찾아볼 수 있다.
- **설명**: AWS를 먼저 배운 개발자가 GCP로 넘어올 때 가장 먼저 맞닥뜨리는 벽은 "이게 AWS에서 뭐에 해당하지?"라는 혼란이다. 서비스 이름이 완전히 다르기 때문이다. AWS에서 S3라고 부르던 것을 GCP는 Cloud Storage라고 부르고, Lambda는 Cloud Functions, EKS는 GKE가 된다. 구현 방식에 세부적인 차이가 있는 경우도 있지만, 역할과 개념은 거의 동일하고 이름만 바뀐 경우가 대부분이다. 아래 대응표를 한 번 훑어보면 "아, 이름만 다르구나"라는 감각이 자연스럽게 잡힌다. AWS 경험이 없는 입문자라면 오른쪽 GCP 열만 읽어도 무방하다.

| 분류 | AWS | GCP |
|:--|:--|:--|
| 서버리스 컨테이너 | ECS / App Runner | Cloud Run |
| 관리형 쿠버네티스 | EKS | GKE |
| 가상 머신 | EC2 | Compute Engine |
| 오브젝트 스토리지 | S3 | Cloud Storage |
| 관리형 관계형 DB | RDS | Cloud SQL |
| NoSQL | DynamoDB | Firestore / Bigtable |
| 서버리스 함수 | Lambda | Cloud Functions |
| 컨테이너 레지스트리 | ECR | Artifact Registry |
| 로드 밸런서 | ALB / NLB | Cloud Load Balancing |
| 로깅 | CloudWatch Logs | Cloud Logging |

이 중 GCP를 처음 쓸 때 자주 마주치는 서비스 몇 가지를 짚어두겠다. **Cloud Run**은 Docker 컨테이너를 서버 관리 없이 바로 실행하는 서비스다. 요청이 없으면 자동으로 축소되고 트래픽이 몰리면 자동으로 확장되며, 이 책의 실습에서도 가장 많이 등장한다. **Cloud SQL**은 PostgreSQL이나 MySQL 같은 관계형 데이터베이스를 GCP가 대신 운영해주는 관리형 서비스로, 서버 패치나 백업 설정을 직접 하지 않아도 된다. **Artifact Registry**는 직접 빌드한 Docker 이미지를 안전하게 저장하고 팀원이나 배포 파이프라인에서 가져다 쓸 수 있게 해주는 컨테이너 레지스트리다. **Cloud Logging**은 여러 서비스에서 발생하는 로그를 한 곳에 모아 검색하고 알림을 설정할 수 있게 해준다. 로컬에서 `print()`로 디버깅하던 습관에서 벗어나 운영 환경의 로그를 Cloud Logging에서 찾는 것이 GCP 실무의 기본 루틴이 된다.

대응표는 외울 필요가 없다. 실제 프로젝트에서 특정 기능이 필요할 때 이 표를 펼쳐 GCP 서비스 이름을 확인하면 된다. 이름을 알면 공식 문서 검색도, 팀 내 대화도, 구직 공고 해석도 훨씬 빠르게 진행된다.

- **핵심 키워드**: Cloud Run, Cloud SQL, Artifact Registry, Cloud Logging
- **시각화 연결**: 해당 없음
- **다음 섹션**: III-1-1

### III-1-1. GCP 계정 생성과 프로젝트 구조

- **난이도**: ★★☆
- **선수 지식**: 없음 (Part I, II는 개념 배경)
- **학습 목표**: 이 Section을 마치면 GCP 계정을 만들고, 프로젝트(Project)가 무엇인지 설명할 수 있다.
- **설명**: 회사 건물에 입주한다고 상상해 보자. 건물 자체는 하나지만, 각 층은 서로 다른 팀이 쓴다. 마케팅 팀 문서가 개발 팀 서랍에 섞이지 않도록 층마다 잠금장치가 있다. GCP의 **프로젝트(Project)**가 이 역할을 한다. GCP 계정 하나 아래에 여러 프로젝트를 두고, 청구·권한·리소스를 프로젝트 단위로 완전히 격리한다.

  GCP 계정을 처음 만들면 무료 체험 크레딧($300)이 자동으로 부여된다. 이 크레딧 안에서 Cloud Run, Cloud SQL 등 대부분의 서비스를 실습할 수 있다. 계정 생성과 동시에 기본 프로젝트가 자동으로 만들어지며, 이후 언제든 프로젝트를 추가할 수 있다. GCP 리소스 계층은 **조직(Organization)** — **프로젝트(Project)** — **리소스** 순서로 내려간다. 조직은 회사 전체를 묶는 최상위 단위이고, 프로젝트는 그 아래 실제 서비스나 환경을 담는 그릇이다. **청구 계정(Billing Account)**은 프로젝트에 연결해 요금을 집계한다.

  실무에서는 하나의 서비스를 `dev-project`, `staging-project`, `prod-project`로 나누는 방식이 일반적이다. 개발자가 `dev-project`에서 실험적인 설정을 바꾸더라도, 운영 환경(`prod-project`)에는 영향이 전혀 없다. 프로젝트 간 리소스는 기본적으로 격리되어 있기 때문이다. 프로젝트를 만들 때 부여되는 **프로젝트 ID**는 GCP 전체에서 고유한 식별자로, 한 번 정하면 변경할 수 없다. 신중하게 이름을 짓는 것이 좋다.

  GCP를 시작하는 첫 단계는 계정을 만들고 프로젝트 구조를 이해하는 것이다. 프로젝트가 무엇인지 이해해야 이후 CLI 설정, 리소스 배포, 권한 관리 모두 자연스럽게 이어진다. 다음 Section에서는 터미널에서 GCP를 직접 조작하는 도구인 gcloud CLI를 설치하고 설정하는 방법을 다룬다.
- **핵심 키워드**: 프로젝트(Project), 조직(Organization), 청구 계정(Billing Account), 리소스 계층
- **시각화 연결**: 해당 없음
- **다음 섹션**: III-1-2

### III-1-2. gcloud CLI 설치와 초기 설정

- **난이도**: ★★☆
- **선수 지식**: III-1-1
- **학습 목표**: 이 Section을 마치면 gcloud CLI를 설치하고 `gcloud init`으로 프로젝트·리전을 설정할 수 있다.
- **설명**: TV를 제어하는 방법은 두 가지다. 본체 버튼을 직접 누르거나 리모컨을 쓰거나. 리모컨이 있으면 소파에서 채널을 바꾸고 볼륨을 조절할 수 있다. **gcloud CLI**가 GCP에서 리모컨 역할을 한다. `console.cloud.google.com` 웹 화면을 마우스로 클릭하는 대신, 터미널에서 명령어 한 줄로 동일한 작업을 수행할 수 있다.

  gcloud CLI는 Google Cloud SDK에 포함된 명령줄 도구다. 설치 후 처음 실행하는 명령은 `gcloud init`이다. 이 명령 하나가 세 단계를 순서대로 안내한다. 첫째, 구글 계정 인증(Authentication) — 브라우저가 열리며 계정 로그인을 요청한다. 둘째, 작업할 프로젝트 선택 — 내 계정에 있는 프로젝트 목록이 나타나고 번호를 입력하면 된다. 셋째, 기본 리전 설정 — 이후 명령에서 `--region`을 매번 쓰지 않아도 되도록 기본값을 저장한다. 설정을 마쳤으면 `gcloud config list`로 현재 활성화된 계정·프로젝트·리전을 한 번에 확인할 수 있다. 프로젝트를 바꿔야 할 때는 `gcloud config set project`에 원하는 프로젝트 ID를 입력한다.

  실무에서 gcloud CLI가 빛을 발하는 순간은 반복 작업을 자동화할 때다. 콘솔 화면에서 10번 클릭해야 하는 배포 과정을 셸 스크립트 한 파일로 만들어두면, 이후에는 파일 하나를 실행하는 것으로 끝난다. CI/CD 파이프라인에서도 gcloud 명령어가 핵심으로 쓰인다. 처음에는 콘솔에서 작업 흐름을 익히고, 익숙해지면 CLI로 옮기는 방식이 효율적이다.

  gcloud CLI는 GCP를 다루는 기본 도구다. Part IV에서 Cloud Run 배포, Cloud SQL 연결 등 핵심 작업을 수행할 때 이 CLI가 계속 등장한다. CLI 설치와 `gcloud init` 설정을 마쳤으면, 다음 단계로 리소스를 어느 지역에 배치할지 결정하는 리전과 존 개념을 이해할 차례다.

  ```bash
  # gcloud CLI 설치 후 초기 설정
  gcloud init

  # 현재 설정 확인
  gcloud config list

  # 프로젝트 변경
  gcloud config set project MY_PROJECT_ID
  ```

- **핵심 키워드**: gcloud CLI, `gcloud init`, 인증(Authentication), 프로젝트 ID(Project ID), 리전(Region)
- **시각화 연결**: 해당 없음
- **다음 섹션**: III-1-3

### III-1-3. 리전(Region)과 존(Zone) 선택하기

- **난이도**: ★★☆
- **선수 지식**: III-1-2
- **학습 목표**: 이 Section을 마치면 리전·존의 차이를 이해하고 서비스 특성에 맞는 리전을 선택할 수 있다.
- **설명**: 전국에 물류 창고가 있는 택배 회사를 떠올려보자. 서울 고객에게 제주 창고에서 물건을 보내면 시간이 오래 걸린다. 그래서 서울에도 창고를 두고, 서울 고객의 주문은 서울 창고에서 처리한다. GCP의 **리전(Region)**이 이 역할을 한다. 리전은 GCP 데이터센터들이 모여 있는 지리적 위치다.

  리전은 `asia-northeast3`(서울), `us-central1`(미국 중부)처럼 구체적인 지명으로 구분된다. 한국 사용자에게 서비스한다면 서울 리전을 선택해야 **지연 시간(Latency)**이 짧아진다. 같은 기능이라도 리전이 멀면 응답이 느려지기 때문이다. **존(Zone)**은 리전 안의 더 작은 단위다. 서울 리전에는 `asia-northeast3-a`, `asia-northeast3-b`, `asia-northeast3-c` 세 존이 있다. 각 존은 독립된 전력·냉각·네트워크를 갖춘 데이터센터 집합으로, 한 존에 장애가 생겨도 다른 존은 영향을 받지 않는다.

  실무에서 리전과 존 선택은 단순한 옵션이 아니다. 데이터 규정 준수가 중요한 서비스는 사용자 데이터가 특정 국가 안에 머물도록 리전을 강제해야 하는 경우가 있다. 고가용성(High Availability)이 필요한 서비스는 리소스를 여러 존에 분산해 단일 존 장애로 인한 서비스 중단을 막는다. `gcloud init`에서 설정한 기본 리전은 Cloud Run, Cloud SQL 배포 시 기본값으로 사용된다. 필요에 따라 서비스마다 다른 리전을 명시할 수도 있다.

  리전은 성능과 가용성을 동시에 결정하는 핵심 설정이다. Part IV에서 Cloud Run과 Cloud SQL을 배포할 때 `asia-northeast3`가 반복해서 등장하는 이유가 여기에 있다. 리전·존 개념을 이해한 뒤, 다음 단계에서는 GCP 웹 콘솔 화면을 직접 살펴본다.
- **핵심 키워드**: 리전(Region), 존(Zone), 지연 시간(Latency), 고가용성(High Availability), asia-northeast3(서울)
- **시각화 연결**: 해당 없음
- **다음 섹션**: III-2-1

### III-2-1. Cloud Console 주요 화면 둘러보기

- **난이도**: ★★☆
- **선수 지식**: III-1-1
- **학습 목표**: 이 Section을 마치면 Cloud Console에서 리소스 목록 조회·생성·삭제를 수행할 수 있다.
- **설명**: 공항 관제탑에는 모든 항공기의 위치·상태·연료를 한 화면에서 보고 제어하는 시스템이 있다. GCP의 **Cloud Console**(`console.cloud.google.com`)이 이 역할을 한다. GCP 계정에 속한 모든 리소스 — 실행 중인 컨테이너, 데이터베이스, 스토리지 버킷, 네트워크 설정 — 를 웹 브라우저에서 시각적으로 확인하고 조작하는 포털이다.

  콘솔에서 할 수 있는 대부분의 작업은 gcloud CLI로도 동일하게 수행할 수 있다. 두 방법은 서로 배타적이지 않다. 처음에는 콘솔의 시각적 인터페이스로 서비스 구조를 파악하고, 이후에는 같은 작업을 CLI 명령어로 옮겨 자동화하는 식이 효율적이다. 콘솔에서 리소스를 만든 뒤 CLI로 동일한 작업을 재현해 보면 두 방법의 대응 관계가 빠르게 익혀진다.

### III-2-2. Billing 알람 설정하기

- **난이도**: ★★☆
- **선수 지식**: III-1-1, I-2-1
- **학습 목표**: 이 Section을 마치면 GCP 프로젝트에 예산 알림을 설정해 예상치 못한 비용 초과를 사전에 방지할 수 있다.
- **설명**: 수도 계량기 옆에 월 사용량 한도를 설정해두면, 기준선을 넘는 순간 알림이 온다. 미리 설정한 덕분에 청구서가 날아오기 전에 사용 습관을 바꿀 수 있다. GCP의 **Budget Alert(예산 알림)**도 같은 역할을 한다. Part I(I-2-1)에서 "클라우드는 쓴 만큼 과금되며, Budget Alert으로 예상치 못한 청구를 막을 수 있다"고 배웠다. 이 Section에서는 그 개념을 실제 명령으로 구현한다.

  학습 과정에서 비용 사고가 발생하는 가장 흔한 패턴이 있다. 실습용으로 Cloud Run, Cloud SQL, Compute Engine 인스턴스를 켜놓은 채 다음 주제로 넘어가다가 잊어버리는 것이다. GCP의 종량제 특성상 인스턴스가 실행 중인 동안에는 사용량이 계속 누적된다. 특히 Cloud SQL과 Compute Engine은 Scale to Zero가 없기 때문에, 명시적으로 중지하거나 삭제하지 않으면 비용이 멈추지 않는다. Budget Alert은 이런 상황에서 "지금 뭔가 계속 켜져 있다"는 신호를 이메일로 보내주는 안전망이다.

  `gcloud billing budgets create` 명령은 청구 계정(Billing Account) 단위로 예산을 생성한다. `--budget-amount`로 월 예산 금액을 지정하고, `--threshold-rule`로 세 가지 임계값(50%, 90%, 100%)을 설정한다. 임계값에 도달하면 청구 계정에 연결된 이메일로 알림이 발송된다. 50% 시점 알림은 "리소스를 점검할 시간"이라는 신호이고, 90%는 "지금 당장 불필요한 리소스를 정리하라"는 경고이며, 100%는 "예산을 넘었다"는 최종 알림이다. 세 단계를 모두 설정해두면 단계적으로 상황을 파악하고 대응할 수 있다.

  Budget Alert은 GCP가 자동으로 리소스를 중지하는 기능이 아니다. 알림을 받은 뒤 직접 정리하는 것은 여전히 사용자의 몫이다. 그러나 실수를 인지하는 타이밍을 청구서 날짜에서 임계값 도달 시점으로 앞당긴다는 것만으로도 충분히 가치 있다. 학습을 시작하기 전에 이 명령 하나를 먼저 실행해두는 습관이 불필요한 비용 낭비를 막는 첫 번째 방어선이 된다.

  ```bash
  # 예산 알림 생성 (월 $50)
  gcloud billing budgets create \
    --billing-account=BILLING_ACCOUNT_ID \
    --display-name="Monthly Budget" \
    --budget-amount=50USD \
    --threshold-rule=percent=0.5 \
    --threshold-rule=percent=0.9 \
    --threshold-rule=percent=1.0
  ```

- **핵심 키워드**: Budget Alert, `gcloud billing budgets create`, 임계값(Threshold), 청구 계정, 비용 초과 방지
- **시각화 연결**: 해당 없음
- **다음 섹션**: IV-1-1

### IV-1-1. Cloud Run 요청이 처리되는 흐름 (시각화)

- **난이도**: ★★☆
- **선수 지식**: III-1-2, III-1-3
- **학습 목표**: 이 Section을 마치면 클라이언트 HTTPS 요청이 Cloud Run 인프라에서 처리되는 6단계를 순서대로 설명할 수 있다.
- **설명**: 택배 회사 물류 센터를 떠올려보자. 주문이 들어오면 창고 직원이 출근해 상자를 꾸리고, 주문이 없는 밤에는 직원을 대기시키지 않는다. Cloud Run이 동작하는 방식이 정확히 이와 같다. 요청이 오면 인스턴스가 켜지고, 요청이 없으면 인스턴스를 0개까지 줄인다. 이를 **Scale to Zero**라 부른다.

  Cloud Run은 GCP가 제공하는 서버리스 컨테이너 실행 서비스다. 개발자는 코드를 Docker 이미지로 만들어 GCP에 올리기만 하면, 나머지 서버 관리는 GCP가 맡는다. 컨테이너는 **Stateless(상태 없음)** 구조로 동작한다. 즉, 각 요청은 독립적으로 처리되며 이전 요청의 데이터를 메모리에 남겨두지 않는다. 영구 데이터는 Cloud SQL이나 Cloud Storage 같은 별도 저장소를 사용한다. gcloud CLI 설치와 GCP 인증 방법은 Part III에서 다룬다.

  클라이언트가 HTTPS 요청을 보내면 처리 흐름은 6단계로 이어진다. ① 클라이언트가 HTTPS 요청을 전송한다. ② **Google Load Balancer**가 요청을 받아 SSL 인증서를 처리하고(SSL Termination) 적절한 Cloud Run 인스턴스로 라우팅한다. ③ Cloud Run 인스턴스가 코드를 실행해 요청을 처리한다. ④ 데이터 조회나 저장이 필요하면 **Cloud SQL**에 접근한다. ⑤ 처리 결과를 클라이언트에 응답으로 반환한다. ⑥ 모든 요청과 에러 로그는 **Cloud Logging**에 자동으로 기록된다. 이 6단계 흐름을 `index.html` 시각화로 단계별로 확인할 수 있다.

  Cloud Run의 핵심은 Scale to Zero다. 트래픽이 없을 때 비용이 발생하지 않고, 트래픽이 몰릴 때는 자동으로 인스턴스가 늘어난다. Load Balancer가 SSL을 처리하므로 개발자는 인증서 관리 없이 HTTPS 엔드포인트를 바로 얻는다. 다음 Section에서는 이 흐름의 출발점인 컨테이너 이미지를 직접 만드는 방법을 살펴본다.

  ```bash
  # Cloud Run 배포
  gcloud run deploy my-api \
    --image gcr.io/myproject/my-api \
    --region asia-northeast3 \
    --allow-unauthenticated

  # Cloud SQL 연결 설정
  gcloud run services update my-api \
    --add-cloudsql-instances myproject:asia-northeast3:mydb

  # 로그 확인
  gcloud logging read "resource.type=cloud_run_revision" --limit 10
  ```

- **핵심 키워드**: Cloud Run, Scale to Zero, Stateless 컨테이너, Load Balancer, SSL Termination, Cloud Logging
- **시각화 연결**: `index.html` (6단계 Cloud Run 요청 흐름 인터랙티브 시각화)
- **다음 섹션**: IV-1-2

### IV-1-2. Dockerfile과 컨테이너 이미지 만들기

- **난이도**: ★★☆
- **선수 지식**: IV-1-1
- **학습 목표**: 이 Section을 마치면 Python 앱을 Docker 이미지로 만들고 Artifact Registry에 푸시할 수 있다.
- **설명**: 택배 상자를 생각해보자. 상자 안에 물건과 함께 "이 물건은 냉장 보관, 습기 주의" 같은 취급 안내서를 동봉하면, 어떤 택배 기사가 가져가든 동일하게 처리된다. 컨테이너 이미지(Container Image)가 이 역할을 한다. 앱 코드뿐 아니라 Python 버전, 설치 라이브러리, 시작 명령어까지 하나의 패키지로 묶어두면, 어떤 서버에서도 동일한 환경으로 실행된다.

  **Dockerfile**은 이 상자를 조립하는 설명서다. "어떤 베이스 이미지를 쓸 것인지", "어떤 패키지를 설치할 것인지", "앱을 어떻게 시작할 것인지"를 순서대로 적는다. 이 파일을 바탕으로 `docker build` 명령을 실행하면 컨테이너 이미지가 만들어진다. GCP 환경에서는 `gcloud builds submit` 명령으로 로컬 빌드 없이 GCP 서버 위에서 직접 이미지를 빌드할 수도 있다. 빌드된 이미지는 **Artifact Registry**(구 Container Registry)에 업로드되고, Cloud Run이 이곳에서 이미지를 내려받아 실행한다.

  위 Dockerfile 예시를 보면 각 줄의 역할이 명확하다. `FROM python:3.11-slim`은 Python 3.11이 설치된 경량 리눅스 환경을 베이스로 삼는다. `COPY requirements.txt .`와 `RUN pip install` 순서가 중요한데, 의존성 파일을 먼저 복사하고 설치한 뒤 나머지 소스코드를 복사하면 코드만 변경됐을 때 pip 설치 단계를 다시 실행하지 않아 빌드 속도가 빨라진다. 마지막 `CMD`는 컨테이너가 시작될 때 실행할 명령으로, Cloud Run은 기본적으로 8080 포트에서 요청을 받는다.

  컨테이너 이미지는 Cloud Run 배포의 출발점이다. Dockerfile에 실행 환경을 선언하고, `gcloud builds submit`으로 이미지를 Artifact Registry에 올린 뒤, Cloud Run에 배포하면 이전 Section에서 살펴본 6단계 흐름이 시작된다. "로컬에서는 되는데 서버에서는 안 된다"는 상황을 컨테이너가 근본적으로 해결해준다.

  ```dockerfile
  # Dockerfile 예시 (Python FastAPI)
  FROM python:3.11-slim
  WORKDIR /app
  COPY requirements.txt .
  RUN pip install -r requirements.txt
  COPY . .
  CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]
  ```

  ```bash
  # 이미지 빌드 및 Artifact Registry 푸시
  gcloud builds submit --tag gcr.io/myproject/my-api
  ```

- **핵심 키워드**: Dockerfile, 컨테이너 이미지(Container Image), Artifact Registry, `docker build`, `gcloud builds submit`
- **시각화 연결**: 향후 구현 예정
- **다음 섹션**: IV-1-3

### IV-1-3. Cloud Run 배포 옵션 이해하기

- **난이도**: ★★☆
- **선수 지식**: IV-1-2
- **학습 목표**: 이 Section을 마치면 주요 gcloud run deploy 옵션(리전, 인증, 동시성, 메모리)을 설명하고 실습할 수 있다.
- **설명**: `gcloud run deploy` 명령은 이미지를 클라우드에 올리는 단순한 업로드 명령이 아니다. 서비스가 어떤 조건에서 실행될지를 명시하는 계약서에 가깝다. 식당에 자리를 예약할 때 "4명, 창가 자리, 조용한 구역"처럼 조건을 세세하게 지정하듯, 배포 명령에서도 인증 방식·동시 처리 수·메모리·인스턴스 수를 직접 정해야 한다. 옵션을 모르면 기본값에 맡기게 되고, 기본값이 내 서비스에 맞지 않을 때 장애나 비용 낭비로 이어진다.

  가장 먼저 이해해야 할 옵션은 인증 설정이다. `--allow-unauthenticated`는 누구든 URL만 알면 호출할 수 있도록 서비스를 공개하는 옵션이다. 외부 사용자가 접근하는 공개 API나 웹 프론트엔드에 적합하다. 반대로 `--no-allow-unauthenticated`를 쓰면 GCP 내부 서비스나 인증된 계정만 호출할 수 있어, 백오피스나 내부 마이크로서비스에 사용한다. 다음은 **동시성(Concurrency)** 설정이다. `--concurrency 80`은 인스턴스 하나가 최대 80개의 요청을 동시에 처리하도록 제한하는 값이다. 요청이 CPU를 많이 쓰는 작업이라면 이 숫자를 줄여야 인스턴스 하나가 과부하 없이 안정적으로 동작한다. `--memory 512Mi`는 인스턴스에 할당하는 메모리 크기로, 기본값은 256MiB이지만 이미지 처리나 데이터 가공처럼 메모리를 많이 쓰는 작업은 직접 높여줘야 OOM(메모리 부족) 오류를 피할 수 있다.

  실무에서 가장 자주 놓치는 옵션은 `--min-instances`다. Cloud Run은 요청이 없으면 인스턴스를 0으로 줄이는 **Scale to Zero** 방식으로 비용을 아끼는데, 이때 새 요청이 들어오면 컨테이너가 처음부터 시작되면서 몇 초의 지연이 발생한다. 이것을 **콜드 스타트(Cold Start)**라고 부른다. 대부분의 API에서는 허용 가능한 수준이지만, 사용자가 매일 아침 로그인하는 대시보드 서비스처럼 첫 응답 속도가 UX와 직결되는 경우에는 치명적이다. `--min-instances 1`로 설정하면 인스턴스를 항상 1개 이상 유지해 콜드 스타트 없이 즉시 응답할 수 있다. 물론 인스턴스를 유지하는 만큼 소정의 비용이 발생하므로, 트래픽 패턴과 비용을 함께 고려해 값을 결정해야 한다.

  배포 옵션 하나하나가 서비스의 보안, 성능, 비용을 결정하는 레버(lever)다. 처음에는 기본값으로 시작해도 좋지만, 서비스가 실제 사용자에게 노출되기 전에 반드시 각 옵션의 의미를 확인하는 습관을 들이자. 아래 명령은 이 Section에서 다룬 옵션을 모두 적용한 배포 예시다.

  ```bash
  gcloud run deploy my-api \
    --image gcr.io/myproject/my-api \
    --region asia-northeast3 \
    --no-allow-unauthenticated \
    --concurrency 80 \
    --memory 512Mi \
    --min-instances 1
  ```

- **핵심 키워드**: `--allow-unauthenticated`, 동시성(Concurrency), 콜드 스타트(Cold Start), `--min-instances`, 메모리 설정
- **시각화 연결**: 해당 없음
- **다음 섹션**: IV-2-1

### IV-2-1. Cloud Storage — 파일을 클라우드에 보관하기

- **난이도**: ★★☆
- **선수 지식**: III-1-2
- **학습 목표**: 이 Section을 마치면 Cloud Storage 버킷을 만들고, 파일을 업로드·다운로드·삭제할 수 있다.
- **설명**: 창고를 상상해 보자. 창고 안에는 여러 개의 캐비닛이 있고, 각 캐비닛 안에는 물건들이 들어 있다. **Cloud Storage**는 바로 이 창고 역할을 하는 GCP의 파일 저장 서비스다. 창고 칸 하나하나가 **버킷(Bucket)**, 그 안에 담긴 파일 하나하나가 **오브젝트(Object)**에 해당한다. 이미지, 동영상, PDF, 백업 파일, 정적 웹사이트 파일 — 형식을 가리지 않고 모든 종류의 파일을 저장할 수 있다. AWS를 먼저 접한 개발자라면 익숙한 개념이다. Cloud Storage는 AWS의 S3와 동일한 역할을 하는 서비스로, II-1-3의 대응표에서도 나란히 놓였던 바로 그 서비스다.

  버킷을 만들 때 이름은 전 세계에서 유일해야 한다. 인터넷 도메인처럼 다른 누군가가 이미 선점한 이름은 쓸 수 없다는 뜻이다. 리전을 지정하면 해당 지역의 데이터센터에 파일이 저장되므로, 서비스 사용자가 주로 있는 지역과 가까운 리전을 선택하면 파일 접근 속도가 빨라진다. 오브젝트마다 고유한 URL이 부여되는데, 이 URL을 어떻게 공개하느냐에 따라 두 가지 접근 방식으로 나뉜다. **퍼블릭 접근(Public Access)**을 허용하면 누구나 URL만으로 파일을 다운로드할 수 있어 공개 이미지나 정적 에셋 배포에 적합하다. 반면 민감한 파일은 비공개로 두고, **Signed URL**을 발급해 일시적으로만 접근을 허용하는 방식을 쓴다. 예를 들어 사용자가 구매 영수증 PDF를 다운로드하는 링크는, 24시간 후에 만료되는 Signed URL로 만들면 파일 자체는 외부에 노출하지 않고도 안전하게 전달할 수 있다.

  아래 명령은 `gcloud` CLI로 버킷을 만들고 파일을 업로드하는 기본 흐름을 보여준다. `gcloud` 명령을 실행하려면 CLI 설치와 인증이 선행되어야 하며, 이 과정은 Part III에서 자세히 다룬다. 버킷 이름(`my-bucket-name`)과 리전은 본인 환경에 맞게 교체하면 된다. `gcloud storage ls` 명령으로 버킷 안의 파일 목록을 확인할 수 있고, 파일이 제대로 올라갔는지 빠르게 검증하는 용도로 자주 쓴다.

  Cloud Storage는 단순한 파일 보관함을 넘어, 서비스 이미지 에셋을 배포하고 사용자가 업로드한 파일을 처리하고 서버 로그를 장기 보관하는 등 거의 모든 GCP 프로젝트에서 등장한다. Cloud Run 애플리케이션이 사용자 프로필 사진을 저장하거나, 배치 작업 결과물을 버킷에 쌓는 패턴도 흔하다. 버킷 생성과 파일 업로드를 손에 익혀두면, 이후 Cloud SQL과 연동하는 단계에서도 자연스럽게 활용하게 된다.

  ```bash
  # 버킷 생성
  gcloud storage buckets create gs://my-bucket-name \
    --location=asia-northeast3

  # 파일 업로드
  gcloud storage cp ./image.png gs://my-bucket-name/

  # 파일 목록 확인
  gcloud storage ls gs://my-bucket-name/
  ```

- **핵심 키워드**: Cloud Storage, 버킷(Bucket), 오브젝트(Object), Signed URL, 퍼블릭 접근(Public Access)
- **시각화 연결**: 해당 없음
- **다음 섹션**: IV-3-1

### IV-3-1. Cloud SQL — 관리형 관계형 데이터베이스

- **난이도**: ★★☆
- **선수 지식**: III-1-3, IV-1-1
- **학습 목표**: 이 Section을 마치면 Cloud SQL 인스턴스를 생성하고 Cloud Run에 연결할 수 있다.
- **설명**: 호텔에 묵을 때 침구를 직접 빨거나 고장 난 샤워기를 손수 고칠 필요가 없는 것처럼, **Cloud SQL**은 데이터베이스(DB) 서버의 설치·패치·백업·장애 복구를 GCP가 대신 맡아 처리하는 관리형 DB 서비스다. PostgreSQL, MySQL, SQL Server 세 가지 엔진을 지원하며, 직접 서버를 프로비저닝하지 않아도 몇 줄의 명령으로 인스턴스(Instance)가 만들어진다. `gcloud` 명령 사용을 위한 계정 설정과 CLI 인증은 Part III에서 다룬다.

  Cloud SQL 인스턴스를 생성하고 나면, Cloud Run 서비스가 그 DB에 어떻게 연결하는지가 관건이다. 공개 IP로 연결하면 외부 인터넷을 거치므로 보안 설정이 복잡해지고 지연도 생긴다. 반면 **Private IP** 연결은 GCP 내부 네트워크만 통과하므로 외부 인터넷이 끼어들 틈이 없다. 보안과 성능 두 마리 토끼를 동시에 잡는 구조다. Cloud Run에서 Private IP로 연결할 때 **Cloud SQL Auth Proxy**를 사용하거나 `--add-cloudsql-instances` 옵션을 지정하면, GCP가 연결 인증과 암호화를 자동으로 처리해 준다. `index.html` 시각화의 STEP 4(Cloud SQL Private IP 연결)에서 이 흐름을 단계별로 확인할 수 있다.

  실무에서 Cloud SQL의 가치가 드러나는 장면은 패치 주기다. 직접 운영하는 DB 서버라면 OS 보안 패치, DB 버전 업그레이드, 백업 스케줄 설정, 장애 시 복구 절차를 개발팀이 직접 챙겨야 한다. Cloud SQL은 이 모든 작업을 GCP가 스케줄대로 수행하며, 필요하면 자동 장애 조치(Failover)까지 해준다. 주니어 개발자가 DB 운영의 복잡함에 신경 쓰는 대신 애플리케이션 로직에 집중할 수 있는 이유가 바로 관리형 서비스의 본질이다.

  Cloud SQL은 "DB 서버도 GCP에 맡기면 된다"는 한 문장으로 요약된다. 인스턴스 생성과 Cloud Run 연결 명령을 익혀두면 아래 코드 블록이 바로 실전 도구가 된다.

  ```bash
  # Cloud SQL 인스턴스 생성 (PostgreSQL)
  gcloud sql instances create mydb \
    --database-version=POSTGRES_15 \
    --region=asia-northeast3 \
    --tier=db-f1-micro

  # Cloud Run에 Cloud SQL 연결
  gcloud run services update my-api \
    --add-cloudsql-instances myproject:asia-northeast3:mydb
  ```

- **핵심 키워드**: Cloud SQL, Cloud SQL Auth Proxy, Private IP, 관리형 DB(Managed DB), 인스턴스(Instance)
- **시각화 연결**: `index.html` (STEP 4 — Cloud SQL Private IP 연결 단계)
- **다음 섹션**: IV-3-2

### IV-3-2. Cloud SQL 연결 — 환경 변수와 Secret Manager

- **난이도**: ★★★
- **선수 지식**: IV-3-1
- **학습 목표**: 이 Section을 마치면 DB 비밀번호를 Secret Manager에 저장하고 Cloud Run 환경 변수로 주입할 수 있다.
- **설명**: DB 비밀번호를 코드에 직접 적어 두는 것이 왜 위험한지, 한 가지 실제로 일어나는 시나리오를 생각해 보자. 개발 중에 `DATABASE_URL = "postgresql://user:mypassword@..."` 한 줄을 소스 코드에 넣고 GitHub에 push한다. 퍼블릭 저장소라면 크롤러가 수 분 안에 해당 비밀번호를 수집한다. 프라이빗 저장소라도 팀원이 늘거나 저장소 권한이 잘못 설정되면 언제든 노출될 수 있다. 클라우드 계정 정보가 유출되어 대규모 리소스가 무단으로 생성되고 수백만 원의 청구가 발생하는 사고는 현업에서 드물지 않게 보고된다. 코드는 언제든 외부에 공개될 수 있다고 가정하고 설계하는 습관이 필요하다.

  이 문제를 해결하는 것이 **Secret Manager**다. Secret Manager는 비밀 값을 금고에 잠그고, 꺼낼 때마다 누가 꺼냈는지 기록하는 GCP의 보안 저장소다. 비밀번호나 API 키를 Secret Manager에 한 번 저장해 두면, 코드 어디에도 그 값이 남지 않는다. Cloud Run에서는 `--set-secrets` 옵션 한 줄로 Secret Manager의 값을 **환경 변수(Environment Variable)**로 주입할 수 있다. 애플리케이션은 `os.environ["DB_PASSWORD"]`처럼 평범한 환경 변수를 읽기만 하면 되고, 실제 비밀번호가 어디에 있는지는 신경 쓸 필요가 없다.

  여기서 **최소 권한 원칙**이 함께 작동한다. Secret Manager의 특정 Secret에 접근할 수 있는 서비스 계정만 그 값을 읽을 수 있도록 권한을 좁힌다. 즉 Cloud Run 서비스는 자신에게 허용된 Secret만 열 수 있는 열쇠를 가지고, 그 열쇠로는 다른 금고를 열 수 없다. 이 두 가지, 코드 밖에 비밀을 두는 것과 필요한 곳에만 접근을 허용하는 것이 실무 보안의 기본 패턴이다.

  Secret Manager와 `--set-secrets`의 조합을 한 번 손에 익혀두면, 어떤 비밀 값도 코드에 박히지 않는 안전한 구조를 만들 수 있다. 다음 파트에서 다루는 IAM 권한 설정과 함께 이해하면 이 패턴이 더 단단해진다.

  ```bash
  # Secret 생성
  echo -n "my-db-password" | \
    gcloud secrets create db-password --data-file=-

  # Cloud Run에 Secret을 환경 변수로 주입
  gcloud run services update my-api \
    --set-secrets=DB_PASSWORD=db-password:latest
  ```

- **핵심 키워드**: Secret Manager, 환경 변수(Environment Variable), `--set-secrets`, 보안(Security), 최소 권한 원칙
- **시각화 연결**: 해당 없음
- **다음 섹션**: IV-3-3

### IV-3-3. Cloud SQL Studio 사용하기

- **난이도**: ★★☆
- **선수 지식**: IV-3-1
- **학습 목표**: 이 Section을 마치면 Cloud SQL Studio에서 데이터베이스에 접속해 테이블을 조회하고 SQL을 실행할 수 있다.
- **설명**: 은행 앱에서 잔액을 확인하듯, Cloud SQL Studio는 별도 설비 없이 브라우저 하나로 DB 안을 바로 들여다볼 수 있게 해준다. 일반적으로 DB에 접근하려면 DBeaver나 pgAdmin 같은 클라이언트를 설치하고, 호스트·포트·비밀번호를 입력해 연결을 구성해야 한다. 특히 Cloud SQL처럼 Private IP로 구성된 경우, 로컬 머신에서 접근하려면 Cloud SQL Auth Proxy까지 별도로 실행해야 한다. **Cloud SQL Studio**는 이 과정 전체를 생략할 수 있게 해준다. Cloud Console 안에 내장된 웹 기반 SQL 에디터로, 별도 소프트웨어 없이 브라우저에서 직접 DB에 접속한다.

  IV-3-1에서 생성한 Cloud SQL 인스턴스는 이미 GCP 프로젝트에 인증된 상태로 존재한다. Cloud SQL Studio는 이 인증을 그대로 재활용한다. Console에서 Cloud SQL 인스턴스를 선택하고 "Studio" 탭으로 이동하면, 연결할 데이터베이스와 사용자를 선택하는 화면이 나타난다. 인증이 통과되면 곧바로 SQL 에디터가 열린다. 이 에디터에서 `SELECT`, `INSERT`, `CREATE TABLE` 같은 SQL 문을 직접 입력하고 실행할 수 있다. 결과는 에디터 아래 테이블 형태로 바로 출력된다.

  개발 과정에서 Cloud SQL Studio가 특히 유용한 장면은 두 가지다. 첫째, Cloud Run 서비스를 배포한 뒤 데이터가 DB에 올바르게 저장되고 있는지 빠르게 확인할 때다. 로컬에서 Auth Proxy를 켜고 클라이언트를 연결하는 대신, Console을 열고 Studio에서 SELECT 한 줄이면 충분하다. 둘째, 초기 스키마를 잡을 때다. 테이블 구조를 바꾸거나 시드 데이터를 넣는 SQL을 바로 실행해볼 수 있어, 마이그레이션 스크립트를 짜기 전 빠른 프로토타이핑이 가능하다.

  Cloud SQL Studio는 운영 환경보다는 개발·디버깅 단계에서 쓰는 도구다. 프로덕션 데이터를 직접 수정하는 용도로는 적합하지 않다. 하지만 학습 과정이나 개발 초기에 DB 상태를 빠르게 들여다볼 수 있다는 점에서, 별도 클라이언트 없이 Console 하나로 Cloud SQL을 다룰 수 있는 편의성은 분명하다.
- **핵심 키워드**: Cloud SQL Studio, 웹 기반 SQL 클라이언트, 테이블 조회, SQL 실행, 스키마 확인
- **시각화 연결**: 해당 없음
- **다음 섹션**: V-1-1

## Part V. 네트워킹 · IAM

> VPC, 방화벽, IAM 권한 — GCP 보안과 네트워크 구조를 이해한다.

### V-1-1. VPC — 나만의 가상 네트워크

- **난이도**: ★★★
- **선수 지식**: IV-1-1
- **학습 목표**: 이 Section을 마치면 VPC·서브넷·방화벽 규칙의 관계를 설명하고 기본 VPC 구조를 파악할 수 있다.
- **설명**: 회사 사무실 건물을 상상해보자. 건물 전체가 회사 소유이고, 각 층은 팀별로 구분되어 있다. 1층은 영업팀, 2층은 개발팀, 3층은 임원실이다. 각 층 출입구에는 보안 도어가 달려 있어 권한 없는 직원은 들어올 수 없다. GCP에서 **VPC(Virtual Private Cloud)**는 바로 이 건물 역할을 한다. GCP 안에 만드는 나만의 가상 네트워크로, 내 리소스들이 외부 인터넷과 어떻게 통신할지를 내가 직접 제어하는 공간이다.

  VPC 안에는 **서브넷(Subnet)**이 존재한다. 건물의 각 층처럼, 서브넷은 VPC라는 네트워크 공간을 용도별로 나누는 단위다. 서울 리전의 서브넷과 미국 리전의 서브넷을 같은 VPC 안에 둘 수 있고, 리소스는 각자 속한 서브넷의 IP 대역을 부여받는다. 층마다 달린 보안 도어가 **방화벽 규칙(Firewall Rules)**이다. "80, 443 포트만 외부에서 허용", "내부 서브넷끼리는 모든 포트 허용" 같은 규칙을 설정해 트래픽 흐름을 정밀하게 통제한다. GCP는 프로젝트를 생성하면 기본 VPC를 자동으로 만들어 주므로 처음에는 별도 설정 없이 바로 사용할 수 있다.

  Part IV에서 Cloud Run과 Cloud SQL을 다룰 때 **Private IP**라는 표현이 등장했다. 이 개념이 VPC와 연결된다. Cloud Run은 기본적으로 VPC 외부에서 실행되는 서버리스 서비스지만, **VPC Connector**를 설정하면 Cloud Run이 VPC 내부 리소스, 즉 Private IP를 가진 Cloud SQL에 직접 접근할 수 있다. 공인 인터넷을 거치지 않고 GCP 내부 네트워크만으로 통신하게 되어 보안과 속도 두 가지를 모두 잡는다.

  VPC는 GCP 네트워크 구조의 뼈대다. 서브넷으로 공간을 나누고, 방화벽 규칙으로 문을 제어하며, VPC Connector로 서버리스와 내부 네트워크를 잇는 구조를 파악하면 GCP 보안 설계의 절반을 이해한 것이다. 다음 섹션에서는 이 네트워크 위에서 누가 무엇을 할 수 있는지 권한을 제어하는 IAM을 살펴본다.
- **핵심 키워드**: VPC(Virtual Private Cloud), 서브넷(Subnet), 방화벽 규칙(Firewall Rules), VPC Connector, Private IP
- **시각화 연결**: 해당 없음
- **다음 섹션**: V-1-2

### V-1-2. IAM — 누가 무엇을 할 수 있는가

- **난이도**: ★★★
- **선수 지식**: III-1-1
- **학습 목표**: 이 Section을 마치면 IAM의 역할(Role)·정책(Policy)·서비스 계정(Service Account)을 이해하고 최소 권한 원칙을 적용할 수 있다.
- **설명**: 대형 병원을 떠올려보자. 의사는 환자 기록을 읽고 처방을 내릴 수 있다. 간호사는 기록을 읽고 처치를 수행하지만 처방은 내릴 수 없다. 청소부는 병실에 들어가지만 환자 기록에는 접근하지 못한다. 각자의 역할에 딱 맞는 권한만 가진다. GCP의 **IAM(Identity and Access Management)**이 이 출입 통제 시스템이다. 누가(Who) 어떤 리소스에(What) 어떤 행동을(Which Action) 할 수 있는지 정의한다.

  IAM의 핵심 구성 요소는 세 가지다. 첫째, **역할(Role)**은 권한의 묶음이다. `roles/run.admin`은 Cloud Run의 모든 작업을 허용하고, `roles/run.viewer`는 상태 조회만 허용한다. GCP는 수백 개의 사전 정의 역할을 제공하며, 필요하면 커스텀 역할을 직접 만들 수도 있다. 둘째, **서비스 계정(Service Account)**은 사람이 아닌 앱이나 서비스가 GCP 리소스에 접근할 때 사용하는 계정이다. Cloud Run이 Cloud SQL에 쿼리를 날리거나, Cloud Storage에 파일을 쓸 때 사람의 개입 없이 자동으로 인증이 필요한데, 이때 서비스 계정이 그 주체가 된다.

  Part IV에서 Secret Manager를 다루며 등장한 **최소 권한 원칙**이 여기서 구체화된다. Cloud Run이 Cloud SQL에 접근해야 한다면, Cloud Run의 서비스 계정에 `roles/cloudsql.client` 역할만 부여한다. 그 이상의 권한은 주지 않는다. 서비스 계정 하나가 너무 넓은 권한을 가지면, 그 계정이 탈취됐을 때 피해 범위가 그만큼 넓어진다. 역할을 최소한으로 좁히는 것이 보안 사고의 파급을 줄이는 가장 확실한 방법이다. 새 팀원이 합류했을 때 GCP 콘솔에 어떤 역할을 줄지 고민이 된다면, "이 사람이 지금 당장 필요한 것만 주고 나중에 추가하는" 방향이 올바른 기본값이다.

  IAM은 GCP 전체를 관통하는 권한 레이어다. 네트워크(VPC)로 공간을 분리하고, IAM으로 행동을 분리하면 외부 침입과 내부 실수 두 가지 위협을 동시에 방어할 수 있다. 다음 섹션에서는 외부 트래픽이 GCP 네트워크 안으로 들어오는 관문인 로드 밸런서를 더 깊이 살펴본다.
- **핵심 키워드**: IAM, 역할(Role), 서비스 계정(Service Account), 최소 권한 원칙(Least Privilege), `gcloud iam`
- **시각화 연결**: 해당 없음
- **다음 섹션**: V-2-1

### V-2-1. 로드 밸런서(Load Balancer) 심화

- **난이도**: ★★★
- **선수 지식**: IV-1-1, V-1-1
- **학습 목표**: 이 Section을 마치면 GCP 글로벌 로드 밸런서 유형과 Cloud Run과의 관계를 설명할 수 있다.
- **설명**: 국제공항의 입국 심사대를 생각해보자. 비행기 한 대에서 수백 명이 내려도 입국 심사관 여러 명이 동시에 처리하고, 심사대마다 처리할 여권 국적을 다르게 배분하기도 한다. 항공편이 몰리는 피크 타임에는 심사관을 더 늘린다. GCP의 **Cloud Load Balancing**이 이 역할을 한다. 쏟아지는 인터넷 트래픽을 여러 Cloud Run 인스턴스로 분산하고, 전 세계 어디서 요청이 와도 가장 가까운 GCP 인프라로 안내한다.

  Cloud Run을 공개 URL로 노출하면 GCP가 자동으로 **Cloud Load Balancing**을 붙여준다. 이 로드 밸런서는 **글로벌 애니캐스트(Anycast)** IP를 사용한다. 애니캐스트란 하나의 IP 주소가 전 세계 여러 위치에 동시에 존재하고, 사용자의 요청이 그 중 가장 가까운 GCP 거점(PoP, Point of Presence)으로 자동 라우팅되는 기술이다. 서울 사용자의 요청은 서울 거점에서, 뉴욕 사용자의 요청은 뉴욕 거점에서 처리된다. 그리고 **SSL Termination(SSL 종료)**을 로드 밸런서가 담당한다. HTTPS 암호화와 복호화를 여기서 끝내고, 로드 밸런서 내부에서 Cloud Run으로는 HTTP로 전달한다. 개발자가 인증서를 직접 발급하고 갱신할 필요 없이 **Google-managed SSL** 인증서를 사용하면 된다. `index.html` 시각화의 STEP 2에서 이 SSL 종료와 라우팅 흐름을 단계별로 확인할 수 있다.

  실무에서 자주 쓰이는 패턴 중 하나가 **URL Map**을 이용한 경로 기반 라우팅이다. 예를 들어 `/api` 경로로 들어오는 요청은 백엔드 Cloud Run 서비스로, `/admin` 경로는 관리자 전용 Cloud Run 서비스로, 나머지 경로는 프론트엔드 서비스로 보내는 구조다. 이렇게 하면 단 하나의 공개 IP와 도메인으로 여러 Cloud Run 서비스를 운영할 수 있다. Part IV에서 다룬 Cloud Run의 Scale to Zero도 이 구조 아래에서 서비스별로 독립적으로 작동한다.

  로드 밸런서는 트래픽의 첫 번째 관문이다. 글로벌 트래픽 분산, SSL 처리, 경로 기반 라우팅을 한 곳에서 해결해주므로 개발자는 애플리케이션 로직에만 집중할 수 있다. 다음 섹션에서는 이 관문 앞에 보안 경비를 더하는 Cloud Armor를 살펴본다.
- **핵심 키워드**: Cloud Load Balancing, 애니캐스트(Anycast), SSL Termination, Google-managed SSL, URL Map
- **시각화 연결**: `index.html` (STEP 2 — Load Balancer SSL 종료 · 라우팅 단계)
- **다음 섹션**: V-2-2

### V-2-2. Cloud Armor — DDoS 방어와 WAF

- **난이도**: ★★★
- **선수 지식**: V-2-1
- **학습 목표**: 이 Section을 마치면 Cloud Armor를 로드 밸런서에 붙여 IP 차단·WAF 규칙을 적용할 수 있다.
- **설명**: 유명 공연의 콘서트 입장 줄을 상상해보자. 정상적인 관람객들이 줄을 서는 사이, 일부 악의적인 사람들이 가짜 표를 들고 줄을 혼잡하게 만들거나, 수천 명이 동시에 몰려 입구를 막아버리려 한다. 보안 요원이 이런 시도를 미리 걸러내 진짜 관람객만 통과시킨다. **Cloud Armor**가 이 보안 요원 역할을 한다. 로드 밸런서 앞에서 악의적인 트래픽을 필터링해 정상적인 요청만 Cloud Run에 도달하게 한다.

  Cloud Armor는 두 가지 방어 기능을 제공한다. 첫째는 **DDoS(Distributed Denial of Service, 분산 서비스 거부)** 방어다. DDoS는 수만 대의 컴퓨터가 동시에 하나의 서버에 요청을 보내 서비스를 마비시키는 공격이다. Cloud Armor는 비정상적으로 많은 요청이 특정 IP 대역에서 쏟아지면 자동으로 감지하고 차단한다. 둘째는 **WAF(Web Application Firewall, 웹 애플리케이션 방화벽)** 기능이다. WAF는 HTTP 요청 내용을 들여다보고 SQL Injection(DB를 조작하려는 악의적 쿼리 삽입), XSS(Cross-Site Scripting, 악성 스크립트 삽입) 같은 웹 공격 패턴을 인식해 차단한다. GCP는 OWASP(오픈소스 웹 보안 프로젝트) 규칙 셋을 제공하므로, 보안 전문가 없이도 주요 웹 공격에 대한 기본 방어선을 세울 수 있다.

  특정 국가나 IP 대역에서 오는 트래픽을 허용하거나 차단하는 규칙도 **보안 정책(Security Policy)**으로 관리한다. 예를 들어 서비스 대상 국가가 한국뿐이라면 한국 IP만 허용하고 나머지를 차단하는 지리 기반 규칙을 설정할 수 있다. 실무에서는 외부에 노출된 API 앞에 Cloud Armor를 항상 붙이는 것이 권장된다. Part IV에서 Secret Manager로 코드 안의 비밀을 보호했고, V-1-2에서 IAM으로 내부 권한을 제어했다면, Cloud Armor는 외부에서 들어오는 위협에 대한 첫 번째 방어선이다.

  Cloud Armor는 로드 밸런서와 한 쌍으로 동작하는 보안 레이어다. DDoS 자동 감지, WAF 규칙, IP 기반 접근 제어를 조합하면 외부 공격으로부터 GCP 서비스를 안정적으로 보호할 수 있다. VPC로 네트워크를 분리하고, IAM으로 권한을 통제하고, 로드 밸런서로 트래픽을 분산하고, Cloud Armor로 공격을 걸러내는 이 네 겹의 구조가 GCP 네트워킹·보안의 기본 골격이다.
- **핵심 키워드**: Cloud Armor, DDoS, WAF(Web Application Firewall), IP 차단, 보안 정책(Security Policy)
- **시각화 연결**: 해당 없음
- **다음 섹션**: VI-1-1

## Part VI. 실전 배포 프로젝트

> Cloud Run + Cloud SQL + Secret Manager를 조합해 실제 서비스를 배포한다.

---

### VI-1-1. 프로젝트 설계 — 3계층 아키텍처

- **난이도**: ★★★
- **선수 지식**: IV-1-3, IV-3-2, V-1-2
- **학습 목표**: 이 Section을 마치면 Cloud Run · Cloud SQL · Secret Manager를 조합한 3계층 아키텍처를 설계도로 그릴 수 있다.
- **설명**: 건물을 지을 때 1층은 로비(안내·출입), 2층은 업무 공간(실제 일처리), 3층은 창고(자료 보관)로 층을 나눠 설계한다. 각 층이 뒤섞이면 관리가 어려워지고 문제가 생겼을 때 원인을 찾기도 힘들다. 실전 GCP 프로젝트도 이와 같은 방식으로 3계층 아키텍처를 적용해 설계한다.

  **3계층 아키텍처**는 시스템을 역할에 따라 세 층으로 나누는 설계 방식이다. **Presentation 계층**은 외부 요청을 받아 응답을 반환하는 층으로, Cloud Run이 API 서버 또는 웹 서버 역할을 맡는다. **Business Logic 계층**은 실제 비즈니스 규칙을 처리하는 층으로, Cloud Run 내부의 Service 레이어 코드가 담당한다. **Data 계층**은 데이터를 영구적으로 저장하고 조회하는 층으로, Cloud SQL이 이 역할을 한다. 여기에 Secret Manager(비밀 값 관리), Artifact Registry(컨테이너 이미지 저장), Cloud Logging(로그 수집)이 각 계층을 보조하는 역할을 한다.

  Part IV에서 익힌 Cloud Run · Cloud SQL · Secret Manager가 이 설계도 안에서 각자 자리를 찾는다. 각 계층은 IAM 서비스 계정(Service Account)으로 최소 권한을 부여받아 통신한다. 예를 들어, Cloud Run 서비스 계정에는 Cloud SQL 읽기·쓰기 권한만 부여하고 다른 리소스에는 접근하지 못하도록 좁혀둔다. Part V의 IAM 최소 권한 원칙이 여기서 직접 적용된다. 계층이 명확히 분리될수록 장애 범위가 줄어들고, 새 기능을 추가할 때 어느 층을 수정해야 하는지 금방 알 수 있다.

  3계층 아키텍처는 GCP에서 서비스를 설계하는 기본 뼈대다. Presentation · Logic · Data 세 층의 역할과 경계를 명확히 정해두면, 팀이 커지거나 기능이 늘어나도 구조가 흔들리지 않는다. 다음 Section에서는 이 설계도를 실제로 배포하는 과정을 자동화하는 CI/CD 파이프라인을 살펴본다.
- **핵심 키워드**: 3계층 아키텍처, Presentation·Logic·Data 계층, IAM 서비스 계정, 아키텍처 설계
- **시각화 연결**: 해당 없음
- **다음 섹션**: VI-1-2

### VI-1-2. CI/CD 파이프라인 — Cloud Build

- **난이도**: ★★★
- **선수 지식**: VI-1-1, IV-1-2
- **학습 목표**: 이 Section을 마치면 Cloud Build로 코드 커밋 → 이미지 빌드 → Cloud Run 배포까지 자동화할 수 있다.
- **설명**: 자동화가 없는 배포 현장을 상상해보자. 개발자가 코드를 수정할 때마다 터미널을 열고 `gcloud run deploy` 명령을 직접 입력한다. 작업이 반복될수록 명령어 옵션을 빠뜨리거나 잘못된 이미지 태그를 입력하는 실수가 생긴다. 팀원이 여럿이라면 누가 마지막으로 배포했는지, 어떤 버전이 지금 운영 중인지 파악하기도 어려워진다.

  **Cloud Build**는 이 문제를 해결하는 GCP의 CI/CD(지속적 통합·지속적 배포) 서비스다. CI/CD란 코드 변경이 발생할 때 빌드·테스트·배포 과정을 사람 개입 없이 자동으로 실행하는 방식을 말한다. GitHub 레포지터리에 코드를 push하면, Cloud Build가 변경을 감지해 Docker 이미지를 빌드하고 Artifact Registry에 업로드한 뒤 Cloud Run에 자동으로 배포한다. 이 파이프라인 전체를 `cloudbuild.yaml` 파일 하나로 정의한다. Part IV에서 익힌 Dockerfile 빌드와 Cloud Run 배포 명령이 여기서 자동화된 단계로 변환된다.

  아래 예시를 보면 파이프라인이 세 단계로 구성된다. 첫 번째 단계는 Docker 이미지를 빌드하고, 두 번째 단계는 Artifact Registry에 push하며, 세 번째 단계는 Cloud Run에 배포한다. `$PROJECT_ID`는 Cloud Build가 자동으로 현재 GCP 프로젝트 ID로 치환해 주므로 직접 입력할 필요가 없다.

  Cloud Build는 배포를 사람이 아닌 파일이 책임지게 만든다. 누가 언제 어떤 설정으로 배포했는지 Git 히스토리에 고스란히 남고, 실수로 인한 배포 오류가 줄어든다. 다음 Section에서는 이 파이프라인을 환경별로 분리해 운용하는 방법을 살펴본다.

  ```yaml
  # cloudbuild.yaml 예시
  steps:
    - name: 'gcr.io/cloud-builders/docker'
      args: ['build', '-t', 'gcr.io/$PROJECT_ID/my-api', '.']
    - name: 'gcr.io/cloud-builders/docker'
      args: ['push', 'gcr.io/$PROJECT_ID/my-api']
    - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
      args:
        - gcloud
        - run
        - deploy
        - my-api
        - --image=gcr.io/$PROJECT_ID/my-api
        - --region=asia-northeast3
  ```

- **핵심 키워드**: Cloud Build, CI/CD, `cloudbuild.yaml`, 자동 배포(Automated Deployment), GitHub 트리거
- **시각화 연결**: 해당 없음
- **다음 섹션**: VI-1-3

### VI-1-3. 환경 분리 — dev · staging · prod

- **난이도**: ★★★
- **선수 지식**: VI-1-2
- **학습 목표**: 이 Section을 마치면 GCP 프로젝트 또는 Cloud Run 서비스 이름으로 환경을 분리하고 각 환경별 배포 파이프라인을 구성할 수 있다.
- **설명**: 소프트웨어 공장에는 시제품 라인과 양산 라인이 따로 있다. 시제품 라인에서는 새 부품을 실험하고 불량이 나와도 큰 피해가 없다. 양산 라인은 검증된 설계만 들어온다. 실무 서비스도 마찬가지로, 개발 환경에서 실험하고 스테이징 환경에서 검증한 뒤 운영 환경으로 내보내는 흐름을 지킨다.

  실무에서는 **개발(dev) · 스테이징(staging) · 운영(prod)** 환경을 반드시 분리한다. 환경이 섞이면 개발 중인 불완전한 코드가 실제 사용자에게 노출되거나, 운영 DB의 데이터가 테스트 과정에서 오염될 수 있다. GCP에서 환경을 분리하는 방법은 두 가지다. 첫 번째는 **프로젝트 분리** 방식으로, 각 환경마다 별도의 GCP 프로젝트를 만든다. 비용 청구·IAM 권한·네트워크가 프로젝트 단위로 완전히 격리되어 운영 환경 실수 위험이 근본적으로 차단된다. 두 번째는 **서비스 이름 분리** 방식으로, 같은 프로젝트 안에서 Cloud Run 서비스 이름을 `my-api-dev`, `my-api-staging`, `my-api-prod`처럼 구분한다. Secret Manager의 Secret 이름도 환경 접미어를 붙여 `db-password-dev`, `db-password-prod`로 나눠두면 실수를 줄일 수 있다.

  소규모 팀은 서비스 이름 분리가 설정이 간단해 출발점으로 적합하다. 단, 같은 프로젝트 안에 있기 때문에 IAM 권한 실수 한 번으로 개발자가 운영 리소스에 접근하는 상황이 생길 수 있다. 팀 규모가 커지거나 규정 준수(Compliance) 요구가 생기면 프로젝트 분리로 전환하는 것이 낫다. VI-1-2에서 만든 `cloudbuild.yaml`을 브랜치별로 다르게 트리거하면, main 브랜치 push는 prod 환경에, develop 브랜치 push는 dev 환경에 자동으로 배포되도록 파이프라인을 구성할 수 있다.

  환경 분리는 사고를 예방하는 가장 기본적인 운영 규칙이다. 개발·스테이징·운영 세 층의 경계를 명확히 유지하면, 다음 Part에서 다루는 로깅과 모니터링도 환경별로 독립적으로 관리할 수 있다.
- **핵심 키워드**: 환경 분리(Environment Separation), dev·staging·prod, 프로젝트 분리, 서비스 이름 규칙
- **시각화 연결**: 해당 없음
- **다음 섹션**: VII-1-1

### VII-1-1. Cloud Logging으로 로그 확인하기

- **난이도**: ★★★
- **선수 지식**: IV-1-1
- **학습 목표**: 이 Section을 마치면 Cloud Logging에서 Cloud Run 로그를 조회하고, gcloud CLI로 필터링할 수 있다.
- **설명**: 레스토랑 주방을 생각해보자. 요리사가 주문을 받고, 어떤 재료를 썼고, 언제 서빙했는지 주방 일지에 기록해두면 나중에 문제가 생겼을 때 원인을 추적할 수 있다. Cloud Run이 처리하는 요청도 마찬가지다. 개발자가 별도 설정을 하지 않아도, Cloud Run은 서비스가 표준 출력(stdout)으로 내보내는 모든 내용을 자동으로 **Cloud Logging**에 기록한다.

  Cloud Logging은 GCP의 중앙 로그 수집·저장 서비스다. Cloud Run 외에도 Cloud SQL, Cloud Storage 등 GCP 서비스 대부분이 Cloud Logging으로 로그를 보낸다. 별도 로그 에이전트를 설치할 필요가 없다는 점이 가장 큰 장점이다. 로그는 Cloud Console의 **Logs Explorer**에서 시각적으로 조회하거나, 아래 gcloud 명령으로 터미널에서 바로 확인할 수 있다. 로그를 JSON 형식으로 출력하면 Logs Explorer가 severity(심각도) 필드를 자동으로 파싱해 INFO, WARNING, ERROR를 색으로 구분해 보여준다. IV-1-1에서 살펴본 요청 처리 6단계 중 마지막 단계, 즉 STEP 6의 "Cloud Logging 자동 기록" 단계가 바로 이 흐름이며, `index.html` 시각화에서 해당 단계를 단계별로 직접 확인할 수 있다.

  실무에서는 에러가 발생했을 때 Logs Explorer에서 `severity=ERROR` 필터를 걸어 원인을 빠르게 추적한다. `resource.type=cloud_run_revision`이라는 필터 조건은 Cloud Run이 내보낸 로그만 골라내는 기준이다. 여러 서비스가 동시에 운영되는 환경에서 특정 서비스의 로그만 뽑아내야 할 때, 이 조건에 서비스 이름이나 리비전(Revision)을 추가해 범위를 좁힌다. 아래 두 명령은 전체 로그 조회와 에러 로그 필터링을 차례로 보여준다.

  Cloud Logging은 로그를 모으는 창구다. 로그가 쌓이면 다음 단계는 수치 변화를 추적하는 모니터링이다. 장애는 로그에 먼저 흔적을 남기고, 그 흔적이 메트릭으로 드러나므로 Logging과 Monitoring은 항상 함께 활용한다.

  ```bash
  # 최근 Cloud Run 로그 10건 조회
  gcloud logging read "resource.type=cloud_run_revision" \
    --limit 10 \
    --format="table(timestamp, severity, textPayload)"

  # 에러 로그만 필터링
  gcloud logging read \
    "resource.type=cloud_run_revision AND severity=ERROR" \
    --limit 20
  ```

- **핵심 키워드**: Cloud Logging, Logs Explorer, `resource.type=cloud_run_revision`, severity, JSON 로그 형식
- **시각화 연결**: `index.html` (STEP 6 — Cloud Logging 자동 기록 단계)
- **다음 섹션**: VII-1-2

### VII-1-2. Cloud Monitoring — 알림과 대시보드

- **난이도**: ★★★
- **선수 지식**: VII-1-1
- **학습 목표**: 이 Section을 마치면 Cloud Monitoring으로 Cloud Run 요청 수·지연 시간·에러율 메트릭을 대시보드로 만들고 알림 정책을 설정할 수 있다.
- **설명**: 병원 중환자실에는 심박수·혈압·산소포화도를 실시간으로 보여주는 모니터가 있다. 수치가 위험 범위를 벗어나면 경보음이 울리고 의료진이 즉시 달려온다. **Cloud Monitoring**이 하는 일이 바로 이것이다. Cloud Run 서비스가 요청을 얼마나 받고 있는지, 응답이 얼마나 걸리는지, 에러가 얼마나 발생하는지를 실시간으로 수집하고 시각화한다.

  Cloud Run은 별도 설정 없이 네 가지 메트릭(Metric)을 Cloud Monitoring에 자동으로 전송한다. 요청 수(request count), 지연 시간(Latency), 인스턴스 수, 에러율이 그것이다. 이 수치들을 바탕으로 두 가지 핵심 기능을 활용한다. 첫째는 **알림 정책(Alerting Policy)**이다. 예를 들어 에러율이 5%를 초과하면 이메일이나 Slack으로 즉시 알림이 오도록 설정할 수 있다. 코드 배포 직후 에러율이 치솟는 상황을 담당자가 자리를 비운 사이에도 놓치지 않는다. 둘째는 **대시보드(Dashboard)**다. 여러 서비스의 메트릭을 한 화면에 배치해 운영 현황을 한눈에 파악할 수 있다.

  실무에서 알림 정책의 기준을 어떻게 잡느냐가 중요하다. 기준이 너무 낮으면 사소한 변동에도 알림이 쏟아져 담당자가 피로해지고, 너무 높으면 실제 장애를 놓친다. 일반적으로 서비스 초기에는 넉넉한 기준으로 시작해 실제 트래픽 패턴을 보면서 임계값을 조정해 나간다. 지연 시간의 경우 평균값이 아니라 **p99(상위 1% 느린 응답)** 기준으로 임계값을 설정하는 것이 더 실용적이다. 평균이 안정적이어도 일부 사용자가 극단적으로 느린 응답을 받고 있을 수 있기 때문이다.

  Cloud Monitoring은 Logging과 한 쌍이다. 로그에서 에러를 발견했을 때 메트릭에서 에러율이 언제부터 올랐는지를 확인하고, 반대로 메트릭 알림을 받았을 때 Logs Explorer에서 원인 로그를 찾는 흐름이 기본 운영 패턴이다. 다음 Section에서는 이렇게 모니터링하면서 새 버전을 안전하게 배포하는 방법을 살펴본다.
- **핵심 키워드**: Cloud Monitoring, 메트릭(Metric), 알림 정책(Alerting Policy), 대시보드(Dashboard), 지연 시간(Latency)
- **시각화 연결**: 해당 없음
- **다음 섹션**: VII-1-3

### VII-1-3. 트래픽 분산과 점진적 배포 (Traffic Splitting)

- **난이도**: ★★★
- **선수 지식**: IV-1-3
- **학습 목표**: 이 Section을 마치면 Cloud Run의 트래픽 분산 기능으로 카나리 배포(Canary Deployment)를 수행할 수 있다.
- **설명**: 신약을 검증할 때 처음부터 모든 환자에게 투약하지 않는다. 소수의 자원자에게 먼저 투여하고 부작용이 없는지 확인한 뒤 점진적으로 범위를 넓힌다. 서비스 배포도 같은 논리로 접근할 수 있다. 새 버전을 한 번에 전부 배포하면 버그가 있을 때 모든 사용자가 동시에 영향을 받는다. **트래픽 분산(Traffic Splitting)**은 새 버전에 트래픽의 일부만 먼저 보내 위험을 줄이는 방식이다.

  Cloud Run에서는 각 배포를 **리비전(Revision)**으로 관리한다. 새 이미지를 배포할 때마다 리비전이 하나씩 쌓이며, 기존 리비전은 삭제되지 않고 보존된다. gcloud 명령으로 리비전별 트래픽 비율을 숫자로 지정할 수 있다. 새 버전에 10%만 보내고 나머지 90%는 기존 버전이 처리하도록 설정한 뒤, 모니터링에서 에러율과 지연 시간이 정상이면 점차 비율을 높여 100%로 전환한다. 이 방식을 **카나리 배포(Canary Deployment)**라고 부른다. 과거 광부들이 탄광에 카나리아를 먼저 들여보내 유해 가스를 탐지하던 방식에서 이름을 따왔다.

  카나리 배포의 또 다른 장점은 **롤백(Rollback)**이 빠르다는 점이다. 10%의 트래픽을 새 버전으로 보내다가 에러가 급증하면, 아래 두 번째 명령처럼 `--to-latest` 대신 이전 리비전으로 100%를 다시 돌리면 된다. 기존 리비전이 살아있기 때문에 재배포 없이 트래픽 비율 조정만으로 롤백이 완료된다. IV-1-3에서 다룬 `--min-instances` 옵션과 조합하면, 카나리 배포 중 새 리비전에 최소 인스턴스를 0으로 유지해 비용을 아끼는 것도 가능하다.

  트래픽 분산은 Cloud Run이 기본 제공하는 기능이지만, 실제로 활용하려면 리비전 이름을 명확히 관리하는 습관이 필요하다. 배포할 때 `--tag` 옵션으로 의미 있는 이름을 붙여두면 나중에 트래픽을 조정할 때 어떤 리비전이 어떤 버전인지 혼동 없이 파악할 수 있다.

  ```bash
  # 새 배포 후 10% 트래픽만 새 리비전으로
  gcloud run services update-traffic my-api \
    --to-revisions=my-api-v2=10,my-api-v1=90 \
    --region=asia-northeast3

  # 문제 없으면 100% 전환
  gcloud run services update-traffic my-api \
    --to-latest \
    --region=asia-northeast3
  ```

- **핵심 키워드**: 트래픽 분산(Traffic Splitting), 카나리 배포(Canary Deployment), 리비전(Revision), 점진적 배포, 롤백(Rollback)
- **시각화 연결**: 해당 없음
- **다음 섹션**: VII-2-1

### VII-2-1. 비용 최적화 — Cloud Run 설정 튜닝

- **난이도**: ★★★
- **선수 지식**: IV-1-3, VII-1-2
- **학습 목표**: 이 Section을 마치면 Cloud Run 요청 수·CPU·메모리 설정을 조정해 불필요한 비용을 줄일 수 있다.
- **설명**: 음식점에서 낮 영업이 끝나고 저녁 장사 전 3시간 동안 주방을 통째로 비워두면 전기와 인건비가 낭비된다. 반면 영업 중에는 수요에 맞게 주방 인력을 탄력적으로 운영하면 비용을 아낄 수 있다. Cloud Run의 비용 최적화는 이 발상에서 시작한다. Cloud Run의 요금은 **요청 수 × CPU 사용 시간 × 메모리 사용량**으로 계산된다. 세 변수 모두 설정으로 제어할 수 있다는 뜻이기도 하다.

  비용을 줄이는 핵심 설정은 세 가지다. 첫째는 **Scale to Zero**다. `--min-instances 0`을 유지하면 트래픽이 없을 때 인스턴스가 완전히 종료되어 과금이 멈춘다. IV-1-3에서 다룬 콜드 스타트 지연을 감수하더라도 저비용이 우선인 서비스라면 기본값 그대로 두면 된다. 둘째는 메모리 설정이다. Cloud Run의 메모리 기본값은 256MiB인데, 가벼운 API 서버라면 실제 사용량을 확인하고 128MiB로 줄일 수 있다. Cloud Monitoring의 메모리 메트릭에서 실제 사용 피크를 보고, 피크의 약 1.5배를 기준으로 설정하는 것이 실무 관례다. 셋째는 **CPU 할당 방식**이다. `--cpu-throttling`을 적용하면 요청을 처리하는 동안에만 CPU가 할당되어 비용이 절감된다. 반면 웹소켓 유지 연결이나 백그라운드 배치 작업처럼 요청 사이에도 CPU가 필요한 앱이라면 `--no-cpu-throttling`을 선택해야 한다.

  이 세 가지 설정을 상황에 맞게 조합하면 같은 트래픽을 처리하면서도 비용을 크게 다르게 만들 수 있다. 예를 들어 야간에는 트래픽이 거의 없는 내부 어드민 서비스는 Scale to Zero에 128MiB 메모리와 `--cpu-throttling`을 적용하고, 응답 속도가 중요한 공개 API는 `--min-instances 1`로 최소 인스턴스를 유지하되 메모리는 실사용량에 맞게 조정하는 식이다.

  Part I에서 "클라우드란 무엇인가"라는 질문으로 시작해, 계정 설정·네트워크·보안·CI/CD를 거쳐 여기까지 왔다. Cloud Run 서비스를 배포하고, 로그와 메트릭으로 운영 현황을 파악하며, 비용 설정을 튜닝하는 것까지 익혔다면 GCP로 서비스를 배포하고 운영하는 기본기는 충분히 갖춰진 것이다. 이후 규모가 커지거나 요구사항이 복잡해질 때, 이 기초 위에서 필요한 개념을 하나씩 쌓아 나가면 된다.
- **핵심 키워드**: 비용 최적화(Cost Optimization), Scale to Zero, `--min-instances`, `--cpu-throttling`, 메모리 설정
- **시각화 연결**: 해당 없음
- **다음 섹션**: VIII-1-1

## Part VIII. Compute Engine — VM 기반 인프라

### VIII-1-1. Compute Engine이란? — IaaS와 VM

- **난이도**: ★★☆
- **선수 지식**: I-1-3, III-1-2
- **학습 목표**: 이 Section을 마치면 Compute Engine과 Cloud Run의 차이를 설명하고, VM 기반 배포가 필요한 상황을 판단할 수 있다.
- **설명**: Part I에서 IaaS를 "피자 재료를 직접 사다가 오븐을 켜는 방식"으로 설명했다. 밀가루 배합도, 굽는 온도도, 토핑도 내 손에 달려 있다. 그만큼 자유롭지만 해야 할 일도 많다. GCP의 **Compute Engine**이 바로 이 역할이다. 원하는 사양의 가상 서버(VM)를 직접 골라 운영체제부터 세팅하는 IaaS 서비스다.

  **Compute Engine**은 GCP의 가상 머신(VM, Virtual Machine) 서비스다. 사용자는 CPU 개수, 메모리 용량, OS(운영체제) 종류를 선택해 VM을 만들고, 그 안에서 무엇을 설치하고 어떻게 설정할지를 직접 결정한다. 반면 Cloud Run은 PaaS다. 컨테이너 이미지를 올리면 플랫폼이 서버 관리·스케일링·트래픽 처리를 알아서 맡는다. Compute Engine은 그 반대 극단에 있다. OS 패치, 보안 설정, 소프트웨어 설치, 프로세스 관리까지 개발자가 직접 책임진다. 자유도가 높은 대신 운영 부담도 그만큼 크다. VM의 사양을 결정하는 단위가 **머신 타입(Machine Type)**이다. 실습에서 가장 많이 쓰는 `e2-micro`는 GCP 무료 체험 한도에 포함되는 가장 작은 타입으로, 소규모 학습용 서비스를 띄우기에 충분하다.

  Cloud Run으로 처리하기 어려운 워크로드가 Compute Engine의 무대다. GPU가 필요한 머신러닝 학습 작업은 Cloud Run이 지원하지 않는 GPU 머신 타입을 Compute Engine에서 선택할 수 있다. 특정 OS 버전이나 커널 설정을 요구하는 레거시 앱도 VM 위에서 직접 맞춤 설정해 실행할 수 있다. 몇 시간씩 돌아가는 배치(Batch) 작업처럼 Scale to Zero가 오히려 불편한 경우에도 Compute Engine이 적합하다. Cloud Run이 "요청이 올 때만 깨어나는 서비스"라면, Compute Engine은 "항상 켜져 있는 내 서버"에 가깝다.

  Compute Engine은 제어가 필요한 상황에서 선택하는 IaaS 서비스다. 대부분의 웹 서비스는 Cloud Run으로 충분하지만, 지금부터 VM 기반 배포가 어떻게 동작하는지 직접 확인한다. 다음 Section에서는 Compute Engine 인스턴스를 생성하고 방화벽 규칙을 설정하는 실습을 진행한다.
- **핵심 키워드**: Compute Engine, 가상 머신(VM), IaaS, 머신 타입(Machine Type), e2-micro
- **시각화 연결**: 해당 없음
- **다음 섹션**: VIII-1-2

### VIII-1-2. Compute Engine 인스턴스 생성과 방화벽 규칙

- **난이도**: ★★☆
- **선수 지식**: VIII-1-1
- **학습 목표**: 이 Section을 마치면 gcloud 명령으로 Compute Engine 인스턴스를 생성하고, 방화벽 규칙으로 특정 포트를 열 수 있다.
- **설명**: 새 건물에 입주하면 처음에는 현관문이 잠겨 있다. 특정 사람이 들어올 수 있도록 열쇠를 나눠주거나 비밀번호를 공유해야 한다. Compute Engine 인스턴스도 비슷하다. VM을 만들었다고 곧바로 외부에서 접속할 수 있는 것이 아니다. GCP 방화벽이 기본적으로 외부 트래픽을 모두 차단하고 있어서, 필요한 포트를 명시적으로 열어야 비로소 웹 요청을 받을 수 있다.

  Compute Engine 인스턴스를 만들 때는 세 가지를 지정해야 한다. 첫째는 **존(Zone)**이다. 물리적 데이터센터 위치를 결정하며, 한국 서비스라면 `asia-northeast3-a`처럼 서울 리전의 존을 선택한다. 둘째는 **머신 타입**이다. CPU와 메모리 조합을 결정한다. `e2-micro`는 가장 작은 타입으로 학습·테스트에 적합하다. 셋째는 **OS 이미지**다. `debian-12` 같은 이미지 패밀리와 해당 이미지를 제공하는 프로젝트(`debian-cloud`)를 함께 지정한다. 인스턴스를 만든 뒤에는 방화벽 규칙을 추가해야 한다. `gcloud compute firewall-rules create` 명령으로 어떤 포트를 허용할지, 어느 태그가 붙은 인스턴스에 적용할지를 설정한다. **네트워크 태그**가 일치하는 인스턴스에만 방화벽 규칙이 적용되어, 규칙을 세밀하게 제어할 수 있다.

  실무 관점에서 이 과정은 AWS EC2 보안 그룹 설정과 유사하다. 핵심은 "허용하지 않으면 모두 차단"이라는 기본 원칙이다. HTTP(80포트)를 열면 인터넷에서 VM으로 웹 요청이 들어온다. HTTPS(443포트)는 추가로 열어야 하며, SSH(22포트)는 `gcloud compute ssh` 명령이 자동으로 처리하므로 별도 규칙 없이도 접속할 수 있다.

  인스턴스 생성과 방화벽 규칙 설정, 이 두 단계를 마쳐야 외부에서 VM에 접근할 수 있는 환경이 완성된다. 아래 명령이 그 순서를 그대로 보여준다. 다음 Section에서는 이 인스턴스에 직접 접속해 앱을 실행한다.

  ```bash
  # Compute Engine 인스턴스 생성
  gcloud compute instances create my-instance \
    --zone=asia-northeast3-a \
    --machine-type=e2-micro \
    --image-family=debian-12 \
    --image-project=debian-cloud

  # 방화벽 규칙 생성 (HTTP 80포트 허용)
  gcloud compute firewall-rules create allow-http \
    --allow=tcp:80 \
    --target-tags=http-server
  ```

- **핵심 키워드**: `gcloud compute instances create`, 머신 타입, 이미지 패밀리, 방화벽 규칙(Firewall Rules), 네트워크 태그
- **시각화 연결**: 해당 없음
- **다음 섹션**: VIII-1-3

### VIII-1-3. 예제 앱을 Compute Engine에서 실행하기

- **난이도**: ★★★
- **선수 지식**: VIII-1-2
- **학습 목표**: 이 Section을 마치면 Compute Engine에 SSH로 접속해 앱을 빌드·실행하고, 외부에서 접근할 수 있다.
- **설명**: Cloud Run에 앱을 올릴 때는 Dockerfile을 작성하고 컨테이너 이미지를 빌드해야 했다. Compute Engine은 다르다. 집 안에 직접 들어가 소프트웨어를 설치하는 방식이다. SSH로 VM에 접속하고, 패키지를 설치하고, 앱을 실행한다. 로컬 PC에서 개발 서버를 띄우는 과정과 거의 같지만, 그 대상이 클라우드 위에 있는 가상 서버라는 점만 다르다.

  **SSH(Secure Shell)**는 원격 서버에 암호화된 방식으로 접속하는 프로토콜이다. `gcloud compute ssh` 명령을 쓰면 SSH 키 생성과 업로드를 GCP가 자동으로 처리하므로, 별도 키 파일을 관리할 필요가 없다. 인스턴스 내부에 접속한 뒤에는 일반 리눅스 서버와 동일하게 동작한다. `apt`로 패키지를 설치하고, `pip`로 Python 라이브러리를 설치하고, 스크립트를 실행하면 된다. 한 가지 중요한 차이가 있다. VM을 재시작하면 실행 중이던 프로세스가 모두 종료된다. 운영 환경에서는 **systemd** 같은 서비스 관리자로 앱을 등록해 부팅 시 자동으로 시작되도록 설정해야 안정적으로 운영할 수 있다. Cloud Run은 이런 설정이 플랫폼 안에 내장되어 있었지만, Compute Engine에서는 직접 구성해야 하는 부분이다.

  아래 명령 흐름이 Compute Engine 앱 배포의 전형적인 패턴이다. SSH로 접속해 패키지를 설치하고 앱을 실행한다. 80포트에서 요청을 받으려면 Flask 등 앱 코드에서 `0.0.0.0:80`으로 바인딩하거나, 앞 Section에서 열어둔 방화벽 포트와 일치하는 포트를 지정해야 한다. 외부 IP와 해당 포트로 브라우저에서 접근하면 앱이 응답한다.

  Compute Engine에서 앱을 실행하는 과정은 IaaS의 특성을 그대로 보여준다. 편의성은 Cloud Run보다 낮지만, OS와 네트워크 설정을 원하는 대로 제어할 수 있다. 다음 Section에서는 이 VM이 속한 네트워크를 더 세밀하게 설계하는 Subnet 개념을 살펴본다.

  ```bash
  # Compute Engine에 SSH 접속
  gcloud compute ssh my-instance --zone=asia-northeast3-a

  # 인스턴스 내부에서 앱 실행
  sudo apt update && sudo apt install -y python3-pip
  pip3 install flask
  python3 app.py
  ```

- **핵심 키워드**: `gcloud compute ssh`, SSH, 패키지 설치, 포트 바인딩, systemd
- **시각화 연결**: 해당 없음
- **다음 섹션**: VIII-2-1

### VIII-2-1. Public Subnet과 Private Subnet

- **난이도**: ★★★
- **선수 지식**: V-1-1, VIII-1-1
- **학습 목표**: 이 Section을 마치면 Public Subnet과 Private Subnet의 차이를 설명하고, 보안을 위해 서비스를 Private Subnet에 배치하는 이유를 이해할 수 있다.
- **설명**: 회사 사무실 건물을 다시 떠올려보자. V-1-1에서 VPC를 이 건물에 비유했다. 건물 1층에는 누구나 들어올 수 있는 안내 데스크와 회의실이 있다. 외부 손님이 방문하면 1층에서 접수하고, 실제 업무가 이루어지는 2층 이상 개발실에는 사원증이 있는 직원만 출입할 수 있다. GCP의 서브넷(Subnet)도 같은 원리로 나뉜다. **Public Subnet**은 1층 안내 데스크처럼 외부 인터넷에서 직접 접근 가능한 공간이고, **Private Subnet**은 카드키가 없으면 들어갈 수 없는 내부 사무실이다.

  VPC 안의 서브넷은 외부 인터넷과의 연결 여부에 따라 두 종류로 구분된다. Public Subnet에 배치된 인스턴스는 **외부 IP(External IP)**를 부여받아 인터넷에서 직접 접근할 수 있다. 반면 Private Subnet의 인스턴스는 외부 IP가 없어 인터넷에서 직접 도달하지 못한다. 단, Private Subnet의 인스턴스가 외부로 아웃바운드 요청을 보낼 때는 **NAT Gateway**를 경유한다. NAT Gateway는 내부 IP를 외부 IP로 변환해 외부와 통신하되, 외부에서의 인바운드 접근은 차단하는 역할을 한다.

  실무에서는 이 두 영역을 목적에 따라 나눠 배치한다. 웹 서버나 로드 밸런서처럼 외부 요청을 처음 받아야 하는 컴포넌트만 Public Subnet에 두고, DB 서버나 내부 API 서버처럼 외부 노출이 불필요한 컴포넌트는 Private Subnet에 배치한다. 이렇게 하면 설령 웹 서버가 해킹당해도 공격자가 내부 DB에 직접 도달하지 못한다. 외부에서 접근 가능한 표면, 즉 **공격 표면(Attack Surface)**을 최소화하는 것이 이 설계의 핵심 목적이다.

  서브넷 분리는 V-1-1에서 배운 방화벽 규칙과 함께 작동한다. 서브넷으로 공간을 나누고, 방화벽 규칙으로 어떤 트래픽이 서브넷 경계를 넘을 수 있는지 정의한다. 다음 Section에서는 Private Subnet에 배치한 인스턴스에 관리자가 안전하게 접근하는 방법을 살펴본다.
- **핵심 키워드**: Public Subnet, Private Subnet, 외부 IP(External IP), NAT Gateway, 공격 표면(Attack Surface)
- **시각화 연결**: 해당 없음
- **다음 섹션**: VIII-2-2

### VIII-2-2. Private Compute Engine과 Bastion Host

- **난이도**: ★★★
- **선수 지식**: VIII-2-1
- **학습 목표**: 이 Section을 마치면 Private Subnet에 Compute Engine을 배치하고, Bastion Host를 통해 안전하게 접근할 수 있다.
- **설명**: Private Subnet에 인스턴스를 배치하면 외부 공격 표면을 줄일 수 있다. 그런데 관리자는 어떻게 이 인스턴스에 접근할 수 있을까. 외부 IP가 없으니 집에서 SSH로 바로 접속하는 방법은 통하지 않는다. 이 문제를 해결하는 전통적인 방법이 **Bastion Host** 패턴이다. 군사 용어에서 따온 이 이름은 '요새'를 뜻하는데, 외부에서 내부로 들어가는 유일한 관문 역할을 한다.

  Bastion Host는 Public Subnet에 배치하는 인스턴스다. 이 인스턴스만 외부 IP를 가지며, 관리자가 인터넷에서 SSH로 접근할 수 있다. Bastion Host에 접속한 뒤, Bastion 내부에서 다시 Private Subnet의 인스턴스로 내부 IP를 통해 SSH 접속하는 **2단계 SSH** 구조가 완성된다. 외부에서 Private 인스턴스로 가는 직통 경로는 존재하지 않고, 반드시 Bastion이라는 검문소를 거쳐야 한다. V-1-1에서 배운 방화벽 규칙으로 "Bastion Host에서 오는 SSH만 허용"을 Private Subnet 인스턴스에 설정하면 보안 경계가 완성된다.

  아래 명령은 Bastion Host 생성과 접속 흐름을 보여준다. 첫 번째 명령에서 `--subnet=public-subnet`과 `--tags=bastion`을 지정해 Public Subnet에 bastion 태그를 달고, 두 번째 명령에서 Bastion에 SSH 접속 후 내부 IP `10.0.2.3`으로 Private 인스턴스에 재접속한다. `10.0.2.3`은 Private Subnet의 인스턴스에 자동 할당된 내부 IP 예시다.

  Bastion Host는 검증된 패턴이지만 관리 부담이 하나 더 생긴다. Bastion 자체도 OS 패치와 보안 관리가 필요하고, 키 관리가 복잡해진다. 다음 Section에서는 Bastion 없이 더 간결하게 Private 인스턴스에 접근하는 방법을 살펴본다.

  ```bash
  # Public Subnet에 Bastion Host 생성
  gcloud compute instances create bastion-host \
    --zone=asia-northeast3-a \
    --subnet=public-subnet \
    --tags=bastion

  # Bastion을 통해 Private 인스턴스에 접속
  gcloud compute ssh bastion-host --zone=asia-northeast3-a
  ssh 10.0.2.3
  ```

- **핵심 키워드**: Bastion Host, 점프 서버(Jump Server), Private Subnet, 2단계 SSH, 보안 경계
- **시각화 연결**: 해당 없음
- **다음 섹션**: VIII-2-3

### VIII-2-3. IAP로 Private 인스턴스에 접근하기

- **난이도**: ★★★
- **선수 지식**: VIII-2-2
- **학습 목표**: 이 Section을 마치면 IAP(Identity-Aware Proxy) 터널을 사용해 Bastion Host 없이 Private 인스턴스에 안전하게 접근할 수 있다.
- **설명**: Bastion Host는 외부에서 Private 인스턴스에 접근하는 문제를 해결하지만, Bastion 자체가 하나의 관리 대상이 된다. GCP는 이 한계를 넘는 방법으로 **IAP(Identity-Aware Proxy)**를 제공한다. IAP는 GCP가 관리하는 프록시 서비스로, Bastion Host 없이 Private 인스턴스에 직접 SSH 접속할 수 있게 해준다. 비유하자면, 사원증 발급 창구를 회사 내부에 두는 대신 구글 본사가 클라우드에서 신분 확인을 대신 해주는 구조다.

  IAP의 작동 방식은 간단하다. `--tunnel-through-iap` 옵션으로 SSH를 시도하면, gcloud가 GCP의 IAP 서버로 암호화된 터널을 연다. IAP는 구글 계정 인증을 거친 사용자인지 확인하고, IAM에서 해당 사용자가 `iap.tunnelResourceAccessor` 권한을 가지고 있는지 검증한다. 인증과 권한 확인이 모두 통과하면 터널이 열려 Private 인스턴스에 접속된다. 인터넷에서 인스턴스로 향하는 직접 경로는 여전히 없고, 모든 트래픽은 GCP 내부 네트워크를 통해 IAP 경유로만 흐른다.

  방화벽 설정은 간결해진다. Private 인스턴스의 방화벽 규칙에서 IAP의 IP 대역인 `35.235.240.0/20`에서 오는 TCP 22번 포트만 허용하면 된다. Bastion Host처럼 별도 인스턴스를 만들 필요도 없고, SSH 키를 Bastion에 따로 관리할 필요도 없다. V-1-2에서 다룬 IAM 권한 체계가 여기서 실전으로 적용된다. IAP는 **Zero Trust** 보안 모델의 GCP 구현이다. "네트워크 안에 있으면 신뢰한다"는 전통적인 경계 보안 대신, "모든 접근 시도를 항상 인증한다"는 원칙을 따른다.

  IAP는 Bastion Host보다 관리가 단순하고 보안 수준은 더 높다. 실무에서 새 프로젝트를 시작할 때 IAP를 기본 접근 방식으로 채택하는 것이 권장된다. Private Subnet으로 공격 표면을 줄이고, IAP로 안전한 접근 통로를 확보하는 것이 GCP에서 Compute Engine을 운영하는 보안 기본기다.

  ```bash
  # IAP를 통한 SSH 접속 (Bastion 없이)
  gcloud compute ssh private-instance \
    --zone=asia-northeast3-a \
    --tunnel-through-iap

  # IAP 방화벽 규칙 (IAP IP 대역 허용)
  gcloud compute firewall-rules create allow-iap-ssh \
    --allow=tcp:22 \
    --source-ranges=35.235.240.0/20
  ```

- **핵심 키워드**: IAP(Identity-Aware Proxy), `--tunnel-through-iap`, 35.235.240.0/20, Zero Trust
- **시각화 연결**: 해당 없음
- **다음 섹션**: IX-1-1

### IX-1-1. Service Account 생성과 권한 부여

- **난이도**: ★★★
- **선수 지식**: V-1-2
- **학습 목표**: 이 Section을 마치면 Service Account를 생성하고, 특정 GCP 서비스에 대한 역할을 부여할 수 있다.
- **설명**: 카페에서 아르바이트생을 처음 고용하면 업주는 매장 열쇠를 통째로 넘기지 않는다. 에스프레소 머신 사용 권한, POS 단말기 접근 권한은 주되, 금고 열쇠와 직원 급여 내역은 별도로 보관한다. 역할에 꼭 필요한 것만 건네는 것이 보안의 기본이다. GCP에서 **Service Account**도 같은 원리로 동작한다. 사람이 아닌 앱이나 서비스가 다른 GCP 리소스에 접근할 때 사용하는 계정으로, 각 서비스에 딱 필요한 권한만 부여해 운영한다.

  Part V에서 IAM의 개념과 최소 권한 원칙을 이론으로 배웠다. 이 Section은 그 원칙을 실제로 적용하는 첫 실습이다. Cloud Run이 Cloud SQL에 접근하거나 Compute Engine이 Cloud Storage에 파일을 업로드할 때, 서비스 스스로 GCP에 인증을 해야 한다. 이때 개발자 개인 계정을 쓰면 그 계정이 탈취됐을 때 모든 권한이 노출된다. Service Account는 역할을 특정 서비스 하나에만 묶어두기 때문에 피해 범위를 좁힐 수 있다.

  Service Account를 만드는 과정은 두 단계다. 먼저 `gcloud iam service-accounts create` 명령으로 계정 자체를 생성한다. 이름과 표시명을 지정하면 `이름@프로젝트ID.iam.gserviceaccount.com` 형태의 이메일 주소가 만들어진다. 다음으로 `gcloud projects add-iam-policy-binding` 명령에서 `--member`에 방금 만든 Service Account 이메일을, `--role`에 부여할 역할을 지정한다. 예시에서는 `roles/cloudsql.client`를 부여한다. 이 역할은 Cloud SQL에 쿼리를 날리는 것만 허용하고 인스턴스 생성·삭제 같은 관리 권한은 포함하지 않는다. 카페 아르바이트생에게 에스프레소 머신만 쓸 수 있는 열쇠를 준 것과 같다.

  Service Account는 GCP IAM 실습의 출발점이다. 최소 권한 원칙을 아는 것과 실제로 역할 바인딩 명령을 한 번 실행해보는 것은 감각이 다르다. 다음 Section에서는 사람 계정에 대한 접근 제어, 즉 팀원에게 역할을 부여하는 방법을 실습한다.

  ```bash
  # Service Account 생성
  gcloud iam service-accounts create my-sa \
    --display-name="My Service Account"

  # Service Account에 역할 부여
  gcloud projects add-iam-policy-binding MY_PROJECT_ID \
    --member="serviceAccount:my-sa@MY_PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/cloudsql.client"
  ```

- **핵심 키워드**: Service Account, `gcloud iam service-accounts create`, 역할 바인딩, IAM 정책
- **시각화 연결**: 해당 없음
- **다음 섹션**: IX-1-2

### IX-1-2. 특정 사용자의 리소스 접근 제어

- **난이도**: ★★★
- **선수 지식**: IX-1-1
- **학습 목표**: 이 Section을 마치면 특정 사용자에게 특정 역할만 부여하고, 현재 프로젝트의 IAM 정책을 조회할 수 있다.
- **설명**: 팀에 새 백엔드 개발자가 합류했다. 이 개발자는 Cloud Run 배포 상태를 모니터링해야 하지만, Cloud SQL 데이터를 직접 수정하거나 IAM 정책을 바꿀 필요는 없다. 회사 서버실 출입증을 발급해줄 때 일반 직원에게는 서버 조회 권한만 주고, 재부팅이나 설정 변경 권한은 운영팀 전용으로 남겨두는 것과 같다. GCP에서 사람 계정에 역할을 부여하는 방식도 이 출입증 발급 논리와 동일하다.

  `gcloud projects add-iam-policy-binding` 명령 하나로 특정 사용자에게 원하는 역할을 부여할 수 있다. `--member` 옵션에는 `user:이메일` 형태로 대상을 지정하고, `--role`에는 부여할 역할을 명시한다. 예시에서는 `roles/run.viewer`를 부여한다. 이 역할은 Cloud Run 서비스의 상태와 로그를 조회할 수 있지만 배포하거나 삭제할 수는 없다. Owner나 Editor처럼 프로젝트 전체에 광범위한 권한을 주는 역할은 실수나 사고가 생겼을 때 파급 범위가 넓어진다. 필요한 서비스에, 필요한 수준의 역할만 부여하는 것이 최소 권한 원칙의 실천이다.

  IAM 정책을 부여한 뒤에는 반드시 현재 상태를 확인하는 습관이 필요하다. `gcloud projects get-iam-policy` 명령은 프로젝트에 설정된 모든 IAM 정책 목록을 출력한다. 누구에게 어떤 역할이 부여돼 있는지 한눈에 파악할 수 있어, 권한이 의도치 않게 과도하게 설정됐거나 더 이상 필요 없는 역할이 남아 있지 않은지 점검할 수 있다. 실무에서는 새 구성원이 합류하거나 퇴직할 때마다 이 명령으로 정책을 검토하는 것이 보안 운영의 기본 루틴이다.

  사람 계정에 대한 IAM 바인딩은 Service Account 바인딩과 명령 구조가 같다. `--member`의 접두사가 `serviceAccount:`에서 `user:`로 바뀔 뿐이다. 이 구조를 한 번 이해하면 그룹 이메일(`group:`)이나 도메인 전체(`domain:`)로 역할을 부여하는 방식도 자연스럽게 확장할 수 있다.

  ```bash
  # 특정 사용자에게 Cloud Run Viewer 역할 부여
  gcloud projects add-iam-policy-binding MY_PROJECT_ID \
    --member="user:developer@example.com" \
    --role="roles/run.viewer"

  # 현재 IAM 정책 확인
  gcloud projects get-iam-policy MY_PROJECT_ID
  ```

- **핵심 키워드**: IAM Policy Binding, `--member`, `--role`, Owner vs Editor vs Viewer, 최소 권한
- **시각화 연결**: 해당 없음
- **다음 섹션**: IX-2-1

### IX-2-1. VPC 직접 생성하기

- **난이도**: ★★★
- **선수 지식**: V-1-1
- **학습 목표**: 이 Section을 마치면 Custom 모드 VPC를 생성하고 서브넷을 추가할 수 있다.
- **설명**: 신도시 택지 개발을 상상해보자. 토지를 처음 조성할 때 도로망을 먼저 설계한다. 간선 도로를 어디에 놓을지, 주거지와 상업지를 어떻게 나눌지, 각 구역에 어떤 IP 주소 체계에 해당하는 블록 번호를 부여할지 계획한다. GCP 역시 프로젝트를 처음 만들면 기본 VPC를 자동으로 제공한다. 이 기본 VPC는 편리하지만, 신도시 기본 도로망처럼 모든 구역에 동일한 규격으로 깔려 있어 섬세한 네트워크 설계가 어렵다. 실무에서는 IP 대역을 직접 설계하고 서브넷을 용도별로 나누는 **Custom 모드 VPC**를 사용하는 것이 일반적이다.

  V-1-1에서 VPC의 개념을 배우고, VIII-2-1에서 Public Subnet과 Private Subnet의 역할을 살펴봤다. 이 Section은 그 지식을 명령어로 직접 구현하는 실습이다. `gcloud compute networks create` 명령에 `--subnet-mode=custom` 옵션을 붙이면 서브넷이 자동 생성되지 않는 빈 VPC가 만들어진다. 자동 모드와 달리 서브넷을 처음부터 하나씩 직접 추가해야 한다. 그만큼 어떤 리전에 어떤 IP 대역을 배치할지 설계 의도가 네트워크 구성에 그대로 반영된다.

  서브넷을 추가할 때는 `gcloud compute networks subnets create` 명령으로 VPC 이름, 리전, CIDR(Classless Inter-Domain Routing) 범위를 지정한다. 예시에서는 서울 리전(`asia-northeast3`)에 `10.0.1.0/24` 대역을 할당한다. `/24`는 256개의 IP 주소를 쓸 수 있는 블록을 의미한다. 실무에서는 웹 서버용 Public 서브넷과 DB 서버용 Private 서브넷을 다른 CIDR로 분리해 구성한다. IP 대역을 미리 체계적으로 설계해 두면 나중에 서브넷이 늘어나도 충돌 없이 관리할 수 있다.

  Custom VPC를 직접 만들어봄으로써 자동으로 주어지던 네트워크 구조가 어떤 원리로 동작하는지 이해하게 된다. 다음 Section에서는 이 VPC에 방화벽 규칙을 추가해 트래픽 흐름을 직접 제어한다.

  ```bash
  # Custom 모드 VPC 생성
  gcloud compute networks create my-vpc \
    --subnet-mode=custom

  # Subnet 생성
  gcloud compute networks subnets create my-subnet \
    --network=my-vpc \
    --region=asia-northeast3 \
    --range=10.0.1.0/24
  ```

- **핵심 키워드**: Custom VPC, `--subnet-mode=custom`, CIDR(10.0.1.0/24), 서브넷 설계
- **시각화 연결**: 해당 없음
- **다음 섹션**: IX-2-2

### IX-2-2. Subnet 추가와 방화벽 규칙 관리

- **난이도**: ★★★
- **선수 지식**: IX-2-1
- **학습 목표**: 이 Section을 마치면 VPC에 방화벽 규칙을 추가해 내부 통신과 외부 접근을 제어할 수 있다.
- **설명**: 아파트 단지를 생각해보자. 단지 안에 동별로 구역이 나뉘어 있고, 단지 입구 경비실에서는 외부 방문자를 체크한다. 외부인은 경비원 확인을 거쳐야 들어올 수 있고, 단지 내 주민끼리는 자유롭게 이동한다. 특정 동 거주자만 출입할 수 있는 커뮤니티 시설도 있다. GCP의 **방화벽 규칙(Firewall Rules)**이 이 경비실의 역할을 한다. VPC 안을 드나드는 트래픽이 어느 경로로 흐를 수 있는지, 어떤 포트로 누구에게 열려 있는지를 규칙으로 명시한다.

  VPC를 만들고 서브넷을 추가했으면, 다음 단계는 트래픽 제어다. 방화벽 규칙은 크게 두 종류가 필요하다. 첫째는 **내부 통신 규칙**이다. `allow-internal` 규칙은 `10.0.0.0/16` 대역, 즉 VPC 내부 서브넷끼리의 TCP, UDP, ICMP 통신을 허용한다. 같은 VPC 안의 웹 서버와 DB 서버가 서로 통신하려면 이 규칙이 반드시 있어야 한다. 둘째는 **외부 SSH 접근 규칙**이다. `allow-ssh` 규칙은 TCP 22번 포트를 `0.0.0.0/0`, 즉 전체 인터넷에서 허용한다. 관리자가 어디서든 SSH로 인스턴스에 접근할 수 있게 하기 위해서다.

  `--source-ranges` 옵션이 핵심이다. 이 옵션으로 트래픽의 출발지 IP 대역을 지정한다. SSH 규칙에서 `0.0.0.0/0` 대신 관리자 IP 주소만 허용하면 보안이 크게 강화된다. 내부 통신 규칙에서도 `10.0.0.0/16`이라는 내부 대역만 허용함으로써, 외부에서 흘러 들어온 패킷이 내부 서브넷 사이를 가로질러 이동하는 것을 원천적으로 막는다. V-1-1에서 방화벽 규칙의 개념을, VIII-2-1에서 Public/Private Subnet 분리 원리를 배웠다면, 이 Section에서 그 설계를 실제 명령어로 구현하는 셈이다.

  Custom VPC에 서브넷을 추가하고 방화벽 규칙을 직접 설정해보면, 자동으로 생성된 기본 VPC의 구성이 왜 그런 형태였는지 자연스럽게 이해된다. 네트워크 설계와 보안 규칙은 이론으로 읽을 때와 명령어를 직접 실행해볼 때 이해의 깊이가 달라진다.

  ```bash
  # 내부 통신 허용 방화벽 규칙
  gcloud compute firewall-rules create allow-internal \
    --network=my-vpc \
    --allow=tcp,udp,icmp \
    --source-ranges=10.0.0.0/16

  # 외부 SSH 허용
  gcloud compute firewall-rules create allow-ssh \
    --network=my-vpc \
    --allow=tcp:22 \
    --source-ranges=0.0.0.0/0
  ```

- **핵심 키워드**: 방화벽 규칙, `--source-ranges`, Ingress/Egress, 내부 통신, SSH 허용
- **시각화 연결**: 해당 없음
- **다음 섹션**: X-1-1

### X-1-1. 인스턴스 템플릿·그룹·Health Check

- **난이도**: ★★★
- **선수 지식**: VIII-1-2, V-2-1
- **학습 목표**: 이 Section을 마치면 인스턴스 템플릿으로 동일한 VM을 자동 생성하는 인스턴스 그룹을 만들고, Health Check를 설정할 수 있다.
- **설명**: 같은 모양의 가구를 여러 개 만들 때 매번 치수를 새로 재고 재단하지 않는다. 한 번 만들어 놓은 금형(틀)에서 동일한 제품을 찍어낸다. **인스턴스 템플릿(Instance Template)**이 바로 이 금형에 해당한다. VIII-1-2에서 인스턴스를 만들 때 지정했던 머신 타입, OS 이미지, 네트워크 태그를 하나의 틀로 저장해두면, 이후에는 이 틀을 기준으로 동일한 VM을 몇 대든 자동으로 생성할 수 있다.

  **관리형 인스턴스 그룹(MIG, Managed Instance Group)**은 이 템플릿을 사용해 복수의 VM을 묶어 관리하는 단위다. `--size=2`로 지정하면 동일한 설정의 VM 두 대가 자동으로 만들어진다. 인스턴스 그룹은 트래픽 증가 시 VM을 자동으로 늘리고(Auto Scaling), 줄어들 때는 다시 줄인다. Cloud Run의 자동 확장이 컨테이너 단위로 이루어진다면, MIG는 VM 단위로 같은 역할을 한다. Load Balancer와 연결하면 MIG 안의 VM들이 트래픽을 나눠 처리하는 구조가 완성된다.

  그런데 VM이 여러 대 있다고 해서 모두 정상 동작하고 있다는 보장은 없다. 소프트웨어 오류로 VM이 멈추거나 응답이 느려질 수 있다. **Health Check**는 각 인스턴스에 주기적으로 HTTP 요청을 보내 응답이 정상인지 확인하는 감시 장치다. 응답하지 않는 인스턴스를 감지하면 자동으로 삭제하고 새 인스턴스를 생성한다. 이 기능을 **Auto Healing(자동 복구)**이라고 한다. 서버 한 대가 고장나도 서비스가 중단되지 않는 이유가 여기에 있다.

  인스턴스 템플릿, MIG, Health Check는 단독으로 쓰이지 않는다. 세 요소가 함께 연결될 때 고가용성 Compute Engine 클러스터가 완성된다. 다음 Section에서는 이 그룹을 Load Balancer에 연결해 외부 트래픽을 분산하는 마지막 단계를 진행한다.

  ```bash
  # 인스턴스 템플릿 생성
  gcloud compute instance-templates create my-template \
    --machine-type=e2-micro \
    --image-family=debian-12 \
    --image-project=debian-cloud \
    --tags=http-server

  # 인스턴스 그룹 생성
  gcloud compute instance-groups managed create my-group \
    --template=my-template \
    --size=2 \
    --zone=asia-northeast3-a

  # Health Check 생성
  gcloud compute health-checks create http my-health-check \
    --port=80
  ```

- **핵심 키워드**: 인스턴스 템플릿(Instance Template), 관리형 인스턴스 그룹(MIG), Health Check, Auto Healing
- **시각화 연결**: 해당 없음
- **다음 섹션**: X-1-2

### X-1-2. Load Balancer 생성하기

- **난이도**: ★★★
- **선수 지식**: X-1-1
- **학습 목표**: 이 Section을 마치면 백엔드 서비스와 프론트엔드를 연결해 HTTP Load Balancer를 구성할 수 있다.
- **설명**: V-2-1에서 국제공항 입국 심사대 비유로 Load Balancer를 배웠다. 수백 명의 승객이 내려도 심사관 여러 명이 동시에 처리하고, 피크 타임에는 심사관을 더 늘린다. 그때는 개념으로만 이해했다면, 이 Section에서는 그 심사대를 직접 설치한다. Cloud Run은 GCP가 Load Balancer를 자동으로 붙여주지만, Compute Engine 기반 서비스에서는 개발자가 직접 조립해야 한다.

  GCP의 HTTP(S) Load Balancer는 세 구성 요소로 이루어진다. **백엔드 서비스(Backend Service)**는 실제 트래픽을 받을 인스턴스 그룹과 Health Check를 연결하는 설정이다. Load Balancer는 Health Check를 통과한, 즉 정상 응답하는 인스턴스에만 요청을 전달한다. **URL Map**은 어떤 경로로 들어온 요청을 어떤 백엔드 서비스로 보낼지 결정하는 규칙 집합이다. V-2-1에서 언급한 `/api` 경로는 백엔드로, 나머지는 프론트엔드로 보내는 패턴이 바로 이 URL Map으로 구현된다. **프론트엔드(Forwarding Rule)**는 외부 IP와 포트를 수신하는 진입점이다. 사용자의 요청이 이 IP로 들어와 URL Map을 거쳐 백엔드 서비스에 도달한다.

  아래 명령은 백엔드 서비스를 만들고, 앞 Section에서 생성한 인스턴스 그룹을 연결하는 두 단계를 보여준다. `--global` 옵션은 이 백엔드 서비스를 글로벌 Load Balancer에서 사용한다는 의미다. 글로벌 Load Balancer는 애니캐스트 IP를 통해 전 세계 어디서 요청이 와도 가장 가까운 GCP 거점에서 처리한다. URL Map과 Forwarding Rule은 GCP Console에서 추가로 설정하거나 추가 `gcloud` 명령으로 구성할 수 있다.

  백엔드 서비스와 인스턴스 그룹이 연결되면 Load Balancer가 트래픽을 각 VM에 분산하기 시작한다. 다음 Section부터는 이 서비스를 숫자 IP가 아닌 사람이 읽을 수 있는 도메인 주소로 접근하는 방법을 다룬다.

  ```bash
  # 백엔드 서비스 생성
  gcloud compute backend-services create my-backend \
    --protocol=HTTP \
    --health-checks=my-health-check \
    --global

  # 백엔드 서비스에 인스턴스 그룹 연결
  gcloud compute backend-services add-backend my-backend \
    --instance-group=my-group \
    --instance-group-zone=asia-northeast3-a \
    --global
  ```

- **핵심 키워드**: 백엔드 서비스(Backend Service), URL Map, Forwarding Rule, 글로벌 Load Balancer
- **시각화 연결**: 해당 없음
- **다음 섹션**: X-2-1

### X-2-1. 도메인이란? — 도메인 구매와 등록

- **난이도**: ★★☆
- **선수 지식**: X-1-2
- **학습 목표**: 이 Section을 마치면 도메인의 역할을 이해하고, 무료 또는 유료 도메인을 구매해 GCP 서비스에 연결할 준비를 할 수 있다.
- **설명**: 친구에게 집 주소를 알려줄 때 "서울특별시 강남구 테헤란로 152번지 4층 1호"라고 말하는 것과 "강남 CGV 맞은편 건물"이라고 말하는 것 중 어느 쪽이 더 기억하기 쉬운가. Load Balancer를 만들고 나면 서비스에 접근할 수 있는 IP 주소가 생긴다. `34.120.xx.xx` 같은 숫자 조합이다. 이 주소는 컴퓨터가 목적지를 찾는 데는 완벽하지만, 사람이 기억하고 공유하기에는 불편하다. **도메인(Domain)**은 이 숫자 주소에 사람이 읽을 수 있는 이름을 붙이는 시스템이다. `api.example.com`처럼 의미 있는 주소를 IP에 연결하면, 사용자는 숫자를 외울 필요 없이 서비스에 접근할 수 있다.

  도메인을 사용하려면 두 단계가 필요하다. 첫째는 도메인 **구매**다. **등록 업체(Registrar)**는 도메인 이름의 소유권을 관리하는 기관이다. Google Domains, Namecheap, GoDaddy 같은 서비스에서 원하는 도메인 이름을 검색하고, 사용 가능하면 연 단위로 구매할 수 있다. `example.com` 같은 인기 있는 이름은 이미 누군가 보유하고 있을 수 있으므로, 비슷한 이름이나 다른 최상위 도메인(`.io`, `.dev` 등)을 검토한다. 둘째는 **DNS 매핑**이다. 구매한 도메인 이름과 실제 서버 IP를 연결하는 작업으로, 다음 Section의 주제다.

  **DNS(Domain Name System)**는 도메인 이름을 IP 주소로 변환하는 전화번호부 역할을 한다. 사용자가 브라우저에 `api.example.com`을 입력하면, DNS 서버가 이 이름에 연결된 IP를 조회해 브라우저에 알려준다. 브라우저는 그 IP로 실제 HTTP 요청을 보낸다. 이 조회 과정은 보통 수십 밀리초 안에 끝나므로 사용자는 과정 자체를 느끼지 못한다. **A 레코드**는 도메인 이름과 IPv4 주소를 직접 연결하는 DNS 레코드 타입이고, **CNAME**은 도메인 이름을 다른 도메인 이름으로 연결할 때 사용한다.

  도메인은 서비스의 첫인상이자 사용자가 매일 입력하는 주소다. 구매 단계에서는 이름 자체가 서비스를 잘 표현하는지, 기억하기 쉬운지를 우선 기준으로 삼는다. 다음 Section에서는 구매한 도메인을 Cloud DNS에 등록해 Load Balancer IP와 실제로 연결한다.
- **핵심 키워드**: 도메인(Domain), DNS(Domain Name System), 도메인 등록(Registrar), A 레코드, CNAME
- **시각화 연결**: 해당 없음
- **다음 섹션**: X-2-2

### X-2-2. Cloud DNS로 도메인과 IP 매핑하기

- **난이도**: ★★★
- **선수 지식**: X-2-1
- **학습 목표**: 이 Section을 마치면 Cloud DNS에 영역(Zone)을 만들고, A 레코드로 도메인과 Load Balancer IP를 매핑할 수 있다.
- **설명**: 도메인을 구매했다고 해서 곧바로 서비스 IP와 연결되지는 않는다. 새 전화번호를 개통해도 전화번호부에 등록하지 않으면 아무도 이름으로 찾을 수 없는 것과 같다. 도메인 이름과 IP를 실제로 연결하려면 **DNS 서버에 레코드를 등록**하는 작업이 필요하다. **Cloud DNS**는 GCP가 제공하는 관리형 DNS 서비스로, 이 등록 작업을 CLI 명령 몇 줄로 처리할 수 있다.

  Cloud DNS 설정은 두 단계로 이루어진다. 첫째는 **관리 영역(Managed Zone)** 생성이다. 관리 영역은 특정 도메인(`example.com`)에 관한 DNS 레코드를 묶어 관리하는 컨테이너다. 영역을 만들면 Cloud DNS가 네 개의 **네임서버(NS)** 주소를 자동으로 할당한다. 둘째는 이 네임서버 주소를 도메인 등록 업체 설정 페이지에 입력하는 것이다. 등록 업체의 기본 네임서버 대신 Cloud DNS의 네임서버를 사용하도록 변경하면, 이후부터 해당 도메인의 모든 DNS 조회는 Cloud DNS를 통해 처리된다. 변경 사항이 전 세계 DNS 서버에 전파되는 데 최대 48시간이 걸릴 수 있지만, 대부분은 몇 분 안에 반영된다.

  관리 영역이 준비되면 **A 레코드**를 추가한다. A 레코드는 `api.example.com`이라는 이름을 Load Balancer의 외부 IP(`34.xx.xx.xx`)에 직접 연결하는 설정이다. **TTL(Time To Live)**은 이 레코드를 DNS 캐시에 얼마나 보관할지를 초 단위로 지정한다. `300`으로 설정하면 5분마다 재조회한다. IP가 자주 바뀔 가능성이 있다면 TTL을 낮게, 안정적인 환경이라면 높게 설정한다. 레코드 추가 후 `nslookup api.example.com` 또는 `dig api.example.com` 명령으로 매핑이 정상 적용되었는지 확인할 수 있다.

  Cloud DNS로 도메인과 IP 매핑을 마치면, 사용자는 숫자 주소 대신 의미 있는 도메인 주소로 Load Balancer에 접근한다. 인스턴스 템플릿, MIG, Health Check, Load Balancer, 그리고 Cloud DNS까지 이어진 흐름이 Compute Engine 기반 서비스의 완전한 배포 구조를 형성한다.

  ```bash
  # DNS 영역 생성
  gcloud dns managed-zones create my-zone \
    --dns-name="example.com." \
    --description="My DNS zone"

  # A 레코드 추가 (도메인 → IP 매핑)
  gcloud dns record-sets create "api.example.com." \
    --zone=my-zone \
    --type=A \
    --ttl=300 \
    --rrdatas="34.xx.xx.xx"
  ```

- **핵심 키워드**: Cloud DNS, 관리 영역(Managed Zone), A 레코드, 네임서버(NS), TTL
- **시각화 연결**: 해당 없음
- **다음 섹션**: XI-1-1

> GCP 최대 차별화 서비스인 BigQuery로 대규모 데이터를 SQL로 분석하고, Looker Studio로 시각화한다.

### XI-1-1. BigQuery란? — 서버리스 데이터 웨어하우스

- **난이도**: ★★☆
- **선수 지식**: I-1-3
- **학습 목표**: 이 Section을 마치면 BigQuery의 아키텍처(컴퓨팅·스토리지 분리)를 이해하고, 전통적 DB와의 차이를 설명할 수 있다.
- **설명**: Part II에서 GCP를 "데이터·AI·쿠버네티스의 본고장"이라고 소개했다. 그 "데이터" 영역을 대표하는 서비스가 바로 **BigQuery**다. 원래 구글 광고 분석팀이 내부에서 수십억 건의 행동 데이터를 처리하려고 만든 시스템이 출발점으로, 이후 GCP의 대표 서비스로 공개되었다.

  BigQuery를 이해하려면 먼저 두 종류의 DB 작업을 구분해야 한다. Part IV에서 다룬 Cloud SQL은 **OLTP(OnLine Transaction Processing)** 에 특화된 DB다. OLTP는 "주문 1건 삽입", "회원 정보 1건 수정"처럼 소수의 행을 빠르게 읽고 쓰는 작업이다. 반면 BigQuery는 **OLAP(OnLine Analytical Processing)** 에 특화되어 있다. "지난 1년간 모든 주문 데이터에서 지역별 매출 합계를 구해라"처럼 수천만 행 전체를 한꺼번에 훑는 분석 작업이 OLAP의 영역이다. 마치 편의점 계산대(OLTP)와 창고 전체 재고 통계를 내는 ERP 시스템(OLAP)이 다른 용도인 것과 같다. 하나의 DB로 두 역할을 모두 잘하기는 어렵다.

  BigQuery가 대규모 분석에서 빠른 이유는 **컬럼형 스토리지(Columnar Storage)** 구조 때문이다. 일반 관계형 DB는 데이터를 행(row) 단위로 저장한다. 반면 BigQuery는 열(column) 단위로 저장하기 때문에, "매출 금액" 열 하나만 읽으면 될 때 다른 열의 데이터를 읽지 않아도 된다. 수십억 행 중 필요한 열만 골라 읽으니 처리량이 비약적으로 줄어든다. 또한 BigQuery는 **서버리스**다. 쿼리를 실행하면 GCP가 내부에서 수천 개의 노드에 작업을 분산해 처리하고, 쿼리가 끝나면 그 노드들은 해제된다. 개발자는 서버를 미리 띄워두거나 용량을 예측할 필요 없이, 쿼리를 실행한 만큼만 비용을 낸다.

  BigQuery는 "SQL 한 줄로 수십 TB를 분석한다"는 경험을 처음 해보는 서비스다. 다음 Section에서는 실제로 데이터셋을 만들고, CSV를 업로드해 SQL로 분석하는 실습을 진행한다.
- **핵심 키워드**: BigQuery, 데이터 웨어하우스(Data Warehouse), OLAP, 서버리스, 컬럼형 스토리지
- **시각화 연결**: 해당 없음
- **다음 섹션**: XI-1-2

### XI-1-2. BigQuery 사용하기 — 데이터 업로드와 SQL 쿼리

- **난이도**: ★★★
- **선수 지식**: XI-1-1
- **학습 목표**: 이 Section을 마치면 BigQuery에 데이터셋을 만들고, CSV를 업로드하고, SQL 쿼리로 분석할 수 있다.
- **설명**: BigQuery에서 데이터를 다루는 구조는 파일 시스템 비유로 쉽게 이해할 수 있다. **데이터셋(Dataset)**은 여러 테이블을 담는 폴더이고, **테이블(Table)**은 그 폴더 안의 파일이다. 프로젝트 하나 안에 여러 데이터셋을 만들 수 있고, 각 데이터셋 안에 목적별로 테이블을 나눠 보관한다. 예를 들어 `analytics` 데이터셋 안에 `user_events`, `orders`, `sessions` 테이블을 두는 방식이다.

  데이터셋과 테이블을 만드는 도구는 두 가지다. Cloud Console 웹 화면에서 클릭으로 진행하거나, `bq` CLI 명령을 사용한다. `bq mk --dataset` 명령으로 데이터셋을 만들고, `bq load` 명령으로 CSV 파일을 테이블로 업로드할 때는 컬럼 이름과 타입을 스키마로 함께 지정한다. 스키마(Schema)란 테이블의 열 구조를 정의한 설계도다. `name:STRING,age:INTEGER,city:STRING`처럼 열 이름과 타입을 쉼표로 나열하면 BigQuery가 그 형태로 테이블을 생성한다.

  데이터가 올라가면 표준 SQL로 쿼리를 실행한다. BigQuery의 테이블 이름은 백틱(`` ` ``)으로 감싼 `` `프로젝트.데이터셋.테이블` `` 형식으로 지정한다. Cloud SQL의 SQL 문법과 거의 동일하므로, 관계형 DB를 써본 경험이 있다면 바로 적응할 수 있다. 다만 BigQuery는 OLAP 특성상 `UPDATE`나 `DELETE`보다 `SELECT`와 집계 함수가 주된 사용 패턴이다. 쿼리를 실행하면 화면 우측 상단에 처리할 데이터 용량이 미리 표시되므로, 비용을 사전에 가늠하고 실행할 수 있다.

  아래 SQL은 이름별 건수를 집계해 상위 10개를 반환하는 예시이고, bash 블록은 데이터셋 생성과 CSV 업로드 명령이다.

  ```sql
  -- BigQuery SQL 쿼리 예시
  SELECT
    name,
    COUNT(*) AS count
  FROM `my-project.my_dataset.my_table`
  GROUP BY name
  ORDER BY count DESC
  LIMIT 10;
  ```

  ```bash
  # bq 명령으로 데이터셋 생성
  bq mk --dataset my-project:my_dataset

  # CSV 파일 업로드
  bq load --source_format=CSV \
    my_dataset.my_table ./data.csv \
    name:STRING,age:INTEGER,city:STRING
  ```

- **핵심 키워드**: 데이터셋(Dataset), `bq mk`, `bq load`, 표준 SQL, 스키마 정의
- **시각화 연결**: 해당 없음
- **다음 섹션**: XI-1-3

### XI-1-3. Looker Studio로 데이터 시각화하기

- **난이도**: ★★★
- **선수 지식**: XI-1-2
- **학습 목표**: 이 Section을 마치면 BigQuery 테이블을 Looker Studio에 연결해 차트와 대시보드를 만들 수 있다.
- **설명**: SQL 쿼리 결과는 숫자와 텍스트가 빽빽하게 채워진 테이블이다. 이 테이블을 팀 전체가 빠르게 파악하거나, 비개발 직군인 기획자·마케터와 공유하려면 시각화가 필요하다. 수치를 보는 것과 그래프로 보는 것은 정보를 인식하는 속도가 다르다. 마치 여행지 기온을 숫자 목록으로 볼 때와 꺾은선 그래프로 볼 때, 어느 쪽이 추세를 더 빨리 파악하는지를 생각해보면 된다.

  **Looker Studio**(구 Google Data Studio)는 BigQuery 테이블을 데이터 소스로 연결해 차트와 대시보드를 만드는 무료 시각화 도구다. 웹 브라우저에서 구글 계정으로 접속해 별도 설치 없이 바로 사용할 수 있다. BigQuery 데이터 소스를 연결하는 방법은 간단하다. Looker Studio 화면에서 "데이터 추가" → "BigQuery"를 선택하고, 프로젝트·데이터셋·테이블을 순서대로 고르면 연결이 완료된다. 코드 한 줄 없이 클릭만으로 끝난다.

  연결이 완료되면 테이블의 각 열이 "측정기준(Dimension)"과 "측정항목(Metric)"으로 분류된다. 문자열 열은 측정기준으로, 숫자 열은 측정항목으로 자동 분류된다. 막대 그래프, 원형 차트, 꺾은선 그래프, 지역 지도 등 다양한 차트 유형 중 하나를 선택하고 드래그 앤 드롭으로 열을 배치하면 그래프가 즉시 생성된다. 완성된 대시보드는 링크로 공유하거나 특정 구글 계정에만 공유 권한을 줄 수 있어, 팀 보고서를 별도로 만들 필요가 없다. BigQuery 테이블 데이터가 갱신되면 대시보드도 자동으로 최신 데이터를 반영한다.

  Looker Studio는 코드를 쓰지 않고도 데이터를 이야기로 만드는 도구다. BigQuery로 분석하고, Looker Studio로 공유하는 두 단계가 GCP 데이터 분석의 기본 흐름을 이룬다.
- **핵심 키워드**: Looker Studio, 대시보드, 데이터 소스 연결, 차트, 데이터 시각화
- **시각화 연결**: 해당 없음
- **다음 섹션**: XII-1-1

- **선수 지식**: IV-1-2
- **학습 목표**: 이 Section을 마치면 GKE의 Autopilot과 Standard 모드 차이를 설명하고, 프로젝트에 맞는 모드를 선택할 수 있다.
- **설명**: 항공 관제탑을 생각해보자. 소형 비행장에서 경비행기 한 대를 이·착륙시키는 것과, 국제공항에서 수십 대의 항공기를 동시에 유도하는 것은 완전히 다른 문제다. Cloud Run은 전자에 가깝다. 컨테이너 하나를 서버 관리 없이 빠르게 띄우는 데 최적화된 서비스다. 반면 **GKE(Google Kubernetes Engine)**는 후자를 담당한다. 수십에서 수백 개의 컨테이너를 조율하고, 서로 다른 역할의 서비스들이 정해진 규칙 위에서 협력하도록 관리하는 것이 GKE의 역할이다.

  IV-1-2에서 배운 컨테이너 이미지를 다시 떠올려보자. 앱 코드·Python 버전·라이브러리를 하나의 택배 상자에 담는다고 했다. Kubernetes에서는 이 상자들을 **Pod**이라는 단위로 묶어 다룬다. Pod 하나에 컨테이너 하나(또는 소수)가 들어가며, Kubernetes는 Pod들이 노드(물리적 또는 가상 서버) 위에서 안정적으로 실행되도록 스케줄링·재시작·스케일링을 자동으로 처리한다. GKE는 이 Kubernetes를 GCP 위에서 관리형으로 제공하는 서비스다. Kubernetes를 직접 설치하고 마스터 노드를 운영하는 수고 없이, GKE 위에서 곧바로 클러스터를 만들고 앱을 배포할 수 있다.

  GKE에는 두 가지 모드가 있다. **Autopilot** 모드는 노드의 생성·패치·스케일링을 GCP가 전부 대신한다. 인프라를 신경 쓰기보다 앱 배포에 집중하고 싶은 팀에게 적합하다. 입문자라면 Autopilot을 먼저 선택하는 것이 현명하다. **Standard** 모드는 노드 머신 타입, OS, 노드 수를 직접 결정한다. GPU 워크로드나 특수한 네트워크 설정처럼 세밀한 제어가 필요할 때 쓴다. "내가 서버 설정을 직접 다루겠다"는 확신이 없다면 처음에는 Autopilot로 시작한 뒤 필요할 때 전환을 검토하면 된다.

  GKE는 마이크로서비스 아키텍처, 즉 여러 개의 독립된 서비스가 함께 동작하는 구조에서 빛을 발한다. 단일 컨테이너 배포는 Cloud Run으로 충분하지만, 서비스가 분산·확장될수록 Kubernetes 오케스트레이션이 필요해진다. GKE는 그 쿠버네티스를 만든 구글이 직접 운영하는 서비스라는 점에서 II-1-2에서 GCP를 "쿠버네티스의 본고장"이라 표현한 것과 정확히 맞닿아 있다.
- **핵심 키워드**: GKE, Kubernetes, Autopilot, Standard, 노드(Node), Pod, 클러스터(Cluster)
- **시각화 연결**: 해당 없음
- **다음 섹션**: XII-1-2

### XII-1-2. GKE Autopilot 클러스터 생성하기

- **난이도**: ★★★
- **선수 지식**: XII-1-1
- **학습 목표**: 이 Section을 마치면 GKE Autopilot 클러스터를 생성하고, kubectl로 클러스터에 접근할 수 있다.
- **설명**: 새로운 지사 사무실을 개설한다고 상상해보자. 먼저 건물을 계약해야 한다. 그 다음에는 직원들이 해당 건물에 출입할 수 있도록 출입 카드를 발급받아야 한다. GKE 클러스터를 만드는 과정이 이와 비슷하다. 첫 번째 명령은 클러스터 자체를 생성하는 것이고, 두 번째 명령은 로컬 환경의 kubectl이 그 클러스터에 접근할 수 있도록 인증 정보를 연결하는 것이다.

  Autopilot 클러스터를 생성할 때 `create-auto` 명령을 사용하는 것에 주목하자. Standard 모드의 `create` 명령과 달리, `create-auto`는 GCP가 노드 프로비저닝·OS 패치·리소스 스케일링을 전부 자동으로 처리하겠다는 선언이다. 개발자가 해야 할 일은 클러스터 이름과 리전만 지정하는 것으로 끝난다. `--region=asia-northeast3`은 서울 리전을 뜻한다. 리전 선택은 III-1-3에서 배운 원칙과 동일하다. 사용자와 지리적으로 가까운 리전을 고르면 응답 지연을 줄일 수 있다.

  클러스터가 준비되면 두 번째 명령인 `get-credentials`를 실행한다. 이 명령은 클러스터 접속 정보를 로컬의 `kubeconfig` 파일에 기록한다. kubeconfig는 kubectl이 어떤 클러스터에 어떤 방법으로 연결해야 하는지를 저장하는 설정 파일이다. 이 과정이 완료되면 `kubectl` 명령으로 클러스터 내부의 Pod와 서비스를 직접 다룰 수 있게 된다. Autopilot 클러스터는 처음 생성될 때 약간의 시간이 걸릴 수 있는데, 이는 GCP가 백그라운드에서 마스터 노드와 초기 인프라를 구성하기 때문이다.

  클러스터 생성과 인증 설정은 GKE 실습의 출발점이다. 이 두 단계를 마치면 다음 Section에서 실제 앱을 배포할 준비가 된다.

  ```bash
  # GKE Autopilot 클러스터 생성
  gcloud container clusters create-auto my-cluster \
    --region=asia-northeast3

  # 클러스터 인증 정보 가져오기
  gcloud container clusters get-credentials my-cluster \
    --region=asia-northeast3
  ```

- **핵심 키워드**: `gcloud container clusters create-auto`, kubectl, kubeconfig, 인증 정보
- **시각화 연결**: 해당 없음
- **다음 섹션**: XII-1-3

### XII-1-3. GKE에 Nginx 웹서버 배포하기

- **난이도**: ★★★
- **선수 지식**: XII-1-2
- **학습 목표**: 이 Section을 마치면 Kubernetes Deployment와 Service를 작성해 GKE에 Nginx 웹서버를 배포하고 외부에서 접근할 수 있다.
- **설명**: 건물을 짓기 전에 설계 도면을 먼저 완성하는 것처럼, Kubernetes에서 앱을 배포할 때는 YAML 파일로 원하는 상태를 먼저 선언한다. "Nginx 컨테이너를 2개 실행하고, 외부에서 80번 포트로 접근할 수 있게 해달라"는 의도를 YAML에 적어두면 Kubernetes가 그 상태를 만들고 유지한다. 이를 **선언적 관리(Declarative Management)**라고 부른다. "이렇게 해라"고 명령하는 방식이 아니라 "이런 상태여야 한다"고 선언하면, Kubernetes가 현재 상태와 비교해 스스로 조치를 취하는 방식이다.

  YAML 파일의 구조를 살펴보자. `kind: Deployment`는 컨테이너 실행 방법을 정의하는 객체다. `replicas: 2`는 Nginx Pod을 2개 유지하라는 선언이다. 하나의 Pod에 장애가 생기면 Kubernetes가 자동으로 새 Pod을 생성해 항상 2개를 유지한다. IV-1-2에서 배운 컨테이너 이미지가 여기서 `image: nginx:latest` 한 줄로 참조된다. Pod이 어떤 컨테이너를 실행할지는 바로 이 이미지 경로로 결정된다. `containerPort: 80`은 컨테이너 내부에서 수신하는 포트를 Kubernetes에 알려주는 선언이다.

  YAML 파일을 작성한 뒤에는 세 가지 명령으로 배포와 외부 노출을 완성한다. `kubectl apply -f nginx-deployment.yaml`은 파일에 선언된 상태를 클러스터에 적용한다. 이미 실행 중인 Deployment가 있다면 변경 사항만 반영하고, 없다면 새로 생성한다. `kubectl expose deployment nginx --type=LoadBalancer --port=80`은 Deployment 앞에 LoadBalancer 타입 Service를 붙여 외부에서 접근 가능한 IP를 할당한다. Cloud Run에서는 배포하면 URL이 자동으로 만들어졌지만, GKE에서는 이 Service 설정을 직접 해줘야 한다. `kubectl get services`로 서비스 목록을 조회하면 EXTERNAL-IP 열에 공인 IP가 나타난다. 이 IP로 브라우저에서 접근하면 Nginx 기본 페이지를 확인할 수 있다.

  YAML 선언 한 파일로 복수의 Pod 실행·외부 노출·자동 복구를 모두 정의할 수 있다는 것이 Kubernetes 운영의 핵심 편의성이다. 설정 파일을 Git으로 관리하면 배포 이력을 추적하고 언제든 이전 상태로 되돌릴 수 있다.

  ```yaml
  # nginx-deployment.yaml
  apiVersion: apps/v1
  kind: Deployment
  metadata:
    name: nginx
  spec:
    replicas: 2
    selector:
      matchLabels:
        app: nginx
    template:
      metadata:
        labels:
          app: nginx
      spec:
        containers:
        - name: nginx
          image: nginx:latest
          ports:
          - containerPort: 80
  ```

  ```bash
  # 배포 및 서비스 노출
  kubectl apply -f nginx-deployment.yaml
  kubectl expose deployment nginx --type=LoadBalancer --port=80
  kubectl get services
  ```

- **핵심 키워드**: Deployment, Service, Pod, `kubectl apply`, LoadBalancer 타입, YAML 매니페스트
- **시각화 연결**: 해당 없음
- **다음 섹션**: XIII-1-1

### XIII-1-1. AgentBuilder란? — 대화형 AI 에이전트

- **난이도**: ★★☆
- **선수 지식**: 없음
- **학습 목표**: 이 Section을 마치면 AgentBuilder의 개념과 활용 사례를 이해하고, 어떤 유형의 에이전트를 만들 수 있는지 설명할 수 있다.
- **설명**: 챗봇을 만들려면 예전에는 자연어 처리 모델을 직접 학습시키거나, 외부 AI API를 연동하는 코드를 수백 줄 작성해야 했다. 서버를 띄우고, 메시지를 파싱하고, 의도를 분류하고, 답변을 생성하는 파이프라인을 하나하나 구현하는 일이었다. **AgentBuilder**는 이 과정 전체를 Cloud Console의 UI 화면에서 처리할 수 있게 만든 노코드/로우코드 AI 에이전트 빌더다. 코드를 작성하지 않고도 대화형 AI 에이전트를 구성하고 테스트할 수 있다.

  AgentBuilder가 동작하는 핵심은 **LLM(Large Language Model, 대규모 언어 모델)**이다. LLM은 방대한 텍스트 데이터를 학습해 자연어 질문을 이해하고 문맥에 맞는 답변을 생성하는 모델이다. AgentBuilder는 Google이 직접 개발한 LLM을 기반으로 작동하며, 이 LLM 인프라를 통합 관리하는 플랫폼이 **Vertex AI**다. 즉, AgentBuilder는 Vertex AI 위에서 동작하는 에이전트 구성 도구라고 볼 수 있다. II-1-2에서 GCP를 "데이터·AI·쿠버네티스의 본고장"이라고 소개했는데, AgentBuilder는 그 "AI" 영역의 최신 서비스로 자리하고 있다.

  AgentBuilder로 만들 수 있는 에이전트의 유형은 크게 세 가지다. 인터넷을 실시간으로 검색해 최신 정보를 바탕으로 답변하는 **웹검색 기반 질의응답**, 사내 문서나 특정 데이터베이스를 참조해 내부 정보를 안내하는 **사내 문서 검색**, 그리고 정해진 시나리오에 따라 사용자를 안내하는 **고객 상담 자동화**가 대표적이다. 이 모든 구성을 Cloud Console에서 클릭과 입력만으로 완료할 수 있다. CLI 명령이나 API 코드 없이, 브라우저 화면 안에서 에이전트의 이름·동작 방식·데이터 소스를 설정한다.

  AgentBuilder는 GCP가 제공하는 AI 기술을 가장 낮은 진입 장벽으로 활용할 수 있는 도구다. 다음 Section에서는 실제로 웹검색 기반 에이전트를 Cloud Console에서 단계별로 만들어본다.
- **핵심 키워드**: AgentBuilder, 대화형 에이전트, 노코드, LLM, Vertex AI
- **시각화 연결**: 해당 없음
- **다음 섹션**: XIII-1-2

### XIII-1-2. 웹검색 기반 대화형 에이전트 구현하기

- **난이도**: ★★★
- **선수 지식**: XIII-1-1
- **학습 목표**: 이 Section을 마치면 AgentBuilder로 웹검색 기반 대화형 에이전트를 만들고, 테스트 대화를 실행할 수 있다.
- **설명**: 스마트폰 음성 비서를 떠올려보자. "오늘 서울 날씨 어때?"라고 물으면 비서는 인터넷을 검색해 실시간 날씨를 가져와 자연스러운 문장으로 답한다. 미리 정해진 답을 읽어주는 것이 아니라, 질문 시점에 실제 웹을 탐색해 정보를 만들어낸다. AgentBuilder의 웹검색 기반 에이전트가 바로 이 방식으로 동작한다.

  에이전트를 만드는 전 과정은 Cloud Console에서 진행된다. 먼저 에이전트 유형을 선택한다. 검색형은 웹사이트나 데이터 소스에서 정보를 찾아 요약하는 방식이고, 대화형은 여러 번의 주고받기를 통해 사용자를 안내하는 방식이다. 유형을 선택한 뒤에는 데이터 소스를 연결한다. 웹사이트 URL을 지정하면 에이전트가 해당 사이트를 참조 범위로 삼는다. Cloud Storage에 업로드한 PDF나 BigQuery 테이블을 소스로 연결하는 것도 가능하다. 이 연결 과정을 **Grounding(그라운딩)**이라고 부른다. LLM이 자유롭게 생성하는 답변을 지정한 소스 데이터로 "닻을 내려" 고정시킨다는 의미다. Grounding 덕분에 에이전트가 근거 없는 내용을 지어내는 현상을 줄이고, 실제 데이터에 기반한 답변을 생성할 수 있다.

  에이전트 구성을 마치면 Console 화면에서 바로 테스트 대화를 실행할 수 있다. 질문을 입력하면 에이전트가 웹을 검색하고, 검색 결과를 요약해 답변을 돌려준다. 테스트를 통과한 에이전트는 배포 단계에서 **API 엔드포인트**나 **웹 위젯** 형태로 기존 앱에 통합할 수 있다. 웹 위젯은 HTML 한 줄을 복사해 붙여넣는 것만으로 웹 페이지에 챗봇 UI를 삽입한다.

  여기까지 오는 동안 많은 것을 거쳤다. GCP 계정과 프로젝트를 만들고(Part III), Cloud Run으로 앱을 배포하고(Part IV), IAM·VPC로 보안과 네트워크를 구성하고(Part V), CI/CD로 배포를 자동화했다(Part VI). 로그와 모니터링으로 운영을 익히고(Part VII), Compute Engine으로 VM 기반 인프라를 다루고(Part VIII), 고급 IAM·VPC 실습을 거쳐(Part IX), 로드 밸런서와 도메인으로 인터넷에 서비스를 노출했다(Part X). BigQuery로 데이터를 분석하고(Part XI), GKE로 컨테이너를 오케스트레이션했으며(Part XII), 이제 AI 에이전트를 코드 한 줄 없이 만들어보는 것으로 마무리한다(Part XIII). GCP 위에서 배포·운영·분석·AI를 모두 경험한 셈이다. 이 여정이 다음 단계로 나아가는 출발점이 되길 바란다.
- **핵심 키워드**: 에이전트 유형, 데이터 소스 연결, Grounding, 웹 위젯, API 통합
- **시각화 연결**: 해당 없음

## 부록 B. 자주 묻는 질문 (FAQ)

**Q1. Cloud Run과 GKE 중 무엇을 선택해야 하나요?**
Cloud Run은 단순한 웹 API나 마이크로서비스에 적합합니다. 서버 관리가 없고 자동 스케일이 편리합니다. GKE는 복잡한 마이크로서비스 오케스트레이션, 상태 유지 워크로드(StatefulSet), 커스텀 네트워킹이 필요할 때 선택합니다. 처음에는 Cloud Run으로 시작하고, 복잡도가 높아지면 GKE를 고려하는 것이 현실적입니다.

**Q2. Cloud Run이 Stateless라면 파일 업로드는 어떻게 하나요?**
Cloud Run 인스턴스의 로컬 파일시스템은 임시 저장소(`/tmp`)로, 인스턴스가 종료되면 사라집니다. 파일은 반드시 **Cloud Storage**에 업로드하고 URL 또는 경로를 DB에 저장하는 방식을 사용해야 합니다.

**Q3. `--allow-unauthenticated` 옵션을 쓰면 보안이 취약한가요?**
공개 API는 당연히 인증 없이 접근할 수 있어야 합니다. 이 옵션은 단지 GCP 레벨의 인증(IAM)을 건너뛴다는 의미입니다. 앱 레벨에서 JWT·API Key 등으로 별도 인증을 구현하면 됩니다. 내부 서비스(다른 Cloud Run이 호출하는 서비스)는 `--no-allow-unauthenticated`를 써서 IAM 인증을 강제하는 것이 좋습니다.

**Q4. 콜드 스타트 지연을 없애려면?**
`--min-instances 1` 옵션으로 최소 인스턴스를 1로 설정하면 항상 인스턴스가 켜진 상태를 유지해 콜드 스타트가 없습니다. 단, 요청이 없어도 1개 인스턴스 비용이 발생합니다. 서비스 특성(비용 vs 응답 속도)에 따라 선택합니다.

**Q5. Cloud SQL 비용이 부담스러운데 대안이 있나요?**
Cloud SQL은 인스턴스가 항상 켜져 있어 Scale to Zero가 안 됩니다. 비용이 부담이라면 **Firestore**(NoSQL, Scale to Zero) 또는 **Neon**(서버리스 PostgreSQL, 써드파티)을 검토해 볼 수 있습니다. [확인 필요: 프로덕션 워크로드에서 Neon과 Cloud SQL의 성능 비교]

**Q6. gcloud 명령이 너무 많아서 외우기 힘든데요.**
자주 쓰는 명령은 `gcloud cheat-sheet` 명령으로 빠르게 조회할 수 있습니다. `gcloud help`와 `gcloud [command] --help`도 필수입니다. 외우는 것보다 "패턴"을 이해하는 것이 중요합니다: `gcloud [제품] [동사] [리소스명] [옵션]`.

**Q7. Cloud Build 대신 GitHub Actions를 써도 되나요?**
됩니다. GitHub Actions에서 `google-github-actions/deploy-cloudrun` 공식 Action을 사용하면 Cloud Build 없이 GitHub에서 직접 Cloud Run 배포를 자동화할 수 있습니다. 이미 GitHub Actions를 쓰고 있다면 이쪽이 더 간단할 수 있습니다.

**Q8. 여러 리전에 동시에 배포하려면?**
Cloud Run은 리전별로 독립적으로 배포됩니다. 여러 리전에 동시 배포하려면 각 리전에 `gcloud run deploy`를 실행하고, 앞에 **글로벌 로드 밸런서**를 붙여 사용자를 가장 가까운 리전으로 라우팅합니다.

**Q9. Cloud Logging 로그 보존 기간은 얼마나 되나요?**
기본 보존 기간은 30일입니다. 더 오래 보존하려면 **Cloud Storage**로 로그를 내보내거나, **BigQuery**로 로그를 내보내 장기 분석에 활용할 수 있습니다.

**Q10. Private IP 연결을 위해 VPC Connector가 반드시 필요한가요?**
Cloud Run에서 Cloud SQL Private IP에 접근하려면 **Serverless VPC Access Connector** 또는 **Direct VPC Egress**(최신 방식)를 사용해야 합니다. Direct VPC Egress는 Cloud Run이 VPC에 직접 접속하는 방식으로, VPC Connector보다 지연이 낮고 비용도 저렴합니다. 2024년 이후 신규 프로젝트에서는 Direct VPC Egress 사용을 권장합니다.

## 부록 C. 다음에 공부할 것

이 커리큘럼을 마친 뒤 관심사별로 다음 방향을 추천한다.

### 백엔드 엔지니어링 방향

1. **Terraform** — Infrastructure as Code로 GCP 인프라를 코드로 관리한다. `gcloud` 명령 대신 `.tf` 파일로 인프라를 선언하고 버전 관리한다.
2. **GKE(Google Kubernetes Engine)** — Cloud Run을 넘어 더 복잡한 컨테이너 오케스트레이션이 필요할 때. Kubernetes 기본 개념부터 시작한다.
3. **Cloud Functions** — 이벤트 기반 서버리스 함수. Cloud Run보다 더 가벼운 작업(Cloud Storage 파일 업로드 트리거, Pub/Sub 메시지 처리 등)에 적합하다.

### 데이터 엔지니어링 방향

1. **BigQuery** — SQL로 대용량 데이터를 분석한다. `bq` CLI와 BigQuery ML을 포함한다.
2. **Dataflow** — Apache Beam 기반 실시간·배치 데이터 파이프라인.
3. **Pub/Sub** — 메시지 큐 서비스. 서비스 간 비동기 통신과 이벤트 드리븐 아키텍처에 사용한다.

### AI/ML 방향

1. **Vertex AI** — 모델 학습·배포·모니터링 통합 플랫폼.
2. **Gemini API on Vertex AI** — 구글 LLM(대형 언어 모델)을 API로 사용한다.

## 부록 E. 리소스 명명규칙

클라우드 리소스 이름은 팀 전체가 일관되게 사용해야 혼란을 줄일 수 있다. 아래는 실무에서 자주 쓰이는 명명 패턴이다.

| Cloud SQL | `{서비스}-db-{환경}` | `api-db-prod` |

## 부록 F. 아키텍처 문서화 — draw.io 활용법

클라우드 프로젝트에서 아키텍처 다이어그램은 팀원 간 소통의 핵심 도구다. 코드만으로는 전체 시스템의 연결 구조를 한눈에 파악하기 어렵다.

### 왜 아키텍처 문서화가 중요한가

- 새 팀원이 시스템 전체를 빠르게 이해할 수 있다
- 장애 발생 시 어떤 컴포넌트가 영향을 받는지 즉시 파악할 수 있다
- 확장·변경 계획을 시각적으로 논의할 수 있다

### draw.io(diagrams.net)로 GCP 아키텍처 그리기

1. **draw.io**(`app.diagrams.net`)는 무료 다이어그램 도구다
2. GCP 아이콘 라이브러리가 내장되어 있다: 메뉴 → 검색 → "GCP" 입력
3. 각 서비스(Cloud Run, Cloud SQL, VPC 등)를 아이콘으로 배치하고 화살표로 연결한다
4. 서비스마다 작업이 추가될 때마다 다이어그램도 함께 업데이트하는 습관이 중요하다

### 다이어그램 포함 권장 요소

- 서비스 간 통신 방향 (화살표)
- 네트워크 경계 (VPC, Subnet 구분)
- 외부 사용자 ↔ Load Balancer ↔ 서비스 흐름
- IAM 권한 흐름 (어떤 SA가 어떤 리소스에 접근하는지)
