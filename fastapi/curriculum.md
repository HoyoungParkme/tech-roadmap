## 이 책이 만들어진 이유

이 자료는 FastAPI 공식 문서나 두꺼운 기술서와 다르게 **"아는 만큼 읽고, 필요한 만큼만 깊이 들어가도 되는"** 사다리형 구조로 짜여 있다. 처음 프로그래밍을 시작하는 비개발자가 Part I부터 읽어도 막히지 않게 만들었고, 동시에 이미 실무에서 FastAPI를 다루는 주니어가 Part IV 이후만 발췌해 참고해도 가치가 있게 썼다.

전체 구성은 **설계도(상세 목차) + 인터랙티브 시각화**다. 각 Section은 한 꼭지의 학습 단위이며, 필요한 곳에는 `index.html`과 같은 **단계별 애니메이션 시각화**가 짝을 이뤄, 글 대신 움직이는 그림으로 개념을 체득할 수 있도록 설계되어 있다.

## Part I. 왜 웹인가 (★☆☆)

"웹 개발"이라는 말을 들으면 어렵고 낯선 세계처럼 느껴질 수 있다. 하지만 웹은 결국 컴퓨터끼리 편지를 주고받는 약속에서 시작한다. Part I은 코딩 경험이 전혀 없어도 읽을 수 있도록 설계됐다. 인터넷이 어떻게 연결되는지, HTTP라는 대화 방식이 무엇인지, 서버와 클라이언트가 어떻게 역할을 나누는지, API와 JSON이 무엇인지, 그리고 프론트엔드와 백엔드가 왜 존재하는지를 순서대로 익힌다.

이 Part를 마치면 "웹이 어떻게 돌아가는가"를 한 문장으로 설명할 수 있게 된다. FastAPI를 배우기 전에 이 배경 지식을 갖추면 이후 내용이 훨씬 자연스럽게 연결된다.

### Chapter I-1. 컴퓨터끼리 어떻게 이야기하나

#### I-1-1. 인터넷이 어떻게 연결돼 있나

- **난이도**: ★☆☆
- **선수 지식**: 없음
- **학습 목표**: 이 섹션을 마치면 인터넷이 컴퓨터들을 이어주는 거대한 연결망임을 설명할 수 있다.
- **설명**: 인터넷(Internet)은 전 세계 수십억 대의 컴퓨터가 케이블, 광섬유, 위성 등으로 연결된 거대한 그물망이다. 우리 집 와이파이 공유기는 그 그물망의 한 매듭(노드)이다. 택배 네트워크와 비슷하다. 편지를 보내면 여러 물류 센터를 거쳐 목적지에 도착하듯, 인터넷의 데이터도 여러 컴퓨터(라우터)를 거쳐 상대방에게 전달된다. 중간 경로가 끊겨도 다른 길을 찾아서 간다는 것이 인터넷의 핵심 설계 원칙이다.
- **핵심 키워드**: [인터넷, 네트워크, 라우터, 데이터 패킷]
- **시각화 연결**: [I-1-1 시각화 보기](part1/I-1-1.html) — 컴퓨터 노드 → 라우터 경유 → 메시 네트워크 (5단계)
- **다음 섹션**: I-1-2

#### I-1-2. IP, 도메인, DNS — 주소의 종류

- **난이도**: ★☆☆
- **선수 지식**: I-1-1
- **학습 목표**: 이 섹션을 마치면 IP 주소와 도메인 이름의 차이를 설명하고, DNS가 둘을 어떻게 연결하는지 말할 수 있다.
- **설명**: 인터넷의 모든 컴퓨터는 IP 주소(Internet Protocol Address)라는 숫자 주소를 가진다. 예를 들어 `192.168.0.1` 같은 형태다. 하지만 사람이 숫자를 외우기는 어렵다. 그래서 `www.google.com`처럼 이름(도메인, Domain)을 붙여 사용한다. DNS(Domain Name System)는 도메인 이름을 IP 주소로 번역해주는 전화번호부 역할을 한다. 브라우저에서 주소를 입력하면 먼저 DNS 서버에 물어봐서 실제 IP 주소를 얻고, 그 주소로 연결하는 순서로 진행된다.
- **핵심 키워드**: [IP 주소, 도메인, DNS, 네임 서버]
- **시각화 연결**: [I-1-2 시각화 보기](part1/I-1-2.html) — 브라우저 → DNS 질의 → IP 반환 → 서버 연결 (5단계)
- **다음 섹션**: I-1-3

#### I-1-3. 포트 — 한 컴퓨터 안의 여러 창구

- **난이도**: ★☆☆
- **선수 지식**: I-1-2
- **학습 목표**: 이 섹션을 마치면 포트가 무엇인지, 왜 같은 컴퓨터에서 여러 서비스를 동시에 실행할 수 있는지 설명할 수 있다.
- **설명**: 하나의 컴퓨터에는 IP 주소가 하나지만, 동시에 여러 서비스를 실행할 수 있다. 예를 들어 같은 컴퓨터에서 웹사이트도 제공하고, 이메일도 처리하고, 게임 서버도 돌릴 수 있다. 이것이 가능한 이유가 포트(Port)다. 포트는 컴퓨터 안의 창구 번호다. 우체국 한 건물에 여러 창구(1번: 일반 우편, 2번: 택배, 3번: 국제 우편)가 있는 것과 같다. 웹은 주로 포트 80(HTTP)과 443(HTTPS)를 쓰고, FastAPI 개발 시에는 보통 8000번 포트를 사용한다.
- **핵심 키워드**: [포트, HTTP(80), HTTPS(443), localhost]
- **시각화 연결**: [I-1-3 시각화 보기](part1/I-1-3.html) — 서버 포트 구조, 동시 서비스 실행 (5단계)
- **다음 섹션**: I-2-1

### Chapter I-2. HTTP, 아주 느린 대화로 읽어보기

#### I-2-1. 요청과 응답 — 편지처럼 주고받는다

- **난이도**: ★☆☆
- **선수 지식**: I-1-3
- **학습 목표**: 이 섹션을 마치면 HTTP가 요청(Request)과 응답(Response)의 쌍으로 이루어진다는 것을 설명할 수 있다.
- **설명**: HTTP(HyperText Transfer Protocol)는 웹에서 데이터를 주고받는 약속이다. 편지를 주고받는 방식과 똑같다. 클라이언트(손님)가 먼저 "이 정보 좀 줄 수 있어요?"라고 요청(Request)을 보내면, 서버(직원)가 "네, 여기 있습니다" 또는 "그런 건 없습니다"라고 응답(Response)을 돌려준다. 한 번의 요청에 한 번의 응답이 기본 원칙이다. 웹 브라우저에서 주소를 입력할 때마다 이 교환이 일어난다.
- **핵심 키워드**: [HTTP, 요청(Request), 응답(Response), 프로토콜]
- **시각화 연결**: [I-2-1 시각화 보기](part1/I-2-1.html) — 요청-응답 편지 교환 (5단계)
- **다음 섹션**: I-2-2

#### I-2-2. 메서드 — GET / POST / PUT / DELETE

- **난이도**: ★☆☆
- **선수 지식**: I-2-1
- **학습 목표**: 이 섹션을 마치면 HTTP 메서드 4가지의 역할을 구분해서 말할 수 있다.
- **설명**: HTTP 요청에는 "어떤 행동을 할 것인지"를 나타내는 메서드(Method)가 붙는다. 식당으로 비유하면 이렇다. GET은 메뉴를 달라고 "보여주세요"다. POST는 새 주문을 넣는 "추가해주세요"다. PUT은 이미 넣은 주문을 통째로 바꾸는 "교체해주세요"다. DELETE는 주문을 취소하는 "삭제해주세요"다. 웹 API는 이 네 가지 메서드를 조합해서 데이터를 생성(Create), 읽기(Read), 수정(Update), 삭제(Delete)하는 CRUD 작업을 표현한다.
- **핵심 키워드**: [GET, POST, PUT, DELETE, CRUD, HTTP 메서드]
- **시각화 연결**: [I-2-2 시각화 보기](part1/I-2-2.html) — 4가지 메서드 카드 + CRUD 매핑 (6단계)
- **다음 섹션**: I-2-3

#### I-2-3. 상태 코드 — 200 / 404 / 500

- **난이도**: ★☆☆
- **선수 지식**: I-2-2
- **학습 목표**: 이 섹션을 마치면 HTTP 상태 코드의 의미를 대역별로 분류하고, 주요 코드 3개를 설명할 수 있다.
- **설명**: 서버가 응답할 때는 숫자 코드를 함께 돌려준다. 이것이 상태 코드(Status Code)다. 200은 "성공"이다. 404는 "찾을 수 없다"다. 500은 "서버가 고장났다"다. 코드의 첫 번째 자리가 대역을 나타낸다. 2xx는 성공, 3xx는 이동(Redirect), 4xx는 클라이언트 잘못, 5xx는 서버 잘못이다. 브라우저에서 "404 Not Found" 페이지를 본 경험이 있다면 이미 상태 코드를 만난 것이다.
- **핵심 키워드**: [상태 코드(Status Code), 200 OK, 404 Not Found, 500 Internal Server Error]
- **시각화 연결**: [I-2-3 시각화 보기](part1/I-2-3.html) — 4개 대역 2xx/3xx/4xx/5xx 분류 (5단계)
- **다음 섹션**: I-2-4

#### I-2-4. 헤더와 바디 — 봉투와 편지지

- **난이도**: ★☆☆
- **선수 지식**: I-2-3
- **학습 목표**: 이 섹션을 마치면 HTTP 메시지가 헤더와 바디로 구성됨을 설명하고, 각각의 역할을 말할 수 있다.
- **설명**: HTTP 요청과 응답은 봉투처럼 생겼다. 헤더(Header)는 봉투 겉면에 적힌 발신인, 수신인, 우편번호 같은 정보다. 브라우저 종류, 데이터 형식, 인증 토큰 같은 부가 정보가 여기 들어간다. 바디(Body)는 봉투 안의 편지지다. 실제로 전달하려는 내용(JSON 데이터, 이미지, HTML 등)이 담긴다. GET 요청은 보통 바디가 없고, POST나 PUT은 바디에 새 데이터를 담아 보낸다.
- **핵심 키워드**: [헤더(Header), 바디(Body), Content-Type, Authorization]
- **시각화 연결**: [I-2-4 시각화 보기](part1/I-2-4.html) — 봉투 구조, 헤더/바디 하이라이트 (5단계)
- **다음 섹션**: I-3-1

### Chapter I-3. 서버와 클라이언트 — 주방과 손님

#### I-3-1. 서버라는 직원과 클라이언트라는 손님

- **난이도**: ★☆☆
- **선수 지식**: I-2-4
- **학습 목표**: 이 섹션을 마치면 서버와 클라이언트의 역할 차이를 일상 비유로 설명할 수 있다.
- **설명**: 식당으로 생각하면 이해하기 쉽다. 클라이언트(Client)는 식당에 온 손님이다. 웹 브라우저, 스마트폰 앱, 또는 다른 프로그램이 모두 클라이언트가 될 수 있다. 서버(Server)는 주방 직원이다. 손님의 주문을 받아서 음식을 만들고 내어준다. 항상 대기 중이며, 손님이 요청해야만 반응한다. 서버가 먼저 손님에게 말을 거는 일은 기본 HTTP에서는 일어나지 않는다. 요청이 먼저, 응답이 나중이다.
- **핵심 키워드**: [서버(Server), 클라이언트(Client), 요청-응답 모델]
- **시각화 연결**: 해당 없음

#### I-3-2. URL 한 줄에 담긴 정보 읽기

- **난이도**: ★☆☆
- **선수 지식**: I-3-1
- **학습 목표**: 이 섹션을 마치면 URL을 프로토콜, 도메인, 경로, 쿼리 파라미터로 분해해서 읽을 수 있다.
- **설명**: URL(Uniform Resource Locator)은 인터넷 주소 전체를 말한다. 예를 들어 `https://shop.example.com:443/products?category=shoes&page=2`를 분해하면 이렇다. `https`는 프로토콜(보안 연결), `shop.example.com`은 도메인(어느 서버로), `:443`은 포트, `/products`는 경로(어떤 자원을), `?category=shoes&page=2`는 쿼리 파라미터(추가 조건)다. FastAPI에서 경로와 쿼리 파라미터를 자주 다루게 되므로 이 구조를 익혀두면 편하다.
- **핵심 키워드**: [URL, 프로토콜, 경로(Path), 쿼리 파라미터(Query Parameter)]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-3-3

---

#### I-3-3. 한 서버가 여러 손님을 받는 방법 (맛보기)

- **난이도**: ★☆☆
- **선수 지식**: I-3-2
- **학습 목표**: 이 섹션을 마치면 서버가 동시에 여러 요청을 처리할 수 있다는 개념을 설명할 수 있다.
- **설명**: 바쁜 식당에서 직원 한 명이 여러 테이블을 동시에 담당하는 것처럼, 서버도 동시에 여러 손님의 요청을 처리할 수 있어야 한다. 전통적인 방식은 손님마다 직원 한 명(스레드, Thread)을 붙이는 것이다. 하지만 손님이 너무 많아지면 직원이 부족해진다. 더 영리한 방식은 직원 한 명이 주문을 받아놓고 음식이 나올 때까지 기다리는 대신, 그 시간에 다른 테이블을 돌보는 것이다. 이 방식이 나중에 배울 비동기(Asynchronous)의 핵심 아이디어다. 지금은 "서버가 동시에 여러 요청을 다룬다"는 것만 기억해두면 된다.
- **핵심 키워드**: [동시 요청, 스레드(Thread), 동시성(Concurrency)]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-4-1

### Chapter I-4. API란 무엇인가

#### I-4-1. API = 식당의 메뉴판

- **난이도**: ★☆☆
- **선수 지식**: I-3-3
- **학습 목표**: 이 섹션을 마치면 API가 무엇인지 비유를 들어 설명할 수 있다.
- **설명**: API(Application Programming Interface)를 한 마디로 설명하면 "두 프로그램이 대화하는 방법을 정해놓은 약속"이다. 식당 메뉴판이 좋은 비유다. 메뉴판에는 주문할 수 있는 것(어떤 요청이 가능한지), 가격(필요한 정보), 나오는 음식(응답으로 받을 것)이 나와 있다. 손님은 주방 안을 몰라도 메뉴판만 보고 주문할 수 있다. 마찬가지로 API를 쓰는 개발자는 서버 내부 코드를 몰라도 정해진 형식대로 요청을 보내면 원하는 데이터를 받을 수 있다.
- **핵심 키워드**: [API(Application Programming Interface), 엔드포인트(Endpoint), 인터페이스]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-4-2

---

#### I-4-2. REST API가 약속하는 것들

- **난이도**: ★☆☆
- **선수 지식**: I-4-1
- **학습 목표**: 이 섹션을 마치면 REST API의 주요 원칙을 3가지 이상 말할 수 있다.
- **설명**: REST(Representational State Transfer)는 API를 만드는 방식 중 가장 널리 쓰이는 약속이다. REST는 몇 가지 원칙을 따른다. 첫째, URL은 자원(Resource)을 나타낸다. `/users/5`는 "5번 사용자"를 가리킨다. 둘째, 행동은 HTTP 메서드로 표현한다. GET은 조회, POST는 생성, PUT은 수정, DELETE는 삭제다. 셋째, 서버는 요청과 요청 사이에 클라이언트 상태를 기억하지 않는다(무상태, Stateless). 이 규칙을 지키면 누가 만들어도 예측 가능한 API가 된다.
- **핵심 키워드**: [REST, 자원(Resource), 무상태(Stateless), RESTful API]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-4-3

#### I-4-3. JSON — 컴퓨터가 알아듣는 메모 형식

- **난이도**: ★☆☆
- **선수 지식**: I-4-2
- **학습 목표**: 이 섹션을 마치면 JSON이 무엇인지 설명하고, 간단한 JSON 구조를 읽을 수 있다.
- **설명**: JSON(JavaScript Object Notation)은 데이터를 표현하는 텍스트 형식이다. 컴퓨터끼리 메모를 주고받을 때 주로 쓰는 형식이라고 생각하면 된다. 키(Key)와 값(Value)을 중괄호로 묶고 콜론으로 연결한다. 예를 들어 `{"name": "홍길동", "age": 30, "is_active": true}` 처럼 생겼다. 목록은 대괄호로 표현한다. 사람도 읽을 수 있고, 프로그램도 읽을 수 있어서 API 응답 형식으로 전 세계 표준처럼 사용된다.
- **핵심 키워드**: [JSON, 키-값(Key-Value), 직렬화(Serialization)]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-5-1

### Chapter I-5. 프론트엔드와 백엔드 — 같은 건물의 다른 방

#### I-5-1. 프론트엔드 — 손님이 보는 메뉴판과 테이블

- **난이도**: ★☆☆
- **선수 지식**: I-4-3
- **학습 목표**: 이 섹션을 마치면 프론트엔드가 무엇을 담당하는지 설명하고, 대표적인 기술을 2가지 이상 말할 수 있다.
- **설명**: 식당에서 손님이 직접 보고 경험하는 공간(메뉴판, 테이블, 인테리어)이 프론트엔드(Frontend)다. 웹에서는 브라우저에 표시되는 화면 전체를 말한다. 버튼, 입력 창, 색상, 레이아웃이 모두 프론트엔드의 영역이다. HTML은 메뉴판의 글자처럼 내용의 뼈대, CSS는 인테리어처럼 보이는 방식, JavaScript는 직원이 버튼을 누르면 반응하는 동작을 담당한다. React, Vue, Angular 같은 도구는 JavaScript로 복잡한 화면을 쉽게 만들도록 도와준다.
- **핵심 키워드**: [프론트엔드(Frontend), HTML, CSS, JavaScript, React]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-5-2

#### I-5-2. 백엔드 — 주방과 창고

- **난이도**: ★☆☆
- **선수 지식**: I-5-1
- **학습 목표**: 이 섹션을 마치면 백엔드가 무엇을 담당하는지 설명하고, FastAPI가 어느 위치에 있는지 말할 수 있다.
- **설명**: 식당 주방과 창고가 백엔드(Backend)다. 손님 눈에는 보이지 않지만, 실제 요리를 하고 재료를 보관하며 계산을 처리하는 곳이다. 웹에서 백엔드는 데이터를 저장하고, 로직을 계산하고, 프론트엔드의 요청에 응답한다. 데이터베이스(Database)는 창고다. FastAPI는 이 백엔드 역할을 하는 Python 웹 프레임워크다. 누가 로그인했는지, 어떤 상품이 남았는지, 결제가 성공했는지를 판단하는 일이 모두 백엔드의 몫이다.
- **핵심 키워드**: [백엔드(Backend), 데이터베이스(Database), 서버 로직, Python]
- **시각화 연결**: 해당 없음
- **다음 섹션**: I-5-3

#### I-5-3. "풀스택"이란 무엇인가

- **난이도**: ★☆☆
- **선수 지식**: I-5-2
- **학습 목표**: 이 섹션을 마치면 풀스택의 의미를 설명하고, 프론트엔드·백엔드·데이터베이스가 어떻게 연결되는지 설명할 수 있다.
- **설명**: 풀스택(Full Stack)은 식당 운영 전체를 혼자 담당하는 것과 같다. 주방(백엔드), 홀(프론트엔드), 창고(데이터베이스)를 모두 다룰 수 있는 개발자를 풀스택 개발자라고 부른다. 일반적으로 프론트엔드는 브라우저에서, 백엔드는 서버에서, 데이터베이스는 별도 저장소에서 각자 실행된다. 이 셋이 API를 통해 대화한다. 예를 들어 쇼핑몰에서 "주문" 버튼을 누르면, 프론트엔드가 백엔드 API에 요청을 보내고, 백엔드가 데이터베이스에 주문을 저장한 뒤 결과를 응답한다.
- **핵심 키워드**: [풀스택(Full Stack), 프론트엔드, 백엔드, 데이터베이스, API 연결]
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-1-1

## Part II. 웹 프레임워크 개론 (★☆☆~★★☆)

Part I에서 웹의 기본 개념을 익혔다면, Part II에서는 "실제로 웹 서버를 어떻게 만드는가"로 한 걸음 더 들어간다. 웹 프레임워크(Web Framework)가 무엇인지, 없으면 얼마나 힘든지, 어떤 선택지들이 있는지를 살펴본다. 그리고 FastAPI가 다른 프레임워크와 어떤 점에서 다른지를 확인한다.

Python 기초 문법(변수, 함수 정도)을 아는 독자라면 더 쉽게 읽을 수 있지만, 완전히 처음인 독자도 개념 이해에는 무리가 없다. 코드 예시가 등장하더라도 읽는 수준으로만 살펴보면 된다.

### Chapter II-1. 프레임워크란 무엇인가

#### II-1-1. 맨땅에서 짓지 않고 틀을 쓰는 이유

- **난이도**: ★☆☆
- **선수 지식**: I-5-3
- **학습 목표**: 이 섹션을 마치면 웹 프레임워크가 왜 필요한지를 "레고 틀" 비유로 설명할 수 있다.
- **설명**: 집을 지을 때 벽돌 하나하나를 직접 굽는 사람은 없다. 미리 만들어진 자재를 조합해서 빠르고 안정적으로 짓는다. 웹 서버도 마찬가지다. HTTP 요청을 받고, URL을 분석하고, 데이터를 검증하고, 응답을 만드는 일을 매번 처음부터 구현하면 시간이 엄청나게 걸리고 실수도 많다. 프레임워크(Framework)는 이 반복 작업을 미리 구현해둔 틀이다. 개발자는 "어떤 URL로 오면 어떤 데이터를 돌려준다"는 핵심 로직에만 집중하면 된다. 건물 구조는 프레임워크가, 내용은 개발자가 채운다.
- **핵심 키워드**: [프레임워크(Framework), 재사용, 생산성, 표준화]
- **시각화 연결**: [II-1-1 시각화 보기](part2/II-1-1.html) — 맨땅 vs 프레임워크 빌드 과정 비교 (5단계)
- **다음 섹션**: II-1-2

---

#### II-1-2. "라이브러리"와 "프레임워크"의 차이 — 전화 받는 쪽이 누구인가

- **난이도**: ★☆☆
- **선수 지식**: II-1-1
- **학습 목표**: 이 섹션을 마치면 라이브러리와 프레임워크의 핵심 차이(제어 역전)를 설명할 수 있다.
- **설명**: 라이브러리(Library)와 프레임워크(Framework)는 비슷해 보이지만 "누가 주도권을 쥐는가"에서 다르다. 라이브러리는 내가 필요할 때 불러서 쓰는 도구다. 공구함에서 드라이버를 꺼내 쓰는 것처럼, 내가 주도한다. 반면 프레임워크는 내가 프레임워크 안으로 들어가서 정해진 방식으로 코드를 작성한다. 악단에서 지휘자(프레임워크)가 연주 순서를 정하고, 연주자(개발자)는 자기 파트만 연주하는 것에 가깝다. 이 개념을 "제어의 역전(Inversion of Control, IoC)"이라고 부른다.
- **핵심 키워드**: [라이브러리(Library), 프레임워크(Framework), 제어의 역전(Inversion of Control, IoC)]
- **시각화 연결**: [II-1-2 시각화 보기](part2/II-1-2.html) — 라이브러리 vs 프레임워크 제어 흐름 비교 (4단계)
- **다음 섹션**: II-2-1

### Chapter II-2. 썼을 때 vs 안 썼을 때

#### II-2-1. 생(raw) 소켓 코드의 양 비교

- **난이도**: ★★☆
- **선수 지식**: II-1-2
- **학습 목표**: 이 섹션을 마치면 프레임워크 없이 웹 서버를 만들 때 얼마나 많은 코드가 필요한지 설명할 수 있다.
- **설명**: 프레임워크 없이 Python만으로 "Hello" 한 마디를 응답하는 웹 서버를 만들려면 소켓(Socket)을 직접 열고, TCP 연결을 관리하고, HTTP 요청 문자열을 직접 파싱하고, 응답 헤더를 직접 작성해야 한다. 최소 50~100줄이다. FastAPI로 같은 서버를 만들면 5줄이다. 프레임워크가 HTTP 파싱, 라우팅, 에러 처리, 응답 직렬화를 대신 해주기 때문이다. 물론 프레임워크 내부를 이해하는 것도 중요하지만, 처음부터 모든 것을 구현하는 것은 비효율적이다.
- **핵심 키워드**: [소켓(Socket), HTTP 파싱, 라우팅(Routing), 보일러플레이트(Boilerplate)]
- **시각화 연결**: [II-2-1 시각화 보기](part2/II-2-1.html) — 생 소켓 31줄 vs FastAPI 7줄, 단계별 코드 비교 (5단계)
- **다음 섹션**: II-2-2

#### II-2-2. 프레임워크가 대신 해주는 일 목록

- **난이도**: ★★☆
- **선수 지식**: II-2-1
- **학습 목표**: 이 섹션을 마치면 웹 프레임워크가 자동으로 처리해주는 기능을 5가지 이상 열거할 수 있다.
- **설명**: 웹 프레임워크는 개발자가 매번 다시 짜야 하는 공통 작업을 대신 처리한다. 주요 항목을 나열하면 이렇다. 첫째, 라우팅(Routing) — URL과 처리 함수를 연결한다. 둘째, 요청 파싱 — HTTP 요청에서 헤더, 바디, 쿼리 파라미터를 꺼낸다. 셋째, 응답 직렬화(Serialization) — Python 객체를 JSON으로 변환한다. 넷째, 에러 처리 — 예외 발생 시 적절한 상태 코드로 응답한다. 다섯째, 미들웨어(Middleware) 연결 — 로깅, 인증 같은 공통 처리를 요청 전후에 끼운다. 이 모든 것을 직접 구현하지 않아도 된다는 것이 프레임워크의 가장 큰 장점이다.
- **핵심 키워드**: [라우팅, 직렬화(Serialization), 미들웨어(Middleware), 에러 처리]
- **시각화 연결**: 해당 없음

### Chapter II-4. 동기 vs 비동기

#### II-4-1. 블로킹과 논블로킹 — 햄버거 가게 비유

- **난이도**: ★★☆
- **선수 지식**: II-3-3
- **학습 목표**: 이 섹션을 마치면 동기(블로킹)와 비동기(논블로킹)의 차이를 일상 비유로 설명할 수 있다.
- **설명**: 햄버거 가게에서 손님이 주문하면 두 가지 방식이 있다. 첫 번째는 동기(Synchronous, 블로킹) 방식이다. 카운터 직원이 한 손님의 햄버거가 완성될 때까지 그 자리에 서서 기다렸다가 건네주고, 그다음 손님을 받는다. 한 번에 한 명씩이다. 두 번째는 비동기(Asynchronous, 논블로킹) 방식이다. 직원이 주문을 받으면 번호표를 주고, 햄버거가 준비되는 동안 다음 손님을 받는다. 완성되면 번호를 호출한다. 웹 서버에서 "기다리는 시간"은 대부분 데이터베이스 조회나 외부 API 호출 같은 I/O(입출력) 대기다. 비동기는 이 대기 시간에 다른 요청을 처리함으로써 전체 처리량을 높인다.
- **핵심 키워드**: [동기(Synchronous), 비동기(Asynchronous), 블로킹(Blocking), 논블로킹(Non-blocking), I/O 대기]
- **시각화 연결**: [II-4-1 시각화 보기](part2/II-4-1.html) — 햄버거 가게 비유, 동기 9분 vs 비동기 3분 타임라인 (4단계)
- **다음 섹션**: II-4-2

#### II-4-2. WSGI vs ASGI (Python 한정)

- **난이도**: ★★☆
- **선수 지식**: II-4-1
- **학습 목표**: 이 섹션을 마치면 WSGI와 ASGI의 차이를 설명하고, FastAPI가 어느 표준을 사용하는지 말할 수 있다.
- **설명**: Python 웹 프레임워크는 웹 서버와 소통하기 위한 표준이 필요하다. WSGI(Web Server Gateway Interface)는 동기 전용 표준이다. Gunicorn, uWSGI 같은 서버와 함께 쓰이며, Django와 Flask의 전통적인 실행 방식이다. ASGI(Asynchronous Server Gateway Interface)는 동기와 비동기를 모두 지원하는 더 현대적인 표준이다. WebSocket(실시간 통신), HTTP/2도 지원한다. FastAPI는 ASGI 전용 프레임워크로, Uvicorn이라는 ASGI 서버 위에서 실행된다. 간단히 정리하면, WSGI는 전통 방식, ASGI는 비동기 시대의 새 표준이다.
- **핵심 키워드**: [WSGI, ASGI, Uvicorn, Gunicorn, WebSocket]
- **시각화 연결**: [II-4-2 시각화 보기](part2/II-4-2.html) — WSGI vs ASGI 좌우 비교 + 요약 표 (4단계)
- **다음 섹션**: II-4-3

#### II-4-3. 비동기가 언제 유리하고 언제 불리한가

- **난이도**: ★★☆
- **선수 지식**: II-4-2
- **학습 목표**: 이 섹션을 마치면 비동기 방식이 효과적인 상황과 그렇지 않은 상황을 각각 예를 들어 설명할 수 있다.
- **설명**: 비동기가 항상 더 빠른 것은 아니다. 비동기는 I/O 대기(데이터베이스 조회, 외부 API 호출, 파일 읽기) 시간이 많을 때 유리하다. 직원이 주방을 기다리는 대신 다른 손님을 받을 수 있기 때문이다. 반면 이미지 변환, ML 추론(기계 학습 결과 계산) 같은 CPU를 많이 쓰는 작업은 비동기로 처리해도 빠르지 않다. 직원이 계속 바쁘게 일하는 상황에서는 비동기 전환의 이점이 없다. 이 경우에는 CPU 코어를 여러 개 쓰는 멀티프로세싱(Multiprocessing)이나 별도 작업 워커가 필요하다. 비동기는 I/O 집약(I/O-Bound) 작업을 위한 도구임을 기억하자.
- **핵심 키워드**: [I/O 집약(I/O-Bound), CPU 집약(CPU-Bound), 멀티프로세싱(Multiprocessing), async/await]
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-5-1

### Chapter II-5. FastAPI의 차별점

#### II-5-1. 자동 OpenAPI 문서

- **난이도**: ★★☆
- **선수 지식**: II-4-3
- **학습 목표**: 이 섹션을 마치면 FastAPI의 자동 문서 생성 기능이 무엇인지, 왜 편리한지 설명할 수 있다.
- **설명**: API를 만들면 사용법을 문서로 남겨야 한다. 다른 개발자가 "어떤 URL로, 어떤 데이터를 보내야 하는지" 알아야 하기 때문이다. 보통은 문서를 따로 작성해야 한다. FastAPI는 이 과정을 자동으로 처리한다. 코드를 작성하면 `/docs` 경로에 Swagger UI(API를 브라우저에서 직접 테스트할 수 있는 화면)가 자동으로 생성된다. `/redoc`에는 더 깔끔한 읽기 전용 문서가 생긴다. 두 문서 모두 OpenAPI(API 문서 국제 표준) 형식을 따른다. 코드와 문서가 항상 일치하기 때문에 "코드는 바꿨는데 문서는 옛날 것"이라는 문제가 사라진다.
- **핵심 키워드**: [OpenAPI, Swagger UI, ReDoc, 자동 문서화]
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-5-2

#### II-5-2. Pydantic 타입 검증

- **난이도**: ★★☆
- **선수 지식**: II-5-1
- **학습 목표**: 이 섹션을 마치면 Pydantic이 무엇이고, FastAPI가 왜 Pydantic을 사용하는지 설명할 수 있다.
- **설명**: API에서 클라이언트가 보내는 데이터가 항상 올바른 형식일 수는 없다. 나이에 문자열을 보내거나, 필수 항목을 빠뜨리거나, 이메일 형식이 틀릴 수 있다. 이를 검증하는 코드를 매번 직접 짜는 것은 번거롭다. FastAPI는 Pydantic(파이댄틱)이라는 검증 라이브러리를 활용한다. Python 타입 힌트(예: `name: str`, `age: int`)만 작성하면 FastAPI가 자동으로 들어오는 데이터를 검증한다. 형식이 잘못되면 422 Unprocessable Entity 에러와 함께 어디가 틀렸는지 자세한 메시지를 돌려준다. 이를 통해 검증 코드를 따로 짜지 않고도 안전한 API를 만들 수 있다.
- **핵심 키워드**: [Pydantic, 타입 힌트(Type Hint), 데이터 검증(Validation), 422 에러]
- **시각화 연결**: 해당 없음
- **다음 섹션**: II-5-3

#### II-5-3. async/await 기본 + Depends() 의존성 주입

- **난이도**: ★★☆
- **선수 지식**: II-5-2
- **학습 목표**: 이 섹션을 마치면 FastAPI에서 async/await를 쓰는 이유와 Depends()가 하는 역할을 간단히 설명할 수 있다.
- **설명**: FastAPI는 Python의 `async def`(비동기 함수)를 기본으로 지원한다. `await`를 붙이면 I/O 대기 중에 다른 요청을 처리할 수 있다. 기존 `def`(동기 함수)도 함께 사용할 수 있어서 점진적으로 전환하기 편하다. Depends()는 FastAPI의 의존성 주입(Dependency Injection, DI) 도구다. 예를 들어 "로그인한 사용자만 이 API를 쓸 수 있다"는 인증 로직을 함수로 만들어두고, 여러 엔드포인트에서 `Depends(get_current_user)` 한 줄로 재사용할 수 있다. 복잡한 초기화 코드(데이터베이스 세션 등)도 같은 방식으로 간단하게 주입할 수 있다.
- **핵심 키워드**: [async/await, Depends(), 의존성 주입(Dependency Injection, DI), 엔드포인트(Endpoint)]
- **시각화 연결**: 해당 없음

- **설명**: FastAPI는 다음 상황에서 특히 강점을 발휘한다. 첫째, 고성능 API가 필요할 때다. 많은 동시 요청을 처리해야 하거나 응답 속도가 중요한 서비스에 적합하다. 둘째, ML 모델 서빙(Machine Learning Model Serving)이다. 학습된 AI 모델을 API로 제공할 때 FastAPI가 Python ML 생태계(PyTorch, TensorFlow 등)와 잘 어울린다. 셋째, 타입 안전성과 자동 문서가 중요한 팀이다. 타입 힌트를 기반으로 하기 때문에 IDE(코드 편집 도구) 자동완성과 실수 방지가 뛰어나다. 넷째, 소규모 팀에서 빠르게 API를 만들어야 할 때다. 보일러플레이트가 적고, 설정 없이 바로 시작할 수 있다.

#### II-6-2. 부적합한 시나리오 — FastAPI가 정답이 아닌 경우

- **난이도**: ★★☆
- **선수 지식**: II-6-1
- **학습 목표**: 이 섹션을 마치면 FastAPI보다 다른 프레임워크가 더 나은 상황을 2가지 이상 설명할 수 있다.
- **설명**: FastAPI가 항상 최선의 선택은 아니다. 어떤 상황에서는 다른 프레임워크가 더 적합하다. 첫째, 관리자 페이지(Admin UI)가 핵심인 프로젝트다. Django는 `/admin` 경로에 강력한 관리 화면을 자동으로 제공한다. FastAPI에는 이 기능이 없어서 직접 만들어야 한다. 둘째, 팀 전체가 Django나 Flask에 익숙한 경우다. 기술 전환 비용보다 기존 스택의 생산성이 더 클 수 있다. 셋째, 완성된 풀스택 솔루션이 필요할 때다. Django는 인증, ORM, 마이그레이션(Migration), 템플릿 엔진이 모두 내장돼 있어서 빠르게 "전체를 만들어야 하는" 상황에 더 유리하다. 도구는 맥락에 맞게 선택하는 것이 중요하다.

## Part III. FastAPI 첫 걸음 (★★☆)

Part III는 실제로 손을 움직이는 파트다. Python 기초 문법(변수, 함수, 클래스)을 한 번이라도 본 적 있다면, 이 파트를 마친 뒤 자신의 컴퓨터에서 돌아가는 API를 직접 확인할 수 있다.

#### III-2-2. venv 만들기·켜기·끄기

- **난이도**: ★★☆

- **설명**: `pip`은 Python의 패키지 설치 도구다. 앱스토어에서 앱을 내려받듯, `pip`으로 오픈소스 라이브러리를 설치한다. FastAPI는 `pip install fastapi[standard]`로 설치한다 — `[standard]`를 붙이면 FastAPI와 함께 Uvicorn(비동기 웹서버, ASGI Server)도 같이 설치된다. ### Chapter III-4. Swagger UI

#### III-4-1. /docs 경로 열기

- **난이도**: ★★☆
- **선수 지식**: III-3-3
- **학습 목표**: 이 섹션을 마치면 `/docs` 경로에서 Swagger UI(스웨거 UI)가 자동으로 생성된 이유를 설명하고, 화면 구성 요소(엔드포인트 목록, HTTP 메서드 색상)를 읽을 수 있다.

### Chapter III-5. 처음 만나는 에러

#### III-5-1. "Port already in use" — 포트 충돌

- **난이도**: ★★☆
- **선수 지식**: III-4-2
- **학습 목표**: 이 섹션을 마치면 "Address already in use" 또는 "Port 8000 is already in use" 에러의 원인을 설명하고, 이미 실행 중인 서버를 찾아 종료하거나 다른 포트를 지정할 수 있다.
- **설명**: uvicorn을 두 번 실행하거나, 이전 서버가 백그라운드에서 아직 돌고 있을 때 이 에러가 난다. 창구(포트) 하나에 두 직원(서버 프로세스)이 동시에 들어가려는 상황이다. 해결법은 두 가지다. 첫째, 터미널에서 이미 실행 중인 uvicorn을 `Ctrl+C`로 종료한 뒤 다시 실행한다. 둘째, 다른 포트를 쓴다 — `uvicorn main:app --reload --port 8001`처럼 `--port` 옵션으로 포트 번호를 바꾼다. Windows에서 강제로 프로세스를 찾으려면 `netstat -ano | findstr :8000`, Mac/Linux에서는 `lsof -i :8000`을 사용한다.

#### III-5-2. "ModuleNotFoundError" — 설치 확인

- **학습 목표**: 이 섹션을 마치면 `ModuleNotFoundError: No module named 'fastapi'` 에러를 보고 가상환경 활성화 여부를 먼저 확인하는 디버깅 습관을 만들 수 있다.
- **설명**: `ModuleNotFoundError`는 Python이 `import fastapi` 같은 코드를 실행할 때 해당 패키지를 찾지 못했다는 뜻이다. 대부분의 원인은 가상환경을 활성화하지 않은 상태에서 실행했거나, 가상환경을 켠 뒤 `pip install`을 하지 않은 경우다. 확인 순서는: ① 터미널 앞에 `(venv)` 표시가 있는가? → 없으면 `source venv/bin/activate`(Mac) 또는 `venv\Scripts\activate`(Windows) 실행. ② 활성화 후에도 에러 나면 `pip install fastapi[standard]` 다시 실행. ③ `pip list | grep fastapi`로 설치 여부 확인.
- **핵심 키워드**: [ModuleNotFoundError, 가상환경 활성화, pip install]

#### III-5-3. 422 Unprocessable Entity — 타입 오류

- **난이도**: ★★☆
- **선수 지식**: III-5-2
- **학습 목표**: 이 섹션을 마치면 HTTP 422 상태 코드가 어디서 발생하는지 설명하고, 응답 바디의 `detail` 필드를 읽어 어떤 파라미터가 잘못됐는지 파악할 수 있다.
- **설명**: 422는 "요청 형식은 맞는데, 데이터 내용이 기대와 다를 때" FastAPI가 자동으로 돌려보내는 응답이다. 예를 들어 엔드포인트가 `age: int`를 기대하는데 `"age": "스물둘"` 같은 문자열을 보내면 422가 발생한다. 이것은 에러가 아니라 FastAPI(정확히는 Pydantic)가 대신 해주는 검증이다 — 코드 안으로 잘못된 데이터가 들어오기 전에 입구에서 걸러주는 역할이다. 응답 바디의 `detail` 배열을 보면 어느 필드가 어떤 이유로 거부됐는지 설명이 있다. 자세한 Pydantic 검증 동작은 IV-1-2에서 다룬다.
- **핵심 키워드**: [422 Unprocessable Entity, Pydantic 검증, detail 필드]

### Chapter IV-1. 비동기의 세계

#### IV-1-1. 비동기 I/O와 Event Loop

- **난이도**: ★★☆
- **선수 지식**: III-3-3 (브라우저로 Hello World 동작 확인)
- **학습 목표**: 이 섹션을 마치면 비동기 I/O(Asynchronous I/O)와 Event Loop가 무엇인지 설명하고, `async def`와 `await`를 사용하는 이유를 말할 수 있다.
- **설명**: 식당에 점원이 1명 있다고 상상하자. 동기(Blocking) 방식이라면 손님 A의 햄버거 패티가 다 익을 때까지(I/O 대기) 점원이 가만히 서서 기다린다. 그동안 손님 B, C는 아무도 못 받는다. 비동기(Non-blocking) 방식이라면 패티를 올려두고 "다 구워지면 알려줘"라고 맡긴 뒤, 그 시간에 다른 손님의 주문을 계속 받는다. FastAPI의 Event Loop는 이 "알려줘" 신호를 기다리고 관리하는 관제탑이다. `async def`로 함수를 만들고, I/O 작업 앞에 `await`를 붙이면 Event Loop가 대기 중에 다른 작업을 처리한다. CPU 계산처럼 대기 없이 바로 끝나는 작업은 비동기로도 빨라지지 않는다 — 비동기의 이점은 오직 I/O 대기 시간에 있다.
- **핵심 키워드**: [비동기 I/O, Event Loop, async/await]
- **시각화 연결**: [IV-1-1 시각화 보기](part4/chapter-01-01-async.html) — 비동기 I/O & Event Loop (7단계)

#### IV-1-2. Pydantic 데이터 검증

- **난이도**: ★★☆

- **설명**: Pydantic은 FastAPI 주방의 입구 검수원이다. 손님(클라이언트)이 주문서(JSON 바디)를 보내면, 검수원이 "나이는 숫자여야 하는데 글자가 왔다", "이름 필드가 빠졌다"처럼 형식을 확인하고 잘못된 주문은 주방 안으로 들이지 않는다(422 자동 반환). `class User(BaseModel): name: str; age: int` 처럼 Python 타입 힌트로 스키마를 정의하면, FastAPI가 자동으로 요청 바디를 이 모델로 파싱하고 검증한다. Pydantic v2(FastAPI 0.100.0 이상의 기본값)는 Rust로 재작성돼 이전 버전보다 훨씬 빠르다. `.model_dump()`로 모델을 딕셔너리로 변환할 수 있다(v1의 `.dict()`가 v2에서 변경됨).

### Chapter IV-2. 엔드포인트 설계

#### IV-2-1. Path Parameter와 Query Parameter

- **난이도**: ★★☆
- **선수 지식**: IV-1-3
- **학습 목표**: 이 섹션을 마치면 Path Parameter(경로 파라미터)와 Query Parameter(쿼리 파라미터)를 구별하고, 함수 파라미터로 자동 매핑되는 FastAPI 규칙을 설명할 수 있다.
- **설명**: URL의 주소 부분인 `/users/123`에서 `123`이 Path Parameter이고, `?skip=0&limit=10`처럼 `?` 뒤에 붙는 것이 Query Parameter다. FastAPI에서는 함수 파라미터 이름만 맞추면 자동으로 매핑된다. `@app.get("/users/{user_id}")` 아래 `def get_user(user_id: int):`처럼 쓰면 URL의 `{user_id}` 부분이 정수로 변환돼 함수에 들어온다. 쿼리 파라미터는 `def list_users(skip: int = 0, limit: int = 10):`처럼 기본값(default)이 있는 파라미터로 선언한다. Pydantic이 타입도 자동 변환해주므로 문자열 URL에서 꺼낸 값을 정수로 직접 변환할 필요가 없다.
- **핵심 키워드**: [Path Parameter, Query Parameter, 자동 타입 변환]

#### IV-2-2. Response Model과 데이터 필터링

- **난이도**: ★★★
- **선수 지식**: IV-2-1
- **학습 목표**: 이 섹션을 마치면 `response_model` 파라미터로 응답 스키마를 지정하고, DB에서 꺼낸 데이터 중 민감한 필드(예: 비밀번호)가 응답에서 자동으로 제외되는 원리를 설명할 수 있다.
- **설명**: DB에서 사용자 데이터를 꺼낼 때 비밀번호, 내부 ID 같은 민감한 필드가 포함될 수 있다. 이것을 실수로 응답에 그대로 내보내면 보안 사고가 된다. `response_model`은 응답의 출구 역할을 한다 — DB에서 꺼낸 거친 데이터가 이 모델을 통과하면서 허용된 필드만 남기고 나머지는 자동으로 제거된다. `@app.get("/users/{user_id}", response_model=UserResponse)`처럼 데코레이터에 모델을 지정하면, FastAPI가 반환값을 자동으로 해당 Pydantic 모델로 변환하고 직렬화한다. 요청용 모델(`UserCreate`)과 응답용 모델(`UserResponse`)을 분리하는 것이 실무 관례다.

#### IV-2-3. 예외 처리 (HTTPException)

- **난이도**: ★★★
- **선수 지식**: IV-2-2

### Chapter IV-3. 의존성 주입과 인증

#### IV-3-1. Depends() 동작 원리

- **난이도**: ★★★
- **선수 지식**: IV-2-3
- **학습 목표**: 이 섹션을 마치면 `Depends()`를 사용해 공통 로직(인증 확인, 파라미터 파싱 등)을 함수로 분리하고, 여러 엔드포인트에 재사용할 수 있다.
- **설명**: 의존성 주입(Dependency Injection)은 "내가 필요한 것을 직접 만들지 않고, 누군가 만들어서 내 손에 쥐어주는 방식"이다. FastAPI에서 `Depends()`는 그 "누군가"다. `def get_current_user(token: str = Depends(oauth2_scheme)):` 처럼 파라미터에 `Depends(함수)`를 선언하면, 라우터 함수가 실행되기 직전에 FastAPI가 자동으로 해당 함수를 실행하고 반환값을 라우터에 꽂아준다(주입). 인증 확인, 공통 쿼리 파라미터 파싱, 권한 검사처럼 여러 엔드포인트가 공유하는 로직을 `Depends()`로 분리하면 중복 코드 없이 재사용할 수 있다.
- **핵심 키워드**: [Depends(), 의존성 주입(Dependency Injection), 재사용]

- **다음 섹션**: IV-3-2

#### IV-3-2. 데이터베이스 세션 관리

- **난이도**: ★★★
- **선수 지식**: IV-3-1
- **학습 목표**: 이 섹션을 마치면 `yield`를 사용한 제너레이터 기반 `Depends()`로 DB 세션(Session)을 요청 시작에 열고 요청 종료 시 닫는 생명주기 패턴을 구현할 수 있다.
- **설명**: DB 세션은 설거지통처럼 생각하면 된다. 손님이 오면(요청 시작) 설거지통에 물을 채우고(세션 열기), 식사를 마치면(요청 종료) 물을 비운다(세션 닫기). 세션을 닫지 않으면 커넥션이 쌓여 DB가 더 이상 연결을 받지 못하게 된다. FastAPI에서는 `def get_db(): db = SessionLocal(); try: yield db; finally: db.close()` 패턴으로 구현한다. `yield` 전에 세션을 열고, `yield`가 라우터에 세션을 넘겨주고, 라우터 실행이 끝나면 `finally` 블록에서 반드시 세션을 닫는다. 이 함수를 `Depends(get_db)`로 등록하면 모든 엔드포인트에서 자동으로 세션 생명주기가 관리된다.
- **핵심 키워드**: [DB 세션, yield, 생명주기 관리]
- **시각화 연결**: [`chapter-03-02-db-session.html`](./part4/chapter-03-02-db-session.html) — 세션 열기→yield→라우터 실행→finally 닫기 5단계
- **다음 섹션**: IV-3-3

#### IV-3-3. JWT 인증 (전역 Bearer Token)

- **선수 지식**: IV-3-2
- **학습 목표**: 이 섹션을 마치면 JWT(JSON Web Token) 기반 Bearer Token 인증 흐름을 설명하고, `Depends()`를 활용해 전역 인증을 적용하는 패턴을 구현할 수 있다.
- **설명**: JWT는 "위조가 어려운 디지털 출입 증명서"다. 사용자가 로그인하면 서버가 JWT를 발급하고, 이후 요청마다 클라이언트는 HTTP 헤더에 `Authorization: Bearer <토큰>`을 담아 보낸다. 서버는 요청마다 이 토큰을 복호화(Decode)해 유효한 사용자인지 확인하고, 통과하면 라우터로 진입시킨다 — 공항의 출입 통제소와 같다. FastAPI에서는 `OAuth2PasswordBearer`와 `python-jose` 라이브러리를 조합하고, 인증 함수를 `Depends(get_current_user)`로 등록해 여러 엔드포인트에 공통 적용한다. **보안 주의: 절대 JWT 비밀키(SECRET_KEY)를 코드에 하드코딩하지 않는다. 반드시 환경변수로 관리한다.**
- **핵심 키워드**: [JWT, Bearer Token, OAuth2PasswordBearer]
- **시각화 연결**: [`chapter-03-03-jwt.html`](./part4/chapter-03-03-jwt.html) — 유효한 토큰 vs 만료된 토큰 시나리오 토글로 검문소(get_current_user) 통과/차단 흐름을 비교
- **다음 섹션**: V-1-1

### Chapter V-2. MVC / MTV

#### V-2-1. MVC — Model, View, Controller

- **난이도**: ★★★

- **설명**:

#### V-2-2. MTV — Django의 이름 바꾸기 (왜 Template인가)

- **난이도**: ★★★
- **선수 지식**: V-2-1
- **학습 목표**: 이 섹션을 마치면 Django의 MTV가 MVC와 어떻게 대응되는지 표로 설명할 수 있다.
- **설명**:
  Django는 MVC와 같은 구조를 쓰지만, 이름을 다르게 붙였다.
  혼동하기 가장 쉬운 포인트는 Django의 "View"가 MVC의 "Controller"라는 점이다.
  아래 표를 보면 차이가 명확해진다.

  | Django (MTV) | MVC | 하는 일 |
  |:--|:--|:--|
  | Model | Model | 데이터 구조 + DB 쿼리 |
  | **Template** | **View** | HTML 등 화면 렌더링 |
  | **View** | **Controller** | 요청 수신 → Model 조회 → Template 전달 |

  **핵심**: Django의 "View"는 MVC의 "Controller"다. FastAPI에는 Template 개념이 없으므로 MTV는 Django 전용이다.

- **시각화 연결**: [`chapter-05-04-mtv.html`](./part5/chapter-05-04-mtv.html) — MVC↔MTV 좌우 대응 다이어그램. View≠View 혼동 포인트를 강조하고 FastAPI와의 관계까지

### Chapter V-3. 3계층 아키텍처 (FastAPI 실무 표준)

#### V-3-1. Controller / Service / Repository 역할

- **난이도**: ★★★
- **선수 지식**: V-2-3
- **학습 목표**: 이 섹션을 마치면 세 계층 각각의 책임을 정의하고, 잘못 배치된 코드를 찾아낼 수 있다.
- **설명**:
  3계층은 코드를 세 구역으로 나눈다. 각 구역의 역할은 아래와 같다.

  | 계층 | FastAPI 구현 | 책임 | 넣으면 안 되는 것 |
  |:--|:--|:--|:--|
  | Controller | `routers/` (APIRouter) | HTTP 요청 수신, 파라미터 추출, 응답 반환 | DB 쿼리, 비즈니스 계산 |
  | Service | `services/` | 비즈니스 규칙, 여러 Repository 조합 | HTTP 상태코드, ORM 직접 호출 |
  | Repository | `repositories/` | DB 쿼리, ORM 조작 | 비즈니스 판단, HTTP 개념 |

  데이터 흐름은 한 방향이다: HTTP 요청 → Controller → Service → Repository → DB.
  응답은 반대 방향으로 올라온다. 계층 건너뛰기(Controller가 Repository 직접 호출)는 금지한다.
- **핵심 키워드**: [Controller, Service, Repository, 단방향 의존성]
- **시각화 연결**: [`chapter-05-06-layered.html`](./part5/chapter-05-06-layered.html) — HTTP 요청이 Controller→Service→Repository→DB를 거치는 5단계 흐름. 각 계층 블록이 순차 활성화
- **다음 섹션**: V-3-2

#### V-3-2. FastAPI 프로젝트 폴더 구조 예시

- **난이도**: ★★★
- **선수 지식**: V-3-1
- **학습 목표**: 이 섹션을 마치면 3계층 규칙에 맞는 FastAPI 폴더 구조를 직접 만들 수 있다.
- **설명**:
  아래가 커뮤니티에서 가장 흔히 쓰는 3계층 폴더 구조다.

  ```
  app/
  ├── main.py            ← FastAPI 인스턴스 생성, 라우터 등록
  ├── core/
  │   ├── config.py      ← 환경변수 로드 (Pydantic BaseSettings)
  │   ├── database.py    ← DB 세션 팩토리
  │   └── security.py   ← JWT 유틸
  ├── routers/           ← Controller (APIRouter 모음)
  │   ├── users.py
  │   └── items.py
  ├── services/          ← 비즈니스 로직
  │   ├── user_service.py
  │   └── item_service.py
  ├── repositories/      ← DB 쿼리
  │   ├── user_repo.py
  │   └── item_repo.py
  ├── models/            ← SQLAlchemy ORM (테이블 정의)
  │   └── user.py
  └── schemas/           ← Pydantic (API 요청·응답 타입)
      └── user.py
  ```

  `models/`는 DB 테이블 구조, `schemas/`는 API 인터페이스 타입이다.
  둘을 분리하는 이유는 DB 구조와 API 응답이 항상 1:1이 아니기 때문이다
  (예: DB에는 비밀번호 해시가 있지만 API 응답에서는 제거).
- **핵심 키워드**: [폴더 구조, models, schemas, core, routers, services, repositories]
- **시각화 연결**: [`chapter-05-07-folder.html`](./part5/chapter-05-07-folder.html) — 커뮤니티 표준 FastAPI 폴더 구조를 단계별 탐색. routers→services→repositories→models/schemas 하이라이트

#### V-3-3. APIRouter로 Controller 나누기

- **난이도**: ★★★
- **선수 지식**: V-3-2
- **학습 목표**: 이 섹션을 마치면 APIRouter를 사용해 라우터를 파일 단위로 나누고 main.py에 등록할 수 있다.
- **설명**:
  모든 엔드포인트를 `main.py` 하나에 넣으면 파일이 금방 수백 줄이 된다.
  `APIRouter`를 쓰면 도메인별로 파일을 나눌 수 있다.

  ```python
  # routers/users.py
  from fastapi import APIRouter

  router = APIRouter(prefix="/users", tags=["users"])

  @router.get("/{user_id}")
  async def get_user(user_id: int):
      ...
  ```

  ```python
  # main.py
  from fastapi import FastAPI
  from app.routers import users, items

  app = FastAPI()
  app.include_router(users.router)
  app.include_router(items.router)
  ```

  `prefix`를 설정하면 라우터 안의 모든 경로에 `/users`가 자동으로 붙는다.
  `tags`는 Swagger UI에서 그룹 이름으로 표시된다.
  이렇게 나뉜 라우터 파일 하나가 3계층의 Controller 역할을 한다.
- **핵심 키워드**: [APIRouter, prefix, tags, include_router, 모듈 분리]
- **시각화 연결**: [`chapter-05-08-apirouter.html`](./part5/chapter-05-08-apirouter.html) — main.py 하나에서 routers/users.py, items.py로 분리하는 4단계. Before→분리→등록→완성

### Chapter V-4. 더 멀리 — 헥사고날 / 클린

#### V-4-1. 헥사고날 (Ports & Adapters)

- **난이도**: ★★★
- **선수 지식**: V-3-3
- **학습 목표**: 이 섹션을 마치면 헥사고날 아키텍처의 Port와 Adapter 개념을 한 문장으로 설명할 수 있다.
- **설명**:
  헥사고날은 Alistair Cockburn이 제안한 패턴이다.
  핵심 비즈니스 로직(Application Core)이 DB나 HTTP 같은 외부 세계에 직접 의존하지 않도록

  `외부 시스템 → Adapter → Port → Application Core` 순으로만 의존한다.
  덕분에 DB를 PostgreSQL에서 MongoDB로 바꿔도 비즈니스 로직 코드는 건드리지 않는다.
  단점은 초기 구조 설계가 복잡하다는 점이다 — 소규모 프로젝트에는 과설계다.

#### V-4-2. 클린 아키텍처와 의존성 역전

- **설명**:

### Chapter V-5. 비동기 작업 분리

#### V-5-1. BackgroundTasks로 느린 일을 뒤로

- **난이도**: ★★★
- **선수 지식**: V-4-3
- **학습 목표**: 이 섹션을 마치면 FastAPI의 BackgroundTasks를 사용해 응답 후에 실행되는 작업을 등록할 수 있다.
- **설명**:
  사용자가 회원가입을 하면, 환영 이메일을 보내야 한다.
  이메일 전송은 보통 수 초가 걸린다 — 이 동안 사용자는 응답을 기다려야 할까?
  `BackgroundTasks`를 쓰면 "가입 완료" 응답을 즉시 보내고, 이메일 전송은 뒤에서 처리할 수 있다.

  ```python
  from fastapi import BackgroundTasks

  def send_welcome_email(email: str):
      # 실제 이메일 전송 로직
      ...

  @router.post("/signup")
  async def signup(background_tasks: BackgroundTasks, email: str):
      background_tasks.add_task(send_welcome_email, email)
      return {"message": "가입 완료"}
  ```

  **주의**: BackgroundTasks는 같은 프로세스 안에서 실행된다.
  CPU를 많이 쓰는 작업(이미지 변환, ML 추론)에는 적합하지 않다.
  그런 작업은 Celery 같은 별도 워커가 필요하다.
- **핵심 키워드**: [BackgroundTasks, 비동기 작업, 응답 후 처리, 이메일 전송]
- **시각화 연결**: [`chapter-05-12-background.html`](./part5/chapter-05-12-background.html) — Before(동기 대기)→After(즉시 응답) 비교. add_task 동작 원리 + 적합/부적합 작업 분류
- **다음 섹션**: V-5-2

#### V-5-2. 더 큰 규모 — Celery / 메시지 큐 개요

- **난이도**: ★★★
- **선수 지식**: V-5-1
- **학습 목표**: 이 섹션을 마치면 Celery와 메시지 큐가 무엇인지, BackgroundTasks와 언제 구분해서 쓰는지 설명할 수 있다.
- **설명**:
  BackgroundTasks의 한계는 서버가 꺼지면 작업도 사라진다는 점이다.
  **Celery**는 작업을 **메시지 큐**(Redis나 RabbitMQ 같은 대기열)에 저장하고,
  별도 워커 프로세스가 큐에서 꺼내 실행한다.
  서버가 재시작돼도 큐에 남은 작업은 보장된다.
  메시지 큐는 "번호표 뽑는 기계"와 같다 — API 서버가 번호표를 발급하면, 워커가 번호 순서대로 처리한다.

  | 비교 | BackgroundTasks | Celery + 메시지 큐 |
  |:--|:--|:--|
  | 설정 복잡도 | 낮음 | 높음 (Redis/RabbitMQ 별도) |
  | 작업 보장 | 없음 (서버 종료 시 유실) | 있음 (큐에 저장) |
  | 적합한 작업 | 빠른 I/O 후처리 | CPU 집약, 재시도 필요, 대용량 |

  이 섹션은 개요만 다룬다. Celery 실습은 Part VII 이후 심화 과정에서 다룬다.
- **핵심 키워드**: [Celery, 메시지 큐, Redis, 워커, 작업 보장]
- **시각화 연결**: [`chapter-05-13-celery.html`](./part5/chapter-05-13-celery.html) — 번호표 비유로 메시지 큐 구조 시각화. BackgroundTasks vs Celery 비교표
- **다음 섹션**: VI-1-1

#### VI-1-3. ERD 그리기 (맛보기)

- **난이도**: ★★★
- **선수 지식**: VI-1-2
- **학습 목표**: 이 섹션을 마치면 할 일 관리 API에 필요한 DB 테이블을 ERD로 표현할 수 있다.
- **설명**:
  ERD(Entity Relationship Diagram)는 DB 테이블과 관계를 그림으로 표현한 것이다.
  이번 프로젝트는 테이블 하나만 필요하다.

  ```
  todos
  ┌─────────────────────────────┐
  │ id          INTEGER (PK)    │
  │ title       VARCHAR(200)    │
  │ description TEXT (nullable) │
  │ completed   BOOLEAN         │
  │ created_at  DATETIME        │
  └─────────────────────────────┘
  ```

  `id`는 DB가 자동 부여한다 (AUTO INCREMENT).
  `completed`는 기본값 `False`로 시작한다.
  `description`은 없어도 되므로 nullable로 설정한다.
  실제 프로젝트에서는 여러 테이블과 관계선(1:N, N:M)이 추가되지만, 첫 번째 ERD는 이 정도로 충분하다.
- **핵심 키워드**: [ERD, 테이블 설계, PRIMARY KEY, nullable, AUTO INCREMENT]

#### VI-2-2. Create — 할 일 추가

- **난이도**: ★★★
- **선수 지식**: VI-2-1
- **학습 목표**: 이 섹션을 마치면 POST /todos 엔드포인트를 3계층 구조로 구현하고 Swagger에서 테스트할 수 있다.
- **설명**:
  3계층 순서대로 구현한다: Schema → Model → Repository → Service → Router.

  ```python
  # schemas/todo.py
  from pydantic import BaseModel
  from typing import Optional

  class TodoCreate(BaseModel):
      title: str
      description: Optional[str] = None

  class TodoResponse(BaseModel):
      id: int
      title: str
      description: Optional[str]
      completed: bool

      class Config:
          from_attributes = True
  ```

  ```python
  # routers/todos.py
  from fastapi import APIRouter, Depends
  from app.schemas.todo import TodoCreate, TodoResponse
  from app.services.todo_service import TodoService

  router = APIRouter(prefix="/todos", tags=["todos"])

  @router.post("/", response_model=TodoResponse, status_code=201)
  async def create_todo(data: TodoCreate, service: TodoService = Depends()):
      return service.create(data)
  ```

  Router는 요청 수신과 응답 반환만 담당한다. 비즈니스 로직은 Service가, DB 저장은 Repository가 맡는다.

#### VI-2-3. Read — 목록/단일 조회

- **난이도**: ★★★
- **선수 지식**: VI-2-2
- **학습 목표**: 이 섹션을 마치면 GET /todos와 GET /todos/{id} 두 엔드포인트를 구현하고, 404 처리를 추가할 수 있다.
- **설명**:
  조회 엔드포인트 두 개를 추가한다.

  ```python
  # routers/todos.py (기존 코드에 추가)
  from fastapi import HTTPException
  from typing import List

  @router.get("/", response_model=List[TodoResponse])
  async def list_todos(service: TodoService = Depends()):
      return service.get_all()

  @router.get("/{todo_id}", response_model=TodoResponse)
  async def get_todo(todo_id: int, service: TodoService = Depends()):
      todo = service.get_by_id(todo_id)
      if todo is None:
          raise HTTPException(status_code=404, detail="할 일을 찾을 수 없습니다")
      return todo
  ```

  404 판단은 Router에서 하지 않고, Service가 None을 반환하면 Router가 HTTPException을 올린다.
  이렇게 하면 Service는 HTTP 개념을 몰라도 된다.
- **핵심 키워드**: [GET, List, path parameter, 404 HTTPException, None 처리]

#### VI-2-4. Update / Delete

- **난이도**: ★★★
- **선수 지식**: VI-2-3
- **학습 목표**: 이 섹션을 마치면 PUT /todos/{id}와 DELETE /todos/{id}를 구현하고, 204 No Content 응답을 반환할 수 있다.
- **설명**:
  수정과 삭제를 추가하면 CRUD가 완성된다.

  ```python
  from app.schemas.todo import TodoUpdate  # title, description, completed 선택적 필드

  @router.put("/{todo_id}", response_model=TodoResponse)
  async def update_todo(todo_id: int, data: TodoUpdate, service: TodoService = Depends()):
      todo = service.update(todo_id, data)
      if todo is None:
          raise HTTPException(status_code=404, detail="할 일을 찾을 수 없습니다")
      return todo

  @router.delete("/{todo_id}", status_code=204)
  async def delete_todo(todo_id: int, service: TodoService = Depends()):
      deleted = service.delete(todo_id)
      if not deleted:
          raise HTTPException(status_code=404, detail="할 일을 찾을 수 없습니다")
  ```

  DELETE 성공 시 204 No Content를 반환한다 — 본문이 없는 응답이다.
  `TodoUpdate`는 모든 필드를 Optional로 설정한다 (부분 수정 허용).
  이제 Swagger UI(`/docs`)에서 5개 엔드포인트를 모두 테스트할 수 있다.
- **핵심 키워드**: [PUT, DELETE, 204 No Content, Optional 필드, 부분 수정]

  ```python

  from fastapi.testclient import TestClient
  from app.main import app

  client = TestClient(app)

  def test_create_todo():
      response = client.post("/todos/", json={"title": "공부하기"})
      assert response.status_code == 201
      assert response.json()["title"] == "공부하기"
      assert response.json()["completed"] == False

  def test_get_todo_not_found():
      response = client.get("/todos/99999")
      assert response.status_code == 404
  ```

  ```bash
  pip install pytest httpx
  pytest tests/ -v
  ```

  두 개의 테스트만 있어도 "POST가 201을 반환하는가"와 "없는 ID가 404를 반환하는가"를 보장한다.
- **핵심 키워드**: [pytest, TestClient, assert, 자동 테스트, httpx]

## Part VII. 배포와 운영 (★★★)

> 로컬에서만 돌아가는 API는 혼자만의 작품이다.

  ```python
  # core/config.py
  from pydantic_settings import BaseSettings
  ```

  ```bash
  DATABASE_URL=sqlite:///./todos.db
  SECRET_KEY=my-secret-key-here
  ```

#### VII-2-2. requirements.txt로 의존성 고정

- **난이도**: ★★★
- **선수 지식**: VII-2-1
- **학습 목표**: 이 섹션을 마치면 requirements.txt를 생성하고, 배포 서버에서 동일한 환경을 재현할 수 있다.
- **설명**:
  로컬에서 `pip install fastapi`로 설치한 버전과 서버의 버전이 다르면 예상치 못한 오류가 생긴다.

  ```bash
  # 현재 설치된 패키지를 그대로 기록

  ```

  ```
  # requirements.txt 예시
  fastapi==0.111.0
  uvicorn==0.29.0
  sqlalchemy==2.0.30
  pydantic-settings==2.2.1
  ```

  배포 서버에서는 `pip install -r requirements.txt` 한 줄로 동일한 환경을 재현한다.
  Render는 배포 시 이 파일을 자동으로 인식하고 설치한다.
- **핵심 키워드**: [requirements.txt, pip freeze, 의존성 고정, 버전 재현]

#### VII-2-3. 헬스체크 엔드포인트 만들기

- **난이도**: ★★★
- **선수 지식**: VII-2-2
- **학습 목표**: 이 섹션을 마치면 헬스체크 엔드포인트를 추가하고, 배포 플랫폼이 서버 상태를 자동 확인하게 할 수 있다.

#### VII-3-2. Render 연결 + 배포

- **난이도**: ★★★
- **선수 지식**: VII-3-1
- **학습 목표**: 이 섹션을 마치면 Render에서 새 웹 서비스를 만들고 첫 배포를 완료할 수 있다.
- **설명**:
  아래 단계를 순서대로 따른다.

  1. https://render.com 에서 GitHub 계정으로 가입/로그인.
  2. 대시보드 → "New +" → "Web Service" 클릭.
  3. GitHub 저장소 목록에서 `todo-api` 선택.
  4. 설정 입력:
     - **Name**: `todo-api` (서비스 이름, URL에 포함됨)
     - **Runtime**: Python 3
     - **Build Command**: `pip install -r requirements.txt`
     - **Start Command**: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
     - **Instance Type**: Free
  5. "Environment Variables" 섹션에서 `.env`의 키-값 입력 (DATABASE_URL 등).
  6. "Create Web Service" 클릭.
  7. 배포 로그가 실시간으로 표시된다. "Your service is live" 메시지 확인.
  8. `https://todo-api-xxxx.onrender.com/docs` 로 Swagger UI 접속.

  `--host 0.0.0.0`은 외부 접근 허용, `$PORT`는 Render가 자동 할당하는 포트다.
- **핵심 키워드**: [Render, Web Service, Start Command, 0.0.0.0, $PORT, 배포 로그]

#### VII-3-3. 배포 실패 로그 읽기

- **난이도**: ★★★

- **학습 목표**: 이 섹션을 마치면 Render 배포 로그에서 오류 원인을 찾고 수정할 수 있다.
- **설명**:
  첫 배포에서 실패하는 경우가 많다. 당황하지 않고 로그를 읽는 것이 핵심 역량이다.
  Render 대시보드 → 서비스 → "Logs" 탭에서 실시간 로그를 확인한다.

  자주 보이는 오류와 해결법:

  | 오류 메시지 | 원인 | 해결 |
  |:--|:--|:--|
  | `ModuleNotFoundError: No module named 'fastapi'` | requirements.txt 누락 또는 Build Command 오류 | Build Command 확인. pip freeze 다시 실행. |
  | `Address already in use` | $PORT 미사용 | Start Command에 `--port $PORT` 추가 |
  | `No such file or directory: 'app/main.py'` | 경로 오류 | 저장소 루트 위치 확인 |
  | 배포 성공했는데 `/docs` 404 | app.main 경로 오류 | `uvicorn app.main:app` 경로 재확인 |

  로그의 마지막 에러 메시지부터 읽는다. 위로 올라가며 원인을 추적한다.
  수정 후 Git push만 하면 Render가 자동으로 재배포한다.

- **학습 목표**: 이 섹션을 마치면 Python logging 모듈로 구조화된 로그를 남기고, 운영 중 문제를 추적하는 방법을 이해할 수 있다.
- **설명**:
  배포 후 사용자가 오류를 보고하면, 무슨 일이 있었는지 알아야 한다.
  `print()`는 개발용이고, 운영 환경에서는 **logging** 모듈을 쓴다.

  ```python
  import logging

  logger = logging.getLogger(__name__)

  @router.post("/todos/")
  async def create_todo(data: TodoCreate):
      logger.info("새 할 일 생성 요청: %s", data.title)
      try:
          result = service.create(data)
          logger.info("생성 완료: id=%d", result.id)
          return result
      except Exception as e:
          logger.error("생성 실패: %s", str(e))
          raise
  ```

  로그 레벨은 `DEBUG < INFO < WARNING < ERROR < CRITICAL` 순이다.
  운영에서는 보통 INFO 이상만 기록한다.
  Render Logs 탭에서 이 로그를 실시간으로 볼 수 있다.
- **핵심 키워드**: [logging, 로그 레벨, INFO, ERROR, 운영 모니터링]

#### VII-4-2. Docker 맛보기

- **난이도**: ★★★
- **선수 지식**: VII-4-1
- **학습 목표**: 이 섹션을 마치면 Dockerfile의 역할을 이해하고, FastAPI 앱의 기본 Dockerfile을 읽을 수 있다.
- **설명**:
  Docker는 "내 컴퓨터에서는 됐는데 서버에서 안 돼요" 문제를 없애는 도구다.
  앱과 실행 환경(Python 버전, 라이브러리)을 하나의 **컨테이너** 상자에 담아 어디서든 동일하게 실행한다.

  ```dockerfile
  # Dockerfile
  FROM python:3.11-slim

  WORKDIR /app

  COPY requirements.txt .
  RUN pip install --no-cache-dir -r requirements.txt

  COPY ./app ./app

  CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
  ```

  Docker를 지금 당장 쓸 필요는 없다 (Render가 알아서 처리한다).
  팀 규모가 커지거나 AWS/GCP로 이전할 때 Docker가 필수가 된다.
  지금은 "이런 것이다"는 감각만 갖고, 실습은 심화 과정에서 진행한다.

#### VII-4-3. CI/CD란 무엇인가

- **난이도**: ★★★
- **선수 지식**: VII-4-2
- **학습 목표**: 이 섹션을 마치면 CI/CD의 개념과 GitHub Actions를 이용한 자동화 흐름을 설명할 수 있다.
- **설명**:
  **CI (Continuous Integration)**: 코드를 Push할 때마다 자동으로 테스트를 실행한다.
  **CD (Continuous Deployment)**: 테스트가 통과하면 자동으로 배포한다.
  이 둘을 합친 것이 CI/CD — 코드 변경 → 테스트 → 배포가 자동으로 이어지는 파이프라인이다.

  GitHub Actions를 쓰면 저장소에 `.github/workflows/deploy.yml` 파일 하나로 설정한다.
  Render는 GitHub에 Push가 오면 자동 재배포를 해준다 — 이것만으로도 간단한 CD는 완성이다.
  CI(자동 테스트)를 추가하고 싶다면 GitHub Actions에 `pytest` 실행 단계를 추가하면 된다.
  지금 당장보다는 "다음 팀 프로젝트에서 설정해보기"를 목표로 한다.
- **핵심 키워드**: [CI/CD, GitHub Actions, 자동 테스트, 자동 배포, 파이프라인]

## 부록

### A. 용어집 (가나다순)

> 본문에서 등장한 주요 용어를 가나다 순으로 정리한다.
> 형식: **용어** (영어) — 한 줄 정의.

- **가상환경** (Virtual Environment) — Python 프로젝트마다 독립된 패키지 공간을 만드는 도구. `venv` 또는 `conda` 사용.
- **게이트웨이 인터페이스** (WSGI/ASGI) — Python 웹 앱과 웹 서버를 연결하는 표준 약속. FastAPI는 ASGI 전용.
- **라우터** (Router) — 어떤 URL 경로로 요청이 오면 어떤 함수를 실행할지 연결하는 규칙 묶음.
- **레포지터리** (Repository) — 3계층에서 DB 쿼리만 담당하는 계층.
- **마이크로 프레임워크** (Micro Framework) — 뼈대만 제공하고 나머지는 직접 붙이는 가벼운 프레임워크. Flask, FastAPI 등.
- **메시지 큐** (Message Queue) — 작업 목록을 저장하는 대기열. 서버와 워커 사이에서 작업을 안전하게 전달.
- **백그라운드 태스크** (BackgroundTasks) — FastAPI에서 HTTP 응답을 보낸 후 뒷단에서 실행되는 작업 등록 기능.
- **비동기** (Async/Await) — I/O 대기 중 다른 요청을 처리하는 방식. 대기 시간이 많은 서버에서 처리량을 높임.
- **서비스 계층** (Service Layer) — 3계층에서 비즈니스 규칙을 담당하는 중간 계층.
- **스파게티 코드** — 함수 하나에 여러 책임이 뒤섞여 수정이 어려운 코드 상태를 빗댄 표현.
- **스키마** (Schema) — Pydantic으로 정의하는 API 요청/응답 타입. DB 테이블 구조(models)와 구분.
- **엔드포인트** (Endpoint) — API에서 하나의 URL + HTTP 메서드로 정의되는 기능 단위.
- **의존성 주입** (Dependency Injection, DI) — 함수가 필요한 객체를 직접 만들지 않고 외부에서 주입받는 패턴.
- **의존성 역전** (Dependency Inversion) — 고수준 모듈이 저수준 모듈에 의존하지 않도록 인터페이스를 통해 역전시키는 원칙.
- **컨테이너** (Container) — Docker가 만드는 격리된 실행 환경. 앱과 의존성을 하나의 상자에 묶음.
- **컨트롤러** (Controller) — 3계층에서 HTTP 요청 수신과 응답 반환만 담당하는 최상위 계층. FastAPI에서는 라우터.
- **클린 아키텍처** (Clean Architecture) — Uncle Bob이 제안한 4계층 구조. 의존성이 항상 안쪽(도메인)으로만 향함.
- **풀스택 프레임워크** (Full-stack Framework) — ORM, 인증, 관리자 UI 등 필요한 것을 모두 내장한 프레임워크. Django, Rails 등.
- **헤스체크** (Health Check) — 서버가 정상 동작 중인지 확인하기 위한 전용 엔드포인트. 보통 `/health`.
- **헥사고날 아키텍처** (Hexagonal Architecture) — Port와 Adapter로 핵심 로직을 외부 의존에서 격리하는 패턴.
- **환경변수** (Environment Variable) — 코드 밖에서 주입하는 설정값. DB URL, 시크릿 키 등 민감 정보 관리에 사용.
- **ASGI** — Asynchronous Server Gateway Interface. FastAPI 같은 비동기 Python 앱과 서버를 연결하는 비동기 표준.
- **CRUD** — Create, Read, Update, Delete. API가 데이터를 다루는 네 가지 기본 동작.
- **ERD** — Entity Relationship Diagram. DB 테이블과 테이블 간 관계를 그림으로 나타낸 설계도.
- **MVC** — Model-View-Controller. UI와 비즈니스 로직을 분리하는 고전 아키텍처 패턴.
- **MTV** — Model-Template-View. Django가 MVC를 변형한 패턴. Django의 View = MVC의 Controller.
- **ORM** — Object Relational Mapper. 코드에서 객체를 다루듯 DB를 조작할 수 있게 해주는 도구. SQLAlchemy 등.
- **Pydantic** — Python 타입 힌트를 이용한 데이터 검증 라이브러리. FastAPI의 핵심 의존성.
- **Render** — GitHub 연동으로 손쉽게 배포할 수 있는 PaaS 플랫폼. 무료 티어 제공.
- **WSGI** — Web Server Gateway Interface. 동기 Python 앱과 웹 서버를 연결하는 표준. Django/Flask 기본.

### B. 자주 묻는 질문 (FAQ)

**Q1. FastAPI와 Django 중 어떤 걸 먼저 배워야 하나요?**

목표에 따라 다르다. 빠르게 RESTful API를 만들고 싶다면 FastAPI가 적합하다. 관리자 UI, ORM, 인증이 기본 내장된 풀스택 앱이 필요하다면 Django가 적합하다. 이 커리큘럼은 FastAPI 기반이므로, FastAPI를 먼저 익힌 뒤 Django는 필요할 때 배워도 늦지 않다.

**Q2. `async def`를 무조건 써야 하나요?**

아니다. DB 쿼리나 외부 API 호출처럼 I/O 대기가 많은 함수에서만 효과가 있다. CPU를 많이 쓰는 계산(이미지 처리, 수학 연산)은 `async`를 붙여도 빨라지지 않는다. 동기 라이브러리를 쓴다면 `async def` 대신 `def`를 써도 FastAPI가 자동으로 스레드풀에서 실행한다.

**Q3. 3계층 구조를 꼭 써야 하나요? 소규모 프로젝트엔 너무 복잡한 것 같아요.**

맞다. 혼자 하는 프로토타입이나 엔드포인트 5개 이하의 작은 프로젝트라면 `main.py` 하나에 다 넣어도 된다. 3계층은 "도메인 로직이 복잡해지기 시작할 때" 도입하면 된다. 이 커리큘럼에서 3계층을 배우는 이유는 패턴을 미리 이해해두기 위함이지, 모든 프로젝트에 강제 적용하라는 뜻이 아니다.

**Q4. Render 무료 배포가 느린데 정상인가요?**

정상이다. Render 무료 티어는 15분 동안 요청이 없으면 슬립(sleep) 상태로 들어간다. 다음 요청이 오면 30초~1분 동안 서버가 깨어나는 시간이 걸린다. 포트폴리오 시연 전에 미리 `/health` 엔드포인트에 한 번 접속해두면 된다. 프로덕션에서는 유료 플랜 또는 다른 플랫폼을 쓴다.

**Q5. `.env` 파일을 실수로 GitHub에 올렸어요. 어떻게 해야 하나요?**

즉시 두 가지를 해야 한다. 첫째, 유출된 시크릿 키와 DB 비밀번호를 모두 변경한다. 둘째, `git rm --cached .env` 명령으로 파일을 Git 추적에서 제거하고 `.gitignore`에 추가한 뒤 커밋한다. GitHub의 파일 삭제 커밋은 이력에 남아 있으므로, 키 재발급이 핵심이다.

**Q6. Pydantic v1과 v2가 다르다고 들었는데, 어떤 걸 써야 하나요?**

FastAPI 0.100.0 이상에서는 Pydantic v2가 기본이다. 이 커리큘럼은 v2 기준으로 작성됐다. 주요 변경점: `.dict()` → `.model_dump()`, `@validator` → `@field_validator`. 오래된 튜토리얼에서 v1 문법을 보더라도 당황하지 말고, 공식 Pydantic v2 마이그레이션 문서를 참고한다.

**Q7. SQLAlchemy와 SQLModel 중 무엇을 쓰나요?**

SQLModel은 FastAPI 제작자(Sebastián Ramírez)가 만든 SQLAlchemy + Pydantic 통합 라이브러리다. 소규모 프로젝트에서 boilerplate를 줄일 수 있다. 그러나 커뮤니티 규모와 레퍼런스는 SQLAlchemy가 훨씬 크다. 이 커리큘럼은 SQLAlchemy 기준이다. SQLModel은 익숙해진 뒤에 선택지로 고려하면 된다.

**Q8. 헥사고날 아키텍처를 실무에서 FastAPI에 적용한 예시가 있나요?**

있다. Python ABC(Abstract Base Class)로 Port를 정의하고, 구현체를 Adapter로 만드는 방식이다. 그러나 FastAPI 공식 문서는 3계층 + Depends() 패턴을 권장한다. 헥사고날은 팀 합의와 프로젝트 규모가 충분할 때 적용하며, 주니어 단독으로 도입하기는 어렵다.

**Q9. BackgroundTasks 대신 항상 Celery를 써야 하는 건 아닌가요?**

BackgroundTasks가 적합한 경우도 많다. 작업 실패 시 재시도가 필요 없고, 서버 재시작 중에 작업이 유실돼도 괜찮으며, 처리 시간이 짧다면(수 초 이내) BackgroundTasks로 충분하다. Celery는 설정과 운영 비용이 추가된다. 단순한 이메일 알림은 BackgroundTasks, 대용량 파일 변환/배치 처리는 Celery가 적합하다.

**Q10. requirements.txt와 pyproject.toml 중 무엇을 써야 하나요?**

둘 다 의존성 관리 도구다. `requirements.txt`는 간단하고 어디서나 바로 통한다. `pyproject.toml`은 Poetry/PDM 같은 모던 패키지 매니저와 함께 쓰며, 의존성 해결과 가상환경 관리를 통합한다. 이 커리큘럼은 진입 장벽을 낮추기 위해 `requirements.txt`를 사용한다. 팀이나 프로젝트 규모가 커지면 Poetry 도입을 고려한다.
